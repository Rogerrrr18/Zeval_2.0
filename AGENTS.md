# ZERORE MVP Agent Rules

本项目当前阶段目标只有一个：在最短时间内打通 MVP pipeline。

## 0. 北极星目标

- 交付闭环：前端接收 -> 后端解析 -> 评估计算（客观+主观）-> 返回图表与优化建议。
- 优先级：可跑通 > 可解释 > 可演示 > 可扩展 > UI 精细度。
- 禁止在 MVP 阶段做大而全平台化设计。

## 1. 技术与范围约束

- 技术栈固定：`Next.js + TypeScript`（App Router）。
- 输入格式第一版仅支持：`CSV / JSON / TXT / MD`。
- 场景暂不固化，先实现场景无关的通用评估框架。
- 不做登录系统、多租户权限、复杂任务编排。

## 2. 模块边界（按 pipeline 拆分）

- `parser`：多格式输入解析为统一消息结构。
- `normalizer`：字段校验、缺失补齐、session/topic 切分。
- `objectiveEvaluator`：规则和算法直接产出客观指标。
- `subjectiveEvaluator`：LLM Judge 产出主观指标与理由。
- `reportBuilder`：组装图表数据与可执行优化建议。

## 3. 指标实现策略

- 客观指标先稳定：轮次分布、topic 切换率、平均时长、活跃时段、断点分布、响应间隔。
- 主观指标先可用：情绪曲线、共情程度、追问有效性、答非所问风险、说教风险。
- 主观评估必须输出结构化结果：`score / reason / evidence / confidence`。
- LLM 不可用时必须降级：保留客观指标并明确标记主观评估不可用。

## 4. 输出与验收标准

- 统一输出结构：`meta + objectiveMetrics + subjectiveMetrics + charts + suggestions`。
- 第一版前端至少展示 3-4 张核心图和 5 条建议。
- 每条建议必须绑定触发指标，格式为：问题 -> 影响 -> 建议动作。
- 单次评估链路应可在可接受时间内完成（MVP 目标 < 30s）。

## 5. 编码规范（必须遵守）

- 只做与当前任务直接相关的最小改动，避免无关重构。
- 所有新增函数使用 JSDoc 注释，描述输入、输出与降级行为。
- 关键逻辑优先纯函数化，便于后续测试与替换模型。
- 错误处理必须可观测：返回明确错误信息，不吞错。

## 6. 开发节奏

- 每次迭代遵循：先接口可用 -> 再指标正确 -> 再结果展示 -> 最后体验优化。
- 每完成一个子链路，立刻用 mock 长对话数据回归验证。
- 如遇时间不足，优先保住端到端打通，减少指标数量和图表种类。
