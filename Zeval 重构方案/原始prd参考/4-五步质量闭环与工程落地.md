# 4-五步质量闭环与工程落地

## 1. 文档目的

这份文档解决两件事：

1. 把 Zeval 从“评估工作台”升级为“质量闭环系统”的方案写清楚。
2. 以资深工程师视角说明：产品收到 PRD 之后，应该如何拆开发；数据库应该如何接入。

一句话：

**Zeval 下一阶段的核心，不是再加几张图，而是把评测结果直接编译成可执行的修复任务，并验证修复结果。**

---

## 2. 产品闭环升级

### 2.1 从旧闭环升级到新闭环

当前已跑通的链路：

`上传数据 -> 评估 -> 图表/建议 -> 保存基线 -> 在线回放`

下一阶段要升级成：

1. **发现问题**
2. **提取证据**
3. **生成调优包**
4. **交给 agent 执行**
5. **回放 / 沙箱验证**

### 2.2 为什么一定要补“调优包”和“agent 执行”

只做评测，价值会停在“发现问题”。

客户真正愿意长期付费的是：

- 更快定位问题
- 更快产出修复方案
- 更快验证改动是否生效

因此，Zeval 的下一阶段不应停留在 report 层，而应进入 `diagnose -> remediate -> validate`。

---

## 3. 五步闭环的产物设计

## 3.1 Step 1 · 发现问题

输入：

- 真实 `chatlog / trace`
- 工作台上传数据
- baseline replay 数据

输出：

- 风险会话列表
- 关键问题类型
- 失败轮次 / 死亡轮次
- 高优先级 bad case

代表性问题：

- 答非所问
- 情绪恢复失败
- 说教 / 压迫感
- topic 跑偏
- 长间隔后体验下滑

## 3.2 Step 2 · 提取证据

输出必须结构化：

- `score`
- `reason`
- `evidence`
- `confidence`

同时输出：

- 触发指标
- 对话片段
- 情绪拐点
- baseline vs current 的差异

说明：

- 这一层决定产品是否“可信”。
- 没有证据，后续调优包会变成空泛建议。

## 3.3 Step 3 · 生成调优包

建议固定为四个文件：

### `issue-brief.md`

用途：

- 给 PM、工程师、运营快速看懂问题。

内容：

- 问题描述
- 用户影响
- 证据引用
- 当前分数 / 风险
- 优先级

### `remediation-spec.yaml`

用途：

- 给 `Claude Code / Codex` 这样的 coding agent 直接读取。

内容建议：

```yaml
package_id: rem_2026_04_23_001
goal: reduce_off_topic_and_recovery_failure
source_run_id: run_xxx
priority: P0
problem_summary:
  - off_topic_risk_high
  - emotion_recovery_failure
target_metrics:
  off_topic_risk_max: 0.12
  empathy_score_min: 4
  replay_win_rate_min: 0.65
edit_scope:
  - prompt
  - orchestration
constraints:
  - do_not_reduce_safety_refusal_quality
  - do_not_increase_avg_latency_over_20_percent
acceptance:
  - replay_suite_pass
  - badcase_suite_pass
```

### `badcases.jsonl`

用途：

- 保存失败案例，供 agent 和后续回归共同使用。

单条建议字段：

- `case_id`
- `session_id`
- `turn_range`
- `context`
- `current_output`
- `problem_tags`
- `expected_behavior`
- `evidence`

### `acceptance-gate.yaml`

用途：

- 定义“怎样算修好”。

内容建议：

```yaml
replay:
  required: true
  min_win_rate: 0.65
offline_eval:
  sample_batch_id: batch_xxx
  max_regressions: 0
sandbox:
  required: false
  scenarios: []
guards:
  dangerous_reply_count: 0
  off_topic_risk_max: 0.12
```

## 3.4 Step 4 · 交给 agent 执行

建议分两阶段：

### 第一阶段

- 输出 agent-readable 文件
- 人工将其交给 `Claude Code / Codex`
- 由 agent 修改 prompt / policy / orchestration / code

### 第二阶段

- Zeval 自动把调优包发到 issue / PR
- 自动收集 agent 的执行结果
- 自动触发验证流水线

结论：

- 先做 `copilot`，不要一上来做 `autopilot`。

## 3.5 Step 5 · 回放 / 沙箱验证

验证顺序建议：

1. `Baseline Replay`
2. `固定 Sample Batch`
3. `Sandbox / Synthetic / Perturbation`

原因：

- replay 最接近真实线上问题，优先级最高。
- sandbox 应该建立在真实失败模式之上，而不是脱离真实数据独立存在。

## 4. 产品收到 PRD 后，工程上怎么拆

## 4.1 先做需求压缩，不要直接开干

收到 PRD 之后，先压成以下结构：

### A. 用户动作

- 用户是谁
- 在系统里做什么动作
- 动作前后状态是什么

### B. 成功指标

- 哪个指标代表成功
- 哪个指标代表不能退化
- 哪些只是观察项，不是 gate

### C. 输入输出契约

- 输入需要哪些字段
- 输出要哪些字段
- 哪些字段必须结构化

### D. 数据落点

- 这次功能新增的数据是临时的还是长期资产
- 应该落文件、数据库还是对象存储

### E. 回归策略

- 怎么证明新功能没有把旧链路弄坏
- replay / batch / sandbox 哪个要先跑

## 4.2 推荐的研发拆解顺序

### 第一层：先定契约

- `zod schema`
- `TypeScript type`
- API request / response
- 调优包文件格式

### 第二层：再定状态机

至少要定义这些状态：

- ingesting
- evaluating
- evidence_ready
- package_ready
- agent_running
- replay_validating
- sandbox_validating
- passed / failed

### 第三层：再定存储模型

把数据分成三类：

- **事务元数据**
  - run、package、case、validation status
- **大对象**
  - `evaluate_json`
  - `raw_rows_json`
  - 调优包文件
- **可复用资产**
  - dataset case
  - baseline snapshot
  - sample batch

### 第四层：最后定 UI

优先级：

1. 问题卡
2. 证据卡
3. replay 对比卡
4. 调优包导出
5. sandbox 面板

结论：

- 不要先做一个复杂平台壳子。
- 先做一条“发现问题 -> 导出调优包 -> 验证结果”的主链路。

## 6. 数据库应该怎么接

## 6.1 总原则

数据库不要直接从 route handler 开始乱接。

应该先做这三件事：

1. 定抽象接口
2. 定表结构
3. 再定 ORM / 驱动

## 6.2 当前代码里的真实切入点

当前代码已经有正确的一半：

- `DatasetStore`

当前还缺这一半：

- `WorkbenchBaselineStore`
- `RemediationPackageStore`
- `ValidationRunStore`

因此推荐路线不是“把所有文件系统逻辑直接改成 SQL”，而是：

### 第一阶段

- 给 `workbench baseline` 补一层 store interface
- 文件系统继续作为默认实现

### 第二阶段

- 新增 Postgres 版本实现
- 工厂函数按环境变量切换

### 第三阶段

- 再把 remediation / validation 都纳入同一套 metadata store

## 6.3 推荐数据库选型

建议：

- `PostgreSQL`
- `Drizzle ORM`
- `drizzle-kit`

原因：

- 你当前主工程是 `Next.js + TypeScript`
- Drizzle 轻，类型体验好，贴合现有项目
- 不需要引入过重的 ORM 框架

## 6.4 推荐的连接方式

环境变量：

```bash
DATABASE_PROVIDER=filesystem
DATABASE_URL=
```

切数据库后：

```bash
DATABASE_PROVIDER=postgres
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/zeval?sslmode=require
```

代码结构建议：

```txt
src/
  server/
    db/
      client.ts
      schema.ts
      migrate.ts
  workbench/
    baseline-store.ts
    baseline-file-store.ts
    baseline-postgres-store.ts
  eval-datasets/
    storage/
      dataset-store.ts
      file-system-dataset-store.ts
      postgres-dataset-store.ts
```

说明：

- `client.ts` 只负责连接数据库
- 各业务模块只依赖 store interface
- route handlers 只调 service / store，不直接写 SQL

## 6.6 文件和数据库怎么分工

建议：

- **数据库存元数据**
  - id、状态、索引字段、统计字段
- **文件系统 / 对象存储存大对象**
  - `evaluate_json`
  - `raw_rows_json`
  - 调优包文件
  - HTML 报告

原因：

- 避免把大量 artifact 全塞进行存储
- 让后续切 S3 / OSS 更容易

## 7. 开发顺序建议

### 阶段 1：强化现有链路

- 首页输出问题卡和证据卡
- baseline replay 成为核心入口

### 阶段 2：沉淀 bad case

- 将 replay / evaluate 结果中的高风险样本写入案例池
- 完成标签、去重、聚类

### 阶段 3：生成调优包

- 先导出文件
- 不先做自动执行

### 阶段 4：接入数据库

- 抽象 store
- 接 Postgres metadata
- 保留 filesystem fallback

### 阶段 5：接 agent 执行与验证门禁

- 调优包 -> issue / PR
- replay gate
- batch gate

### 阶段 6：补 sandbox

- synthetic seed
- scenario suite
- perturbation suite

---

## 8. 这份文档背后的工程判断

### 8.1 什么要先做

- 问题和证据
- bad case 资产化
- 调优包契约
- replay 验证

### 8.2 什么不要太早做

- 大而全多租户平台壳子
- 复杂任务编排系统
- 纯 synthetic 的炫技 sandbox
- 完全自动化 agent 改代码闭环

### 8.3 最重要的一句

**先把“失败 -> 修复任务 -> 验证结果”做成一条刚性链路，再谈平台化。**
