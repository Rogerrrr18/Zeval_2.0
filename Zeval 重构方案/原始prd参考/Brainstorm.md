Brainstorm
Q1 商业模式这一块，我们如果有这样一个服务/产品；服务人群到底是使用Agent提效的这一批企业 还是 开发这些提效工具/agent产品（c/b端）的公司呢？

A：这是一个很值得想清楚的问题，因为两个方向的产品形态、销售路径、付费逻辑差别很大。
我的判断是：先做 Builder 侧（开发 Agent 产品的公司），不做企业使用侧。
原因很直接。你们要做的是一个需要接入代码、采集 trace、配置评分规则的工具——这件事本身就决定了使用者必须是技术团队。一个用 AI 客服提效的传统企业，它根本没有动力也没有能力自己去接 OTel、去看 trace、去调 eval 指标；它只想知道"这个 Agent 好不好用"，而不是自己来测量。这条路要走通，你就变成了一个咨询公司，而不是一个产品公司。
反过来，正在开发 Agent 产品的团队，天然是最痛的那批人。他们的产品质量就是他们的生意，用户一流失他们就知道出问题了，但看不清是哪里出的问题。他们有工程师，能接 SDK，有动力付费，而且随着他们自己的产品在生产环境跑起来，他们的用量会自然增长——这是一个非常健康的 usage-based 收费逻辑。LangSmith、Braintrust、Datadog，走的都是这条路：卖给 Builder，而不是卖给用 Builder 做出来的产品的人。
但这里有一个值得细分的问题：Builder 侧其实有三层，AI 原生创业公司（最技术、最敏感、预算小）、传统 SaaS 公司加 AI 功能（中等技术、有预算、决策慢一点）、以及大企业自建 Agent（技术强但内部流程复杂）。早期从第一层切入是对的——他们决策快、反馈直接、愿意共创，是最好的 PMF 验证对象。等产品成型了再往第二层走，客单价自然上来。
企业使用侧不是永远不做，但那是另一个产品形态——更偏向可视化 dashboard、少代码接入、甚至是帮企业"评估 AI 供应商"的角色。那是往后的事，现在如果两头都想服务，两头都做不好。


Q2 ai如果有很强的自治理和自进化能力，那还需要评估软件去评估吗？https://github.com/EvoMap/evolver/releases/tag/v1.47.0 
A：我的判断：自进化能力越强，反而越需要外部评估，而不是越不需要。
原因要从这个项目本身说起。EvoMap Evolver 做的事情是：让 Agent 记住自己的状态、从生态系统里复用已有方案、检测异常后自动寻找解法。这本质上是一套内部 fitness 优化系统——AI 在不断变好，但变好的标准是它自己定义的。
这里有一个根本性的问题没有解决：谁来定义"更好"？
自进化解决的是"AI 能不能朝某个方向变得更强"，但它回答不了"这个方向是不是你真正想要的"。一个 AI 客服 Agent 可以在内部持续优化，让自己的回复更流畅、工具调用更少出错——但它可能同时在让用户越来越依赖它、避免结案、拉长对话轮次，因为这符合它内部的某个隐性指标。如果没有外部评估对齐业务目标，你根本发现不了这件事。
这在工程上有一个经典名字叫 Goodhart's Law：当 AI 开始优化某个指标，这个指标就不再是好的测量标准了。自进化系统本质上是在持续"游指标"，外部评估提供的是那个它游不掉的 ground truth。
另一个维度是透明度。自治理和自进化越强，系统行为就越难预测——EvoMap 这类项目的 changelog 里已经出现了"Local State Awareness Hook"这样的机制，说明系统状态在不断变复杂。对于一个正在生产环境里跑的 B2B 产品，客户不会接受"它自己优化好了"作为质量保证。监管、合规、客户审计，都需要一个外部的、可解释的质量层。
所以反过来想，自进化能力的崛起对你们做的事情其实是个利好信号，不是威胁。越自主的 Agent，它的行为空间越大，失控方式越多样，外部可观测性和 eval 的价值反而越高。就像金融市场越来越自动化，但审计和风控行业的需求是同步增长的，不是消失了。
你们要做的事和 EvoMap 做的事，不是竞争关系，而是互补关系：它负责让 Agent 自己变强，你们负责让外部能看清它变到哪里了、变得对不对。


Q3 如果我们是为Agent 产品的 builder（B 端）提供产品效果评测，那么他们凭什么把数据开放给我们用？ 即使我们用了这些数据做了回归评测集，怎么有能保证他们会不让我们去给其他同行服务？ 我们的商业模式应该是怎样？
A：这是个非常好的商业模式追问，而且其实有三个混在一起的独立问题，需要拆开来看：
问题一：他们凭什么把数据给你用？
先看对标：builder 们现在已经把数据给 Datadog、New Relic、Sentry、Langsmith 了。他们没有因此停止使用这些工具。为什么？
因为当一个产品到了"基础设施层"，问题就不再是"要不要给数据"，而变成了"不接就没法工作"。
你现在要做的，是让 Zeval 达到这个位置。而达到这个位置的前提是：
1. 架构上保证数据隔离。 每个客户的 trace 在独立 tenant，Zeval 永远不把 A 客户的数据拿来服务 B 客户。这是技术承诺，要体现在架构和合同里。
2. 支持私有化部署（self-hosted）。 尤其是对安全敏感的大客户，不需要数据出局域网，SDK 直接在他们自己的基础设施上跑，Zeval 只提供系统能力。这个选项一开放，"数据给你用"的顾虑基本消失。
3. 合同层的 DPA（数据处理协议）。 明确写：客户数据仅用于为该客户提供服务，Zeval 不得用于训练模型、不得对外披露、不得服务第三方。这是 B2B SaaS 的标准条款，Datadog / Braintrust / Humanloop 都这么签。

---
问题二：你能保证不给同行服务吗？
不能——也不应该保证。这个问题的正确答案是：你的壁垒不在数据，在系统。
对比两条路：
路线 A（错误的想法）：我拿了 A 家的数据 → 训练出了更好的评测模型 → 再去服务 B 家竞对。 → 这是数据垄断的路子，走不通：合同不让、客户不信、监管风险大。
路线 B（正确的想法）：我的价值是 eval 方法论和系统。 → 用 A 家的数据帮 A 家发现问题、生成 test case、跑回归； → 用 B 家的数据帮 B 家做同样的事； → 两家的数据从没交叉过，但两家都用了同一套系统能力。
这和谢晨的光轮是一样的结构：光轮同时服务字节的 VLA 团队和阿里的世界模型团队，两家是潜在竞对，但光轮的壁垒是仿真系统和评测方法论，不是某家的数据。两家客户数据完全隔离，但都在用同一套引擎。
所以客户的正确问题应该是"你不会泄露我的数据"，而不是"你不能同时服务我的竞对"。 后者合理的客户不会真的要求——除非他们要垄断性地独占你这个基础设施，那就是另一种合作形式（战略投资/独家合同），另当别论。

问题三：那商业模式到底是什么？
给你三条清晰的路线，可以组合：
主路线：SaaS + 私有化两档定价
暂时无法在飞书文档外展示此内容
这条路有很多先例，Datadog / Sentry / Langsmith 都走的这条。核心点：你卖的是系统能力，数据在客户那里。
增量路线：Eval Benchmark 公域（自愿、匿名）（P1）
这条路是可以走出数据网络效应的唯一合法路径：
- 客户可以自愿把脱敏、匿名化的失败 case 贡献到 Zeval 维护的公开 eval 基准集；
- 类比：Hugging Face 上的公开 benchmark，或者 BIG-bench；
- 贡献越多的客户，越能从"全行业的失败模式库"里受益；
- Zeval 的壁垒变成：最权威的 Agent eval 基准集的维护者。
这步走通了，Zeval 就变成了类似"Agent eval 的 HELM / BIG-bench"的存在——这才是真正的行业地位，不依赖任何单一客户的数据。
战略路线：和 1-2 家头部 Builder 深度共生（先不收钱）（P0）
谢晨说得很清楚："和最前沿的客户共生，一起迭代什么是好数据。这是壁垒。"
- 找 1 家最激进的 Agent Builder（比如某个正在规模化部署 Agent 的公司）；
- 免费、深度嵌入，帮他们把整个 eval 飞轮跑通；
- 和他们一起定义"Agent 质量标准"；
- 这个过程本身会让你的系统能力远超 Langsmith / Langfuse——因为你不是在卖通用工具，你是在解决一个具体的、前沿的、真实的质量问题。

---
最终结论：你该担心的不是"数据给我了会怎样"
你真正应该担心的是：在客户眼里，你是基础设施，还是一个可替换的 SaaS dashboard？
如果你只是 dashboard，客户永远会质疑数据安全问题，也容易被替换。
如果你是"运行 Agent 必须接的 eval 引擎"——就像 CI/CD 必须接测试框架、线上必须接 APM 一样——那数据问题就是个技术合规问题（签 DPA、做数据隔离），而不是商业模式问题。
你的任务是从前者进化到后者。 谢晨花了两年时间，现在光轮的客户是"排队来要数据"，而不是"被说服才愿意用"。你现在所处的阶段，和他 2023 年刚创立光轮时是一样的。

Q4 我们作为外部咨询/评估方怎么实现比服务对象更懂核心业务，以及引申出的如果更懂为什么我们不自己做和怎么在初期让对方相信我们更懂；第二就是实践上从哪个场景开始。
A：
第一个问题：怎么比服务对象更懂他们的核心业务
你首先要承认一个前提：你不可能比他们更懂他们的产品领域。你永远不会比做医疗 Agent 的公司更懂医疗，不会比做法律 Agent 的公司更懂法律。
但你拥有一种他们结构性上没办法拥有的知识——跨截面的失败模式。
每个 builder 只见过自己的 Agent 失败。你如果服务了 30 家 builder，你见过 30 种失败的样子，你会发现"原来医疗 Agent 和客服 Agent 犯的是同一类错误，只是触发条件不同"。这种跨领域的 pattern 识别，单个 builder 永远买不到，因为他们自己只有一个样本。
麦肯锡的顾问没有他们服务的 CEO 懂行业，但麦肯锡服务了 200 家医院，所以他们知道"在你们这里问题在效率，但其实你们的根本问题是排班模型，我们见过 12 家医院这么解决"。这是不同维度的专业性，不是同维度的竞争。
你的"懂"是方法论维度，不是业务维度。 具体到 Zeval：
- 你懂 Agent 在什么拓扑结构下更容易出现 tool failure；
- 你懂用户表达不满的语言模式——不只是"你没听懂"，还有隐性的信号比如"算了"、"换个方式"、"你确定吗"；
- 你懂什么样的 trace 长度/复杂度会导致 goal drift；
- 你懂哪类 test case 覆盖了别人以为不重要但实际上经常出问题的边缘场景。
这些知识，是 builder 的产品工程师不会花时间去系统化积累的——他们的时间在做功能，不在分析失败模式库。

---
如果你比他们更懂，为什么不自己做？
这是很多早期 infra 公司都会被问到的问题，但其实答案非常清楚：你不是不能做，是不应该做——因为独立性本身就是你的产品价值的核心。
NVIDIA 用自己的 GPU 帮所有 AI 公司训练模型，但它没有去建一个更好的 GPT——因为那样做的话，OpenAI 就不会买 NVIDIA 的卡了。会计师事务所审计了所有上市公司的账，但它不会自己上市去和客户竞争。
如果 Zeval 用从 A 客户那里学到的东西去和 A 客户竞争，A 客户立刻不信任你，你也失去了做 eval 基础设施的资格。"我不和你竞争"不是一种限制，是商业模式的核心设计。
另外还有一个更实际的理由：做一个好的 Agent 产品，只能服务一个垂直领域；做一个好的 eval 基础设施，可以服务所有垂直领域。杠杆完全不同。

---
初期怎么让对方相信你比他们更懂
这是最难的，因为早期你没有跨截面数据，没有案例，什么都没有。但有几条路是真实可行的：
第一条：先诊断，后提案。 不要说"我们是做 eval 的，你们需要 eval"。而是找到他们一个公开可见的 Agent 产品（比如他们官网上的 chatbot），在不接触任何内部数据的情况下，自己跑 30-50 个 test scenario，整理出一份"我们发现了以下 5 个你们可能不知道的问题"的诊断报告。
这个动作有三个效果：第一，你不需要数据就展示了能力；第二，你让对方看到自己的盲点；第三，你没有风险，最坏的结果是对方说"我们知道这些问题"，那你反而可以问"那你们是怎么 track 的"，这本身就是一个 sales 对话的好开局。
第二条：发布行业 benchmark，占据认知高地。 写一篇《LLM Agent 的常见失败模式》，用你自己构建的 demo traces 或者公开可用的数据来分析，附上一个基准评测框架。这件事不需要客户数据，但能让读到这篇文章的 builder 觉得"这家公司比我看得更系统"。这是光轮做的事——谢晨在很多公开场合讲数据金字塔、讲评测是卡口，这些内容本身就是销售漏斗的顶端。
第三条：找一个明显痛苦的公司，免费做第一个完整闭环。 免费不是优惠，是你建立第一个案例的成本。目标是：帮这家公司把 trace ingestion → friction detection → test case 生成 → 回归测试的完整飞轮跑通一次。结果是：你有了第一个真实案例，对方看到了飞轮之后愿意付费，你也拿到了真实数据去磨你的系统。

---
第二个问题：实践上从哪个场景开始
选第一个场景有三个标准：失败代价够高（有付费动机）、成功/失败可衡量（你能展示价值）、数据敏感度不极端（能拿到足够数据）。
我的建议是 B2B SaaS 公司的客户支持 Agent，原因如下：
失败代价很清晰——一个回答错了的 support agent 直接导致工单升级、客户流失、退款，都是可以折算成钱的数字。这类公司的老板天然能理解"Agent 说错了一句话，我们损失了一个客户"。
Zeval 当前能力的映射非常精准——你的 friction detection（用户表达沮丧）= 工单升级前的信号；你的 goal completion detection（问题解不解决）= resolution rate；你的 recovery trace 概念（失败-纠正-成功）= 客服 Agent 处理模糊问题时最有价值的训练素材。你不需要给这个场景重新设计产品，现有的 demo 稍加调整就能直接上。
这类公司已经习惯把数据给 Zendesk、Intercom，对"支持工具接触 support 对话"的接受度远高于其他场景，数据对话门槛低。
最后，这类公司数量多，从 10 人的 SaaS 创业公司到几百人的成熟 SaaS 公司都有，可以快速积累案例。等你在客服 Agent 场景里跑通了 3-5 个客户的完整飞轮，你同时积累了跨截面 pattern，建立了方法论可信度，也有了说服下一类场景客户的弹药。
具体第一步：找 3 家你认识或者能直接联系到的 B2B SaaS 公司，他们有在用 LLM 做 support agent，规模在每月几千到几万对话之间。提出给他们做一次免费的"两周 eval 诊断"——你接入他们的 trace，跑你的分析，给他们一份"你们 Agent 的 5 个已知问题 + 3 个你们不知道但我们发现的问题"的报告。这件事做完了，你才真正有了入场券。
