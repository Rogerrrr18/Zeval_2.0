一、我们在做什么
人工智能的普及带来了新的复杂性。智能应用程序的行为可能导致意想不到的、不准确的、低质量的甚至有害的输出。这些问题最终会导致客户信任度下降、终端用户体验不佳以及成本增加。因此，团队需要对整个人工智能堆栈进行全面可视性评估，而不仅仅局限于延迟和错误等信号。可观测性必须评估诸如幻觉和偏差等问题，评估安全指标以检测和降低业务风险，并跟踪成本和使用指标以确保清晰的投资回报率。 

Zeval 正在构建 Agent 时代的评测基础设施。
Agent 产品正在大规模落地，但没有人能真正看清它在生产环境里做对了什么、做错了什么。现有工具要么只告诉你系统"挂没挂"，要么只能跑离线 Benchmark——没有人在做贴近真实业务、持续追踪交互质量的在线评测。
我们构建的，是一套 AI Agent 效果诊断与质量运营系统：
- 接入真实业务的对话日志（chatlog / trace）
- 自动分析交互质量：发现对话断点、用户流失节点、目标未完成的轮次
- 持续生成评测数据集，让 AI 产品团队第一次能用数据驱动 Agent 迭代
我们不只是给模型打一个抽象分数，而是帮产品团队看清：用户为什么流失、哪个环节出了问题、改了之后有没有变好。

二、我们解决的核心问题
Agent 产品在上线后，几乎所有团队都面临同一个困境：
 靠人工听对话找问题？成本极高，而且漏掉的比发现的多。
 靠 LLM 自动打分？指标和业务目标对不上，指导不了优化方向。
 靠埋点 + 数据分析？只能看到"用没用"，看不到"用得好不好"。
结果是：大量 Agent 产品在版本迭代中看不见问题，用户体验持续恶化，产品死在优化跟不上的地方。
我们的方法：从真实对话日志中，自动发现交互摩擦点（用户表达不满、重复追问、提前退出）、目标完成率（对话最终是否解决了用户意图）和恢复轨迹（Agent 出错后是否成功纠偏）。最终将这些分析结果转化为可回归测试的评测数据集，嵌入产品迭代流程。

三、我们的核心能力
隐式推断评测
 不要求用户显式打分。通过分析用户后续行为自动推断满意度：用户沉默 = 隐式接受；用户追加纠正 / 重复提问 = 隐式不满。这让评测轻量化，且比显式打分更接近真实体验。
恢复轨迹（Recovery Trace）识别
 自动标记"先失败 → Agent 纠偏 → 最终成功"的对话片段。这类数据是 Agent 改进的最高价值素材，也是我们区别于普通 trace viewer 的核心差异点。
评测数据飞轮
 生产失败 → 自动生成回归测试用例 → 持续入库 → 每次发版前自动跑回归。评测能力随着产品运营时间自动增长，而不是每次重新投入人工。
方法论而非数据依赖
 我们的壁垒在评测引擎和方法论，不依赖任何单一客户的数据。客户数据完全隔离，支持私有化部署。可复用的是评测框架，不是客户的业务数据。

数据主权原则
 客户数据始终归客户所有。Zeval 以 SDK 形式接入，数据在客户自己的基础设施上处理，我们不跨客户共享任何业务数据。我们卖的是评测系统能力，不是数据。
冷启动策略
 第一阶段：与 1-3 家头部 AI 产品团队深度共生，免费提供完整 eval 闭环，换取真实场景打磨机会。以案例建立方法论可信度。
 第二阶段：基于已有案例向同类场景横向复制，进入订阅付费阶段。
 第三阶段：推出可选的匿名 eval 基准库（客户自愿贡献脱敏数据），形成行业评测标准话语权。

六、为什么现在
- 57% 的企业已将 Agent 部署到生产环境，质量问题是最大的部署障碍
- LLM 可观测性平台市场：2026 年约 $27 亿美元，年增速 36%，到 2028 年将有 60% 的软件团队接入 eval/observability 工具（Gartner）
- 现有工具（Langsmith、Braintrust）定位于通用 trace 查看，缺少业务目标对齐、隐式推断、恢复轨迹这类针对 Agent 交互质量的深度能力
- 窗口期：Agent 产品正在从"能用"走向"好用"，质量运营的需求已经被市场验证，但专注于此的基础设施工具还是空白
