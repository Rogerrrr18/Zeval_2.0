2026/04/12-会议纪要
一、会议主题
围绕 Eval 业务的冷启动路径，讨论以下核心问题：
- 作为外部评估方，如何在不比客户“更懂核心业务”的前提下建立价值
- 第一阶段应该优先切入哪个细分场景
- 数据从哪里来、如何处理数据隔离与客户信任问题
- 商业模式应该如何设计
- MVP 的技术实现路径、分工与里程碑如何安排
二、核心结论
1. 价值定位：做“评测引擎 + 方法论”，不是替客户做业务
- 团队的核心不是比客户更懂业务本身，而是更擅长： 
  - 将业务目标拆成可评估的交互指标
  - 构建一套可复用的 evaluation 方法论与系统框架
  - 在不同客户之间复用“引擎”，而不是复用客户数据
- 数据应保持客户间隔离；可复用的是评测框架、流程和方法论。
- 可参考“光轮智能”服务竞对团队的模式：底层引擎复用，数据隔离。
2. 数据策略：前期以真机数据为主，仿真数据为辅
- 前期建议采用： 
  - 80% 真实用户数据 / chatlog
  - 20% 仿真数据用于长尾 case 补充
- 团队认为纯仿真数据在早期说服力不足，客户会担心覆盖真实场景的能力。
- 可先利用开源评测集或公开 CSV 数据，作为框架验证和方法论测试的起点。
3. 场景选择原则已明确
本次讨论收敛出 4 个优先筛选标准：
1. 业务指标是否明确 / 是否足够痛（优先级最高）
2. 开发难度与工作量
3. 数据是否相对干净、结构化、易处理
4. 市场空间是否成立 / 是否存在增量机会
4. 场景判断：优先选“目标明确、指标清晰、ToB/培训类”
讨论后形成的倾向：
- 优先考虑
  - 销售培训 / 新员工培训 / SOP 培训类场景
  - 模拟面试、语言陪练等目标相对明确的交互场景
  - ToB 销售助手等可跟单、可追踪、有明确业务链路的场景
- 暂缓或谨慎考虑
  - 低龄儿童玩具 / AI 学伴：年龄分层复杂、内容安全要求高、数据复杂度高
  - 电商客服：edge case 过多，用户行为难预测
  - 智能家居：当前以短指令为主，长对话价值还不明显
  - 恋爱输入法类 agent：多轮反馈机制弱，用户链路不闭环
  - 公开直播/粉丝互动：存在 prompt injection 与内容安全风险
5. 冷启动策略：先做 MVP + 横评内容，而不是先做大而全 Benchmark
- 公域 Benchmark 有价值，但前期投入大，不应作为 P0。
- 更现实的冷启动路径： 
  - 先从一个细分场景切入
  - 做出可跑通的 MVP
  - 结合横评/测评内容在社媒建立影响力
  - 再吸引潜在客户合作
- 结论：Benchmark 可放到完成 1~2 个合作后再启动。
6. 技术实现思路：先搭通用框架，再用具体数据验证
本次对 MVP 的共识：
- 先不被具体数据卡住，优先把 pipeline 跑通
- 输入格式先收敛为少量标准格式（如 CSV / JSON / TXT / MD）
- 输出先以图表 + 分析建议为核心，交付形式可为 HTML / PDF / PPT
- 是否做完整登录系统先不考虑，前期只需： 
  - 基础数据库
  - 内部可用的分析与展示框架
  - 面向客户展示的 landing page / demo 页面
7. 组织方式：先双人并行 AB Test，再收敛统一版本
- Roger 与章伟伟将各自跑一版框架 / pipeline，先验证流程可用性
- 第一阶段先不追求 UI 精细度，优先打通完整流程
- 之后再由前端设计方向进行统一美化与升级
三、MVP 技术框架（会议中确认的阶段）
M1. 发现业务指标与 chatlog 指标的映射关系
- 在具体业务场景中，识别核心业务目标
- 将其映射成一级/二级评估维度
M2. 设计评估逻辑分层
- 明确评估维度层级
- 区分客观指标与主观指标
M3. 定义输入格式
- 先限定兼容格式：CSV / JSON / TXT / Markdown
- 后续再逐步扩展兼容性
M4. 定义输出格式
- 输出内容：图表、元数据、分析建议、优化建议
- 输出形式：HTML / PDF / PPT
M5. 接入数据验证效果
- 导入首批真实或公开测试数据
- 清洗数据
- 验证框架可用性与分析结果稳定性
M6. 抽取可复用框架 / 方法论
- 沉淀 SOP
- 形成对新业务可迁移的评估流程
- 再进入优化阶段（数据清洗、prompt 调整、算法优化等）
四、关键讨论洞察
- 用户体验与长对话交互质量正在成为 agent 产品竞争重点。
- 真实业务里最难的是 edge case，不是主流程。
- 对外不必强调“我们更懂业务”，而要强调“我们能和客户一起把业务目标转成评估系统”。
- 客户是否付费，关键不在技术炫酷，而在是否解决“足够痛”的问题。
- 交付物不能只是跑分，还要有可执行的分析与优化建议。
