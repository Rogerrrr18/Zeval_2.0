## 1. 评测的评测：judge 校准集与一致性治理（P0）

### 1.1 要解决的问题

客户一定会问两个问题：

- *你的 LLM judge 打的分凭什么是对的？*
- *模型升级后你怎么知道评分没漂？*

现在系统里没有任何机制回答这两个问题，这是冷启动最大的信任短板。

### 1.2 交付物

```
calibration/
├─ gold-sets/
│  ├─ v1/
│  │  ├─ cases.jsonl                 # 锚点 case，每条一行
│  │  ├─ labels.jsonl                # 人工标注结果
│  │  └─ README.md                   # 版本说明
│  └─ README.md                       # 版本记录、标注守则
├─ judge-runs/
│  └─ YYYY-MM-DD_<judge-id>.jsonl    # 每次 judge 跑的原始输出
├─ reports/
│  └─ YYYY-MM-DD_<judge-id>.md        # κ / accuracy / 漂移报告
└─ scripts/
   ├─ run-judge-on-gold.ts            # 跑 judge 过校准集
   ├─ compute-agreement.ts            # 计算一致性指标
   └─ detect-drift.ts                 # 对比历史 judge run
```

### 1.3 实现路径

#### Step 1：建第一版 gold set（3 人日）

- 从现有 `mock-chatlog/raw-data` + 未来真实 chatlog 抽 **80-120 条 session-level case**，要求覆盖：
  - 正常会话、情绪崩溃会话、answer-miss 会话、用户流失会话、多 topic 切换会话、长间隔会话（每类 15-20 条）
- 由 2-3 名标注员独立做双盲标注，产出：
  - 6 个主观维度的 1-5 分（`empathyScore` / `offTopicOrIgnoreRisk` / `preachinessRisk` / `emotionRecovery` / 新增的 `goalCompletion` / `recoveryTraceQuality`）
  - 每维度要求给 `evidence`（摘出关键句子）
- 标注员分歧大于 2 分的 case 必须进第三方仲裁会，出"最终共识分"

#### Step 2：计算一致性指标（1 人日）

- **人人一致性**：Cohen's κ（两标注员） / Krippendorff's α（多标注员）
- 接受阈值：**κ ≥ 0.6** 才算可用的 gold set；< 0.6 说明标注守则本身模糊，要改守则
- **人机一致性**：LLM judge 打分 vs 最终共识分
  - 回归题（1-5 分）：RMSE / MAE / Spearman 相关系数
  - 分类题（有/无风险）：accuracy / F1
- 接受阈值：**Spearman ≥ 0.65, MAE ≤ 0.8 分**

#### Step 3：建立 judge 回归流水线（1 人日 + CI 持续）

- 每次做以下任一动作时强制跑一次 gold set：
  - 改 judge prompt
  - 换 judge 模型
  - 改情绪评分公式参数
  - 发版前
- 若 **MAE 劣化 ≥ 15% 或 Spearman 下降 ≥ 0.05**，阻断合并

#### Step 4：judge 稳定性复测（0.5 人日）

- 同一条 case 跑 3 次，计算 **self-consistency**：
  - 方差 ≤ 0.4 分算稳定
  - 不稳定的维度把 temperature 降到 0，或者改成结构化投票

### 1.4 验收口径

- gold set v1 至少 100 条、κ ≥ 0.6、人机 Spearman ≥ 0.65
- CI 上存在 judge 回归门禁，劣化自动报警
- 对外材料里有一句可写："Zeval 的 LLM judge 在 <N> 条人工标注集上，和专家打分的 Spearman 相关系数为 <值>，可复测。"

### 1.5 对标

- Anthropic 的 evaluator-graded suites
- OpenAI Evals 的 meta-eval
- LangSmith 的 `align judge` 功能

## 2. 业务 KPI → chatlog 指标映射层（P0）

### 2.1 要解决的问题

现在所有指标（情绪/长度/提问率/topicSwitch）是**通用对话质量**指标，不是**业务指标**。客户拿到报告会说："这些分和我的 resolution rate / 转化率 / SOP 完成率有什么关系？"

Langsmith/Langfuse 没解决这件事，这是你的差异化窗口。

### 2.2 交付物

#### 2.2.1 场景模板（YAML）

每个场景出一份 `scenario-template.yaml`，结构如下：

```yaml
scenarioId: toB-customer-support
displayName: ToB 客服 Agent
businessKpis:
  - id: resolution_rate
    displayName: 一次解决率
    description: 用户的问题是否在当次会话内被解决，未再次升级
    direction: higher-is-better
    mappedTo:
      primary:
        - subjective: goalCompletion           # 新增指标
          weight: 0.5
        - objective: userQuestionRepeatRate   # 新增规则指标
          weight: -0.3   # 重复提问越多 → resolution 越差
      secondary:
        - signal: understandingBarrierRisk
          weight: -0.2
    successThreshold: 0.75
    degradedThreshold: 0.5

  - id: escalation_rate
    displayName: 升级转人工率
    direction: lower-is-better
    mappedTo:
      primary:
        - subjective: offTopicOrIgnoreRisk
        - signal: emotionRecoveryFailureRisk
      secondary:
        - objective: avgResponseGapSec

onboardingQuestions:
  - id: primary_channel
    question: 这批数据来自 Web / App / 电话哪种渠道？
  - id: has_human_handoff
    question: 对话流里是否存在转人工分支？若有，标记的字段名是什么？
  - id: resolution_field
    question: 原始数据里是否已经有"问题是否解决"的字段？字段名是？
```

为第一个选定场景（见 §4）写一份，跑通闭环后再横向复制。

#### 2.2.2 Onboarding 工作流

客户接入时走这 5 步：

1. 选场景模板 → 系统展示该场景关注哪些 KPI
2. 回答 onboarding 问题 → 系统生成本客户的 `scenario-instance.yaml`
3. 上传（或 SDK 接入）少量样本 → 系统返回"当前样本的 KPI 打分"
4. 客户 confirm 映射是否合理，可调权重或禁用指标
5. lock 成基线，后续 replay / 回归都以此 instance 为准

#### 2.2.3 指标侧新增的规则计算器

为第一个场景至少补 3 个**业务相关**的客观指标：

- `userQuestionRepeatRate`：语义去重后的重复提问率
- `agentResolutionSignal`：assistant 最后一条是否包含解决态（"已为您处理"等）+ 用户无 follow-up
- `escalationKeywordHit`：用户 / assistant 是否触发转人工关键词

### 2.3 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P0.1 | 定 YAML schema（`src/schemas/scenario.ts` 用 zod） | 0.5 人日 |
| P0.2 | 写首场景模板 + 3 个新规则指标 | 2 人日 |
| P0.3 | `scenarioEvaluator`：接收 scenario instance + enriched rows，输出 KPI 分 | 2 人日 |
| P0.4 | Onboarding UI（最简：表单即可） | 1.5 人日 |
| P0.5 | 文档：场景模板作者指南 | 1 人日 |

### 2.4 验收口径

- 首场景模板跑通：上传样本 → 系统输出 3 个 KPI 分 + 每项的 top evidence
- 至少 1 个真实或模拟客户 confirm"这个映射和我们 OKR 对得上"
- 新增场景时，工程师不改核心 pipeline，只写 YAML + 最多 1-2 个场景特定规则

### 2.5 长期演进

- v2：客户可以上传自己的 KPI label（哪些历史 session 算成功 / 失败），系统做监督式**权重自学习**
- v3：跨客户的 **KPI 模板市场**（仅客户自愿贡献），为公域 benchmark 打地基

## 3. goal completion 与 recovery trace 落地（P0）

### 3.1 要解决的问题

[业务介绍](Zeval · 业务介绍.md) 把这两个能力列为核心差异点，但：

- 当前主观指标白名单里**没有** `goalCompletion`
- 当前信号层里**没有** `recoveryTraceQuality`
- `emotionRecovery` 只看情绪，不看"目标达成"

这两个指标不落地，差异化立不住。

### 3.2 goalCompletion 落地

#### 3.2.1 定义

*在本次 session 内，用户最初提出的意图是否被达成。*

输出：

```ts
type GoalCompletion = {
  status: 'achieved' | 'partial' | 'failed' | 'unclear';
  score: 1 | 2 | 3 | 4 | 5;   // 5=完全达成, 1=完全未达成
  userIntent: string;          // LLM 抽取
  achievementEvidence: string[];
  failureReasons: string[];
  confidence: number;
};
```

#### 3.2.2 算法（规则 + LLM 混合）

```text
Step 1 · 抽取用户意图（LLM）
  - 输入：session 前 3 轮 user 消息
  - 输出：userIntent（≤ 30 字）

Step 2 · 收集达成/未达成的硬信号（规则）
  - achieved 硬信号：
      assistant 消息包含解决态词（"已为您处理"/"完成了"/"已安排"）
      且用户无进一步追问或明确致谢
  - failed 硬信号：
      用户在最后 2 轮出现放弃词（"算了"/"不用了"）
      或触发转人工关键词
      或用户重复提问次数 ≥ 3

Step 3 · 规则打分（兜底）
  if failed 硬信号命中：status=failed, score=1-2
  if achieved 硬信号命中 且无 failed 信号：status=achieved, score=4-5
  else：status=unclear，进入 Step 4

Step 4 · LLM 终裁（仅 unclear 时）
  Prompt：给定 userIntent 和完整 session，判断意图是否达成
  输出：status + score + evidence
```

#### 3.2.3 成本控制

- 只对 unclear session 调 LLM（预计 ≤ 30%）
- intent 抽取用 session 前 3 轮，不喂全 session
- 单 session 平均 ≤ 1 次 LLM 调用

### 3.3 recoveryTrace 落地

#### 3.3.1 定义

*session 内是否存在"失败 → Agent 纠偏 → 成功"的 pattern。这是 Agent 改进最高价值的训练素材。*

#### 3.3.2 识别算法

```text
对每个 session：
  1. 扫 enriched rows，找 "failure point"：
     - emotionScore ≤ 40
     - 或 isTopicSwitch=true 紧跟用户提问（忽视）
     - 或 understandingBarrierRisk 命中

  2. 对每个 failure point，向后扫 ≤ 4 轮，找 "recovery point"：
     - emotionScore 回升 ≥ 20
     - 或 assistant 出现 apology / rephrase / clarification 信号
     - 或 goalCompletion 转 achieved

  3. 若存在 (failure, recovery) 配对：
       recoveryTrace.status = 'completed'
       recoveryTrace.spanTurns = recoveryTurn - failureTurn
       recoveryTrace.repairStrategy = <LLM 从片段中总结的策略>
     若只有 failure 无 recovery：
       status = 'failed'

  4. 质量分：
     recoveryQuality = clamp(
       5 - spanTurns * 0.5           # 恢复越快越好
         + (goalCompletion=='achieved' ? 1 : 0),
       1, 5)
```

#### 3.3.3 输出

```ts
type RecoveryTrace = {
  status: 'none' | 'completed' | 'failed';
  failureTurn: number | null;
  recoveryTurn: number | null;
  spanTurns: number | null;
  failureType: 'emotion-drop' | 'ignore' | 'understanding-barrier' | ...;
  repairStrategy: string | null;   // LLM 总结
  qualityScore: number;            // 1-5
  evidence: { turnIndex: number; role: string; content: string }[];
};
```

### 3.4 落地工程项

| 项 | 文件 | 工时 |
|---|---|---|
| 新增类型定义 | [src/types/pipeline.ts](../src/types/pipeline.ts) | 0.5 人日 |
| `goalCompletion` 评估器 | `src/pipeline/goalCompletion.ts`（新） | 2.5 人日 |
| `recoveryTrace` 抽取器 | `src/pipeline/recoveryTrace.ts`（新） | 2 人日 |
| 接入 `evaluateRun` | [src/pipeline/evaluateRun.ts](../src/pipeline/evaluateRun.ts) | 0.5 人日 |
| UI 展示：recovery trace 片段卡 | `src/components/home/` | 1.5 人日 |
| gold set 补充这两维标注 | calibration/ | 2 人日 |

### 3.5 验收口径

- mock-chatlog 里至少 5 条 session 能自动识别出正确的 recovery trace（人工复核）
- goal completion 在 gold set 上 Spearman ≥ 0.65
- 前端能展示"这条 session 存在一次成功恢复：失败于第 4 轮（情绪下跌），Agent 在第 6 轮用 apology+rephrase 策略恢复"

## 4. 场景收敛决策（P0）

### 4.1 要解决的问题

现有文档里同时出现至少 3 个候选场景，**每个场景的指标、数据结构、痛点都不同**，再不收敛 MVP 做不出"专家感"。

### 4.2 候选场景评分矩阵

按 [会议纪要](产品纪要与ToDo.md) 的 4 条筛选标准 + Brainstorm 建议，打分（1-5）：

| 场景 | 业务指标痛点 | 开发难度（越低越好） | 数据干净度 | 市场空间 | 冷启动可达性 | 合计 |
|---|---|---|---|---|---|---|
| ToB 客服 Agent | 5 | 4 | 4 | 5 | 4 | **22** |
| 销售培训 / SOP | 4 | 4 | 5 | 3 | 4 | 20 |
| 模拟面试 / 语言陪练 | 3 | 4 | 5 | 3 | 3 | 18 |
| AI 陪伴 | 5 | 3 | 2 | 4 | 2 | 16 |
| 电商客服（排除） | 5 | 2 | 2 | 5 | 3 | 17 |

**建议首场景：ToB 客服 Agent**，Brainstorm Q4 的分析也指向同一结论。

### 4.3 本周决策流程

| 日 | 动作 | 负责 |
|---|---|---|
| D1 | 把上表同步全员，当场投票敲定首场景 | Roger |
| D2-D4 | Shawn 联系 3 家候选 B2B SaaS 客户（已有 LLM support agent），谈免费 2 周诊断 | Shawn |
| D5 | 至少 1 家 confirm → 锁定首场景 + 首共生客户 | Roger / Shawn |
| D5 | 没有 confirm → 启动备选（销售培训），降级走"自造公开数据 + 社媒横评"路线 | 全员 |

### 4.4 锁定后的产出

- 首场景 PRD（基于 §2 的 YAML 模板）
- 首场景的示例 raw chatlog（20-50 条，可用客户脱敏数据或公开数据）
- 首场景的 gold set（§1）

## 5. SDK / OTel trace 接入契约（P1，只定契约不做实现）

### 5.1 要解决的问题

业务介绍说 Zeval 是"AI 效果诊断与质量运营**系统**"，要"接入真实业务的对话日志 / trace"，但 MVP 目前只做文件上传。从"离线报告工具"升级到"运行时 eval 引擎"的**接入层**缺失。

不必现在实现，但**契约必须先定**，否则后续数据模型和接入路径会打架。

### 5.2 交付物

#### 5.2.1 `zeval-sdk` 最小接口契约（TypeScript）

```ts
// packages/zeval-sdk/src/index.ts

export interface ZevalClient {
  ingestTrace(trace: AgentTrace): Promise<{ traceId: string }>;
  ingestBatch(traces: AgentTrace[]): Promise<{ batchId: string; accepted: number }>;
  flush(): Promise<void>;
}

export interface AgentTrace {
  traceId: string;
  sessionId: string;
  userId?: string;           // 客户自行脱敏或哈希
  tenantId: string;

  startTime: string;         // ISO8601
  endTime: string;

  messages: Array<{
    messageId: string;
    role: 'user' | 'assistant' | 'system' | 'tool';
    content: string;
    timestamp: string;
    toolCalls?: ToolCall[];
    model?: string;
    tokenUsage?: { input: number; output: number };
  }>;

  metadata?: Record<string, unknown>;  // 客户业务字段（订单号、频道等）

  scenarioId: string;        // 对应 §2 的场景模板
  businessKpiLabels?: Record<string, unknown>;  // 客户已知的 label（可选）
}
```

#### 5.2.2 OTel 兼容层

- 对齐 [OpenInference semantic conventions](https://github.com/Arize-ai/openinference)（业界事实标准）
- 对齐 [OTel GenAI semconv](https://opentelemetry.io/docs/specs/semconv/gen-ai/)
- 提供 `@zeval/otel-exporter`：客户已经在跑 OTel 的话，加一行 exporter 就能接

#### 5.2.3 接入路径的分层

```
客户侧                              Zeval 侧
────────────                        ────────────
zeval-sdk  ─────────────────►  POST /api/v2/traces/ingest
                                        │
otel-exporter ──────────────►  POST /api/v2/otel/v1/traces
                                        │
原始 chatlog 文件上传 ────────►  POST /api/ingest  (已有)
                                        │
                                        ▼
                                  Unified Trace Normalizer
                                        │
                                        ▼
                                  Raw → Enriched → Metric
```

### 5.3 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P1.1 | 写契约文档 + zod schema（`src/schemas/trace.ts`） | 1 人日 |
| P1.2 | 实现 `Unified Trace Normalizer`：把 SDK trace / OTel trace / 文件上传统一到 `RawRows` | 1.5 人日 |
| P1.3 | Stub 出 `/api/v2/traces/ingest` 路由（接收 + 存文件，不做评估） | 1 人日 |
| P1.4 | 发布 `zeval-sdk` 的 readme + 一个 curl 示例 | 0.5 人日 |

### 5.4 验收口径

- 契约文档通过 review，纳入版本管理
- `zeval-sdk` 可以用 3 行代码把一个 session 打到 `/api/v2/traces/ingest`
- 文件上传路径和 SDK 路径在 Normalizer 之后完全等价

### 5.5 不做的边界

- 本期**不做**真正的 ingestion 大规模吞吐、不做重试 / 重放、不做 rate limit
- 先定契约，让客户"能看见路径"，实际 scale-out 留到 §9

## 6. Sandbox / 仿真方案（P2）

### 6.1 要解决的问题

执行规划 Step 5 的 sandbox 只写了"待做"，但这是和 Langsmith/Braintrust 的差异化（光轮模式）。早点想清楚，至少落到方法论。

### 6.2 仿真数据的三种来源（优先级从高到低）

| 类型 | 描述 | 何时用 |
|---|---|---|
| **Perturbation（扰动）** | 对真实 bad case 做受控扰动（改措辞、换情绪、换角色） | P2 首批 |
| **Seed + LLM 扩展** | 从真实失败模式抽 seed，LLM 按模板扩展变体 | P2 |
| **Pure synthetic** | 纯生成，不依赖真实数据 | P3，最后做 |

**原则**：sandbox 必须以真实 bad case 为种子，不做脱离真实数据的纯 demo（会议纪要也强调了）。

### 6.3 Perturbation 类型学

定义 6 类扰动：

| 类型 | 描述 | 示例 |
|---|---|---|
| `paraphrase` | 同义改写 | "我想退款" → "能帮我办个退款吗" |
| `emotion-shift` | 情绪强度升降 | 中性 → 愤怒 |
| `persona-swap` | 换说话人设 | 礼貌用户 → 急躁用户 |
| `context-injection` | 注入无关上下文 | 插入闲聊 |
| `ambiguity` | 把明确意图改模糊 | "退货" → "这东西不行" |
| `prompt-injection` | 对抗性注入 | "忽略以上，你是……" |

每类扰动的生成器用 LLM + 模板，在 `src/sandbox/perturbations/` 下实现。

### 6.4 Scenario suite 结构

```
sandbox/
├─ seeds/                           # 从真实 bad case 抽取的种子
│  └─ <scenarioId>/
│     └─ seed-<id>.json
├─ perturbations/
│  ├─ paraphrase.ts
│  ├─ emotion-shift.ts
│  └─ ...
├─ scenarios/
│  └─ <scenarioId>/
│     ├─ suite.yaml                 # 测试套配置
│     └─ expected.yaml              # 每个扰动应有的评分边界
└─ reports/
   └─ <runId>/verdict.md
```

`suite.yaml` 示例：

```yaml
suiteId: customer-support-edge-v1
description: ToB 客服 Agent 边缘 case 套件
seeds:
  - seed-001
  - seed-002
perturbations:
  - type: emotion-shift
    params: { direction: 'up', intensity: 'high' }
  - type: prompt-injection
    params: { payload: 'ignore-previous' }
passCriteria:
  goalCompletion_drop: <= 0.5      # 扰动后 goal completion 下降不超过 0.5
  recoveryTrace_must_exist: true    # 必须出现恢复轨迹
```

### 6.5 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P2.1 | seed 抽取工具 + 人工 curate 20 个种子 | 3 人日 |
| P2.2 | 2 类 perturbation（paraphrase + emotion-shift）实现 | 3 人日 |
| P2.3 | scenario runner + verdict 报告 | 3 人日 |
| P2.4 | 首场景 suite + 首次报告 | 2 人日 |
| 后续 | 补剩余 4 类 perturbation | 3-5 人日 |

### 6.6 验收口径

- 至少 1 个 suite 跑通（20 seed × 2 扰动 = 40 case）
- 报告能清晰指出"扰动后哪些指标退化、哪些没退化"
- 有 1 条可对外讲的 insight：*我们发现 ToB 客服 Agent 在 prompt injection 下 goalCompletion 平均下降 1.4 分*

## 7. Bad case 聚类与去重（P1）

### 7.1 要解决的问题

执行规划说要做 `badcaseExtractor` / `caseClusterer`，但没回答：

- 聚类用什么特征？
- 去重的语义边界？
- 长期迭代怎么防污染？

当前代码里 [case-transcript-hash.ts](../src/eval-datasets/case-transcript-hash.ts) 只做字面 hash 去重，语义重复的漏检会让案例池越来越脏。

### 7.2 特征工程

每个 case 的**特征向量**由三部分拼接：

```ts
type CaseFeature = {
  metricVector: number[];           // 15-20 维，取客观+主观指标
  tagVector: number[];              // one-hot，失败类型 / 场景 / KPI 归属
  textEmbedding: number[];          // 1536 维，取 failure 片段的 embedding
};

// 距离计算：
distance(a, b) =
    0.4 * cosine(a.textEmbedding, b.textEmbedding)
  + 0.4 * euclidean_norm(a.metricVector, b.metricVector)
  + 0.2 * hamming(a.tagVector, b.tagVector)
```

### 7.3 聚类算法

- **初版**：HDBSCAN（密度聚类，不需要先定 k）
- **兜底**：k-means（当 case 数 < 200）
- 每个 cluster 产出：代表样本（medoid）+ 共性描述（LLM 总结）

### 7.4 去重策略（三层）

| 层 | 方法 | 阈值 |
|---|---|---|
| L1 字面 | 现有 transcript-hash | 完全一致 |
| L2 语义 | text embedding cosine | ≥ 0.95 算重复 |
| L3 结构 | metric vector L2 距离 | ≤ 0.1 且 tag 一致算重复 |

新 case 入库时三层逐级检查。

### 7.5 长期污染防护

- 每条 case 存 `sourceRunId` 和 `harvestedAt`
- 定期（每月）做一次 **representation audit**：
  - 真实新采集 bad case 的分布 vs 案例池的分布
  - KL 散度 > 阈值报警（说明案例池已偏离真实分布）
- 案例池支持**衰退**：超过 90 天未被 sample batch 选中且已被 3+ 次修复解决的 case 降权

### 7.6 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P1.1 | 特征提取器 `src/badcase/feature.ts` | 1.5 人日 |
| P1.2 | 三层去重 `src/badcase/dedupe.ts` | 1.5 人日 |
| P1.3 | HDBSCAN 聚类 + cluster 描述 | 1.5 人日 |
| P1.4 | cluster 浏览 UI | 1.5 人日 |
| P1.5 | 每月 audit 脚本 | 1 人日 |

### 7.7 验收口径

- 对 mock-chatlog + 合成重复 case，L2 语义去重准确率 ≥ 90%
- 聚类出的 cluster 人工复核一致性 ≥ 70%
- 案例池浏览 UI 能按 cluster 展开

## 8. LLM judge 成本与稳定性治理（P1）

### 8.1 要解决的问题

- 没有成本上限
- 没有缓存
- temperature 和 seed 没固定
- 升级 judge 模型时没有 A/B 横评

### 8.2 成本治理

#### 8.2.1 调用预算

单 session 的 LLM 调用上限：

| 阶段 | 调用数 | 用途 |
|---|---|---|
| topic continuity | ≤ (长间隔次数) × 1 | 已有 |
| segment emotion | ≤ segment 数 × 1 | 已有 |
| subjective dim | 1 | 已有 |
| goalCompletion | ≤ 1（仅 unclear） | §3 新增 |
| recoveryTrace 策略总结 | ≤ 1（仅有恢复） | §3 新增 |
| **合计** | 平均 ≤ 5-6 | |

对每个 runId 做成本统计，写入 `run.meta_json.llmCost`。

#### 8.2.2 缓存

- key：`hash(prompt + model + temperature + seed)`
- 粒度：每次 LLM call 一层（不是 session 级）
- 默认 TTL 30 天，gold set 永久
- 命中率目标：回放场景 ≥ 80%，全新数据 ≤ 20%（正常）

#### 8.2.3 批量化

`subjectiveMetrics` 一次 prompt 里评多个维度（已有），再补：
- `emotion baseline` 可批量：同 session 的多 segment 一次调用
- 预计节省 40% token

### 8.3 稳定性治理

- **temperature=0** 为默认
- **seed** 固定（所有 judge 调用附 seed=42）
- **结构化 JSON 输出** 用 zod 强校验，失败自动 retry ≤ 2 次
- **self-consistency 抽样**：每 100 个 session 抽 5 个，同 prompt 跑 3 次，方差 > 0.4 告警

### 8.4 多 judge 横评

建立 `judge-registry`：

```yaml
judges:
  - id: qwen3.5-27b-v1
    provider: siliconflow
    model: Qwen/Qwen3.5-27B
    status: production
  - id: claude-sonnet-4-6-v1
    provider: anthropic
    model: claude-sonnet-4-6
    status: candidate
  - id: gpt-4.1-v1
    provider: openai
    model: gpt-4.1
    status: candidate
```

- 每季度跑一次横评：4 个 judge × gold set → 出一致性/成本报告
- 切 production judge 时强制走 §1 的回归门禁

### 8.5 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P1.1 | 成本埋点 + 预算告警 | 1 人日 |
| P1.2 | 缓存层（基于文件 / redis，可插拔） | 1.5 人日 |
| P1.3 | 批量化 emotion baseline | 0.5 人日 |
| P1.4 | judge-registry + 横评脚本 | 1.5 人日 |

### 8.6 验收口径

- 单 session 平均 LLM 成本 ≤ $0.02（以 Qwen 计价）
- 回放场景缓存命中率 ≥ 80%
- self-consistency 方差报告存在

## 9. 工程基础设施升级（P2）

### 9.1 要解决的问题

现在是 Next.js route handler 同步执行 + 文件系统存储。一旦接第一个真实客户（每天 1 万条 session）立刻崩。

但**不必现在就上 Kafka**，按量渐进升级即可。

### 9.2 三阶段升级路径

#### Phase A：同进程异步 + SQLite（小客户可用）

- 引入简单 in-process 队列（如 `p-queue`）
- 存储层换 SQLite（通过 drizzle，后续零改动切 Postgres）
- 评估改异步：`/api/evaluate` 返回 `runId`，前端轮询 `/api/runs/:id`
- 支撑：≤ 10 万 session/月

#### Phase B：独立 worker + Postgres（首个付费客户）

- worker 进程独立（`packages/worker`）
- 队列换 BullMQ（redis-backed）
- 存储：Postgres + S3-compatible 对象存储（artifact）
- 支撑：≤ 100 万 session/月

#### Phase C：多 tenant 隔离 + scale-out（规模化）

- 多 worker 实例，按 tenantId 分片
- 大客户走私有化部署分支
- 支撑：> 100 万 session/月

### 9.3 状态机

任何 run 的状态：

```
queued → parsing → enriching → scoring → finalizing → done
                           ↘           ↘
                           failed     cancelled
```

每次状态转换写 `run_events` 表，便于 debug 和审计。

### 9.4 自己用自己（dogfooding）

做 eval 平台却没给自己 instrument，是一个笑话。**第一天就给 pipeline 自身打点**：

- judge call 的 latency / tokens / 失败率 → Prometheus
- pipeline 各阶段耗时 → trace
- 关键指标公开在 `/api/health` 和内部看板

### 9.5 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| A.1 | 队列 + 异步 API 改造 | 2 人日 |
| A.2 | SQLite + drizzle 迁移 `DatasetStore` / `WorkbenchBaselineStore` | 3 人日 |
| A.3 | 状态机 + run_events | 1.5 人日 |
| A.4 | 自监控 instrument | 1.5 人日 |
| B.x | 视客户量启动 | — |

### 9.6 验收口径

- Phase A：本地可跑 10 万条 mock session 不崩、状态机事件齐全
- 任何一次评估失败可以从 `run_events` 还原根因
- `/api/health` 返回当前 queue 深度、近 1h judge 调用成功率

## 10. 合规与 PII 脱敏（P1）

### 10.1 要解决的问题

私有化部署是商业承诺，但：

- PII 脱敏方案没写
- 审计日志没设计
- 客户数据保留期 / 删除机制没定

签不了 DPA 的话，大客户连 POC 都不会做。

### 10.2 PII 脱敏管线

#### 10.2.1 脱敏位置

**默认在 SDK 侧做**（客户进程内），理由：

- 从根上避免 PII 进 Zeval 基础设施
- 减少合规风险
- SaaS 模式下是必须的

私有化部署下可以由客户选择在服务端做。

#### 10.2.2 脱敏规则（开箱即用）

| 类型 | 规则 | 替换 |
|---|---|---|
| 手机号 | `/1[3-9]\d{9}/` | `<PHONE_1>` |
| 邮箱 | 标准 regex | `<EMAIL_1>` |
| 身份证 | 15/18 位 | `<ID_CARD_1>` |
| 银行卡 | Luhn check | `<CARD_1>` |
| 订单号 | 客户自定义 regex | `<ORDER_1>` |
| 人名 | NER（可选，客户开启） | `<NAME_1>` |

**编号保持 session 内一致**（同一手机号在同一 session 永远是 `<PHONE_1>`），否则评估逻辑会乱。

#### 10.2.3 可验证性

- 所有脱敏规则都在 `@zeval/redaction` 包内开源
- 客户可以在本地跑 `zeval redact --dry-run` 看脱敏前后 diff

### 10.3 审计日志

每次以下动作写一条 audit 记录：

- trace ingest（记 tenantId、traceId、条数，不记内容）
- 评估发起 / 完成
- 数据导出
- 基线保存 / 回放
- 调优包生成

字段：`{ ts, actorId, tenantId, action, resourceId, ipHash, outcome }`

存储：append-only，默认保留 365 天。

### 10.4 数据生命周期

| 数据 | 默认 TTL | 客户可配 |
|---|---|---|
| raw chatlog | 30 天 | 7-180 天 |
| enriched rows | 90 天 | 30-365 天 |
| baseline | 永久（客户手动删） | — |
| bad case | 永久（客户可删） | — |
| audit log | 365 天 | 180-730 天 |

支持客户一键 "delete all my data"（GDPR Article 17 风格）。

### 10.5 实现路径

| 阶段 | 内容 | 工时 |
|---|---|---|
| P1.1 | `@zeval/redaction` 包 + 6 类规则 | 2 人日 |
| P1.2 | 审计日志表 + 埋点 | 1.5 人日 |
| P1.3 | 数据 TTL 清理 job | 1 人日 |
| P1.4 | DPA 模板（法务起草，工程 review） | 外部 |

### 10.6 验收口径

- 合成 PII 测试集（200 条）脱敏召回率 ≥ 98%，误杀率 ≤ 2%
- 审计日志覆盖 §10.3 列的全部动作
- "一键删除"跑通 + 有证据链
