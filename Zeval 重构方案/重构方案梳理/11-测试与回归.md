# 11-测试与回归

本文档定义重构后 Zeval 在工程侧的测试与回归约束，确保改动可被验证、不退化。范围只覆盖重构必备项，不要求一次性建成完整测试金字塔。

## 11.1 测试分层

| 层级 | 范围 | 要求 |
| --- | --- | --- |
| 类型与编译 | TypeScript | `npx tsc --noEmit` 必须无错误 |
| Lint | ESLint | `npm run lint` 必须无错误 |
| 单元测试 | 关键纯函数 | 客观指标、隐式信号、规则情绪权重必须有单元测试覆盖 |
| 管线 smoke | `evaluate` 主链路 | 至少一份 fixture CSV 可成功跑通 enrich + objective + subjective |
| 校准回归 | LLM Judge | calibration gold set 一致率必须达到阈值（建议 0.85） |
| 端到端 smoke | API 路由 | `npm run smoke:e2e:*` 系列脚本可跑通主要业务路径 |
| Supabase 投影 smoke | DB 写入 | 验证 evaluation projection 完整写入新表 |

## 11.2 必须保留的脚本

| 脚本 | 用途 |
| --- | --- |
| `npm run smoke:e2e:clean` | 干净状态下的端到端冒烟 |
| `npm run smoke:e2e:bad` | badcase 上传 + 评估 + 调优包 |
| `npm run jobs:work:once` | 异步任务 worker 单次跑通 |
| `npm run calibration:ci` | LLM Judge 校准 CI |
| `npm run db:smoke:evaluate-projection` | 投影写入 smoke |
| `npm run db:smoke:stores` | dataset / baseline 类 store 写入 smoke |
| `npm run generalization:ci` | ⚠️ Post-MVP：泛化验证 CI（seen/holdout 分离 + bootstrap CI + 跨能力 guard），Phase 2（P4+）上线 |
| `npm run generalization:report` | ⚠️ Post-MVP：泛化验证报告（per-capability gen_coeff + CI），Phase 2（P4+）上线 |

重构后这些脚本必须继续可用。新脚本按相同命名规则添加，不要散落到不同入口。

## 11.3 客观指标回归

- 维护一份至少 5 条 session 的 fixture CSV，覆盖：长会话、短会话、断点、长间隔、纯客服与纯销售两类风格。
- 客观指标输出的快照（snapshot）必须稳定，任何代码改动改变快照都需在 PR 描述中给出原因。
- 客观指标 snapshot 写入 `tests/fixtures/objective-baseline.json` 或等价位置，禁止与 LLM 输出混合。

## 11.4 主观指标回归

- 准备 calibration gold set：人工标注的 session 集合，含 `dimension`、`expectedScoreRange`、`expectedReasonHints`。
- 每次主观流程或 prompt 改动后跑一次 `npm run calibration:ci`：
  1. 调用现行流程产出主观结果。
  2. 与 gold set 比对：计算分数误差、reason 关键词命中率。
  3. 输出报告并保存到 `calibration/reports/`。
- 一致率低于阈值视为回归失败。

## 11.5 LLM 调用稳定性

- 至少包含两类 mock：`http_failed` 与 `parse_failed`，用于校验降级路径是否被走到。
- mock 测试必须验证：
  - 降级后 `subjective_signals.source = "fallback" or "rule"`；
  - 不会抛出未捕获异常导致整个 run 失败；
  - `judge_runs` 里有失败记录与原始响应。

## 11.6 在线回放回归

- 准备一份固定 baseline + mock customer reply API。
- 回放 smoke 必须验证：
  - `replay_turns` 行数与 baseline 中 user turn 数一致；
  - 失败 turn 不阻断整次 run；
  - `run_comparisons` 至少包含 `objective` 与 `subjective` 两组对比。
- 回放对比展示 N（参与比对的 case 数）；小样本下需在 UI 中明确提示样本量不足，避免误读。
- ⚠️ **Post-MVP**：`min_n_per_capability` 门禁、bootstrap CI 与跨能力 guard 见 11.12 泛化验证机制，Phase 2（P4+）实施。

## 11.7 评测集与 sample batch 回归

- 创建 case 后再创建相同内容的 case：必须命中 exact duplicate 拒绝。
- 修改空白/标点后创建：触发 near duplicate 警告。
- sample batch：当 goodcase 不足 target 时返回部分批次和警告。

## 11.8 回放 / 调优包工作流回归

- 从评估 → harvest badcase → 创建调优包 → 生成 4 件套 artifact → validation run 回放：
  - 整条链路必须可在 smoke 脚本中一次跑通；
  - artifacts 内容字段不允许出现 `workspaceId`、`zerore_records` 等过时字段。

## 11.9 Supabase 投影 smoke

- 一次 evaluate 后验证：
  - `evaluation_runs` 1 行；
  - `topic_segments` 行数 = 实际 segment 数；
  - `objective_signals` 行数 ≥ 客观指标数；
  - `subjective_signals` 行数 = 维度数；
  - `risk_tags` 行数 ≥ 隐式信号数；
  - `evidence_spans` 行数 ≥ 1。
- smoke 失败视为重构进度被打断，必须先恢复才能合入。

## 11.10 CI 建议（仅推荐路径）

- 在 PR 触发：`tsc --noEmit` + `lint` + 单元测试 + 客观指标 snapshot。
- 在 main 触发：增加 calibration:ci 与投影 smoke。
- 部署前在 staging 运行端到端 smoke。
- 不在重构方案中强制 CI 工具（GitHub Actions / Vercel / 自建均可），以推荐为主。

## 11.11 验收

1. 任何 PR 都能给出对应的 smoke / 单元测试覆盖范围。
2. 任何 prompt / 规则改动都能在 calibration 报告中量化影响。
3. 任何 API 契约改动都能在端到端 smoke 中验证响应字段不缺失。
4. 任何数据库 schema 改动都能在投影 smoke 中验证写入完整。
5. ⚠️ **Post-MVP**：任何 validation run 都能在 `validation_generalization_reports` 中查到 `gen_coeff`、bootstrap CI、跨能力 guard 结果（Phase 2 实施）。
6. ⚠️ **Post-MVP**：N < 10 的 capability 桶不允许产出 pass/fail 结论，只能是 `INSUFFICIENT_EVIDENCE`（Phase 2 实施）。

## 11.12 泛化验证机制（M1–M5）⚠️ Post-MVP — Phase 2（P4+）规划

> **本节为 Phase 2 规划，MVP（P1–P3）阶段不实施。** 相关表（`validation_generalization_reports`）、字段（`eval_cases.holdout`）、脚本（`generalization:ci/report`）与 API（`/api/validation-runs/[id]/generalization`）已定义于 schema 以备向前兼容，但 MVP 阶段不写入数据、不在 CI 中强制执行。

当前 offline_eval 只能证明"在见过的 badcase 上改善了"，无法证明能力本身泛化。本节定义 5 个统计保护机制，确保修复的改善具有举一反三的泛化能力。

### M1：Seen / Holdout 强制分离

每个 `capability_dimension` 的案例池**必须双轨维护**：

- `holdout = false`（seen set）：修复者可以查看和使用，用于开发、调参；
- `holdout = true`（holdout set）：**永远不对修复者暴露**，只在最终 acceptance gate 时读取，holdout 比例 ≥ 30%，per-capability 最少 10 条。

`eval_cases` 表增加 `holdout boolean not null default false` 字段；`sample_batches` 构建时按 capability 分桶后做 stratified split，保证两组覆盖。

**泛化系数（gen_coeff）**：

```
gen_coeff = improvement_rate_on_holdout / improvement_rate_on_seen
```

| gen_coeff | 结论 |
| --- | --- |
| ≥ 0.8 | 泛化良好，通过 |
| 0.5–0.8 | 局部改善，输出 `GENERALIZATION_WARNING`，要人工 review |
| < 0.5 | 疑似过拟合，拒绝 |

### M2：最小样本量门禁（min_n_per_capability）

`acceptance_gate.offline_eval` 新增字段：

```json
{
  "min_n_per_capability": 10,
  "insufficient_evidence_policy": "block"
}
```

- 每个 capability 桶的 executed case 数 < `min_n_per_capability` → 输出 `INSUFFICIENT_EVIDENCE`，不输出 pass/fail；
- `insufficient_evidence_policy = "warn"` 时只记录警告，不阻断（用于探索阶段）；
- **不允许用合计通过率掩盖某个桶的样本不足**。

### M3：合成变体探针（与 14 号文档联动）

每批 badcase 入库时，**自动合成 K=5 个同能力、不同场景的变体 case**，打标 `source: synthesized`，强制进入 **holdout set**（`holdout = true`），不出现在 seen set。

```
真实 badcase（instruction_following，退款场景）
  → synthesized × 5（instruction_following，预约/投诉/咨询场景）
  → 全部进 holdout，修复者不可见
```

修复只解决特定场景但不覆盖能力本质 → 合成变体上仍失败 → `gen_coeff` 低 → 拒绝。

触发合成探针的条件：一批 badcase 入库且 `holdout` 剩余量 < seen × 0.43（保证 holdout ≥ 30%）时，自动调用 `POST /api/eval-datasets/synthesize` 补充，`targetHoldout = true` 字段指定目标为 holdout set。

### M4：全量 Capability 跨维度回归（Cross-Capability Guard）

每次 offline_eval 或 validation run 通过 seen_set 后，**必须追加跑一次全量 12 维 capability 检查**：

```
for each capability in 12_dimensions:
  run replay on holdout_set[capability]
  compare to baseline_score[capability]
  if delta ≤ -ε (ε = 0.03):  → flag capability as REGRESSED
```

任何 capability 退化超过 ε → 整包 fail，不管 target metrics 是否通过。防止"修好 A，搞坏 B"被聚合均值掩盖。

结果写入 `validation_generalization_reports.cross_capability_guard`：

```json
{
  "multi_turn_coherence": { "delta": 0.04, "passed": true },
  "style_format": { "delta": -0.05, "passed": false },
  ...
}
```

### M5：Paired Bootstrap CI（统计显著性）

Case-level delta **不报点估计，改报 95% bootstrap CI**：

```json
{
  "mean_delta": 0.08,
  "ci_95_lower": 0.02,   // 必须 > 0 才算显著改善
  "ci_95_upper": 0.14,
  "bootstrap_n": 2000,
  "n_cases": 20
}
```

- `ci_95_lower ≤ 0` → 即使 `mean_delta > 0` 也只输出 `TRENDING_NOT_SIGNIFICANT`，不通过门禁；
- bootstrap 以 case 为单位有放回抽样，重复 2000 次，不依赖正态分布假设；
- 结果保存到 `validation_generalization_reports` 表，供 Copilot 与 PM 复查。

### M1–M5 实施路线

| 机制 | 阶段 | 新增字段 / 表 |
| --- | --- | --- |
| M1 Seen/Holdout | P4 | `eval_cases.holdout` + `sample_batches` stratified split |
| M2 min_n gate | P4 | `acceptance_gate.offline_eval.min_n_per_capability` |
| M3 合成变体探针 | P4 | `synthesize` 新参数 `targetHoldout=true` |
| M4 跨维度 guard | P4 | `validation_generalization_reports.cross_capability_guard` |
| M5 Bootstrap CI | P4 | `validation_generalization_reports.bootstrap_ci_per_capability` |

新增表 `validation_generalization_reports`：

| 字段 | 含义 |
| --- | --- |
| `id` | uuid 主键 |
| `validation_run_id` | FK → validation_runs |
| `gen_coeff_per_capability` | JSONB，每个 capability 的 gen_coeff |
| `bootstrap_ci_per_capability` | JSONB，每个 capability 的 mean_delta / ci_lower / ci_upper / n_cases |
| `cross_capability_guard` | JSONB，12 个 capability 的 delta + passed |
| `insufficient_evidence_capabilities` | text[]，N 不足的 capability 列表 |
| `overall_verdict` | `passed / failed / insufficient_evidence / generalization_warning` |
| `created_at` | timestamptz |

### 11.12 验收

1. `npm run generalization:ci` 可独立跑通，输出 gen_coeff、bootstrap CI、cross-capability guard 三份报告。
2. 构造一个"只针对 seen 改善、holdout 未改善"的场景，`gen_coeff < 0.5` → overall_verdict = `failed`。
3. capability 桶 N = 3 时，`overall_verdict = insufficient_evidence`，不输出 pass/fail。
4. 修复 capability A 后，故意让 capability B 退化 0.06，cross-capability guard 捕获 B `passed=false` → 整包 fail。
5. `ci_95_lower = -0.01`（CI 跨 0）→ overall_verdict = `failed`，即使 mean_delta = 0.06 也不通过。
