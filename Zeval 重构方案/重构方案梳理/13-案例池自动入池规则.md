# 13-案例池自动入池规则

案例池是 Zeval 回归与调优的弹药库。重构后必须实现”评估完成 → 候选自动入池 → 去重与人工复审 → 进入正式案例池”的闭环。本文档定义自动入池的触发条件、阈值、去重策略与可观测性。

## 13.1 设计目标

1. 自动化优先：客户无需逐条点选，系统按隐式信号 + 客观指标自动识别值得入池的会话。
2. 可解释：每条入池记录必须显式记录命中的规则与阈值。
3. 去重：避免相似 session 反复入池，尤其是机器人造成的重复样本。
4. 可复审：误判可被人工标记 `false_positive`，自动入池规则必须尊重该标记。
5. 来源可分：所有 case 必须显式标注 `source`，共 8 类：

| source 值 | 采集通道 | 混淆矩阵定位 | 说明 |
| --- | --- | --- | --- |
| `auto_tp` | 自动-真坏案例 | TP | 规则/主观指标命中，系统判为不达预期且正确 |
| `manual_fp` | 人工复审-假阳性 | FP | 系统判不达预期，人工复审确认实际 OK，作为正例对照组入池 |
| `auto_fn` | 自动-漏报 | FN | 系统判 OK，但下游行为信号（流失/重复提问/负面尾）显示实际不达预期 |
| `auto_tn` | 自动-金标正例 | TN | 分层抽样的高质量对话，作为回归基准正例 |
| `auto_uncertainty` | 自动-边界案例 | 高信息量 | Judge confidence ∈ [0.4, 0.6]，判断不确定，需人工确认 |
| `auto_disagreement` | 自动-评估器分歧 | 评估器质量 | 规则判定与 LLM judge 结论相反，暴露评估器自身缺陷 |
| `synthesized` | 合成 | — | LLM 辅助生成的样本，不进入 baseline / 在线评测统计基线 |
| `imported` | 外部导入 | — | 从外部系统批量导入的历史案例 |

原有 `auto_admission` 和 `manual` 已被上述 8 类取代，读旧数据时兼容映射：`auto_admission → auto_tp`，`manual → manual_fp` 或 `imported`（由导入来源决定）。

## 13.2 触发时机

入池在评估管线完成后串行执行：

```
evaluate done → buildAdmissionCandidates → applyAdmissionRules → dedupe → upsert eval_cases
```

入池只读取 `evaluation_runs / objective_signals / subjective_signals / risk_tags` 等已落库的数据，不重新计算。批量上传场景下，按 session 维度评估每条候选。

## 13.3 默认入池规则

每条规则用 `{ key, condition, severity, caseSetType, source }` 定义。所有规则必须可在 Supabase 中配置（见 13.6）。

### 13.3.1 TP 通道（auto_tp）

系统判定不达预期、自动入池为 badcase：

| 规则 key | 条件 | 入池类型 | 严重度 |
| --- | --- | --- | --- |
| `goal_failed` | `goalCompletion.status in ("failed", "partial")` | badcase | high |
| `low_empathy` | `subjective.empathy.score <= 2` | badcase | medium |
| `off_topic_high` | `subjective.offTopicOrIgnoreRisk.score <= 2` | badcase | medium |
| `preachy_high` | `subjective.preachinessRisk.score <= 2` | badcase | medium |
| `interest_decline` | `risk_tags.interestDeclineRisk.severity in ("medium", "high")` | badcase | medium |
| `understanding_barrier` | `risk_tags.understandingBarrierRisk.severity in ("medium", "high")` | badcase | medium |
| `recovery_failure` | `risk_tags.emotionRecoveryFailureRisk.severity == "high"` | badcase | high |
| `high_dropoff` | `dropoffTurnDistribution` 在 1–3 轮命中且 user 长度趋势下降 | badcase | low |

### 13.3.2 FN 通道（auto_fn）

系统未命中 TP 规则（系统判 OK），但下游行为信号显示实际不达预期，自动入池为 badcase。**这是最容易被忽略、信息量最高的通道。**

| 规则 key | 触发条件 | 说明 |
| --- | --- | --- |
| `fn_dropoff_negative_tail` | 会话末尾 3 条用户消息中 ≥2 条命中负向情绪词，且最后一条为 user | 用户带着负情绪离开 |
| `fn_repeated_question` | 同一 session 内同一归一化问题出现 ≥ 2 次 | 用户得不到有效回答 |
| `fn_length_collapse` | 用户后半段平均消息长度 < 前半段 × 0.6 且总轮次 ≥ 6 | 参与度骤降 |
| `fn_consecutive_short` | 连续 3 条以上 user 消息长度 ≤ 5 且末条为 user | 用户失去表达意愿 |

FN 通道触发后，候选的 `reviewStatus` 默认为 `pending_review`（不直接进入 regression_active），需人工或批量打标确认。

### 13.3.3 边界与分歧通道（auto_uncertainty / auto_disagreement）

| 规则 key | 触发条件 | source | 入池类型 | 说明 |
| --- | --- | --- | --- | --- |
| `judge_uncertainty` | 任意主观维度 `confidence ∈ [0.4, 0.6]` | `auto_uncertainty` | badcase | 高信息量边界案例，需人工确认 |
| `rule_llm_disagreement` | TP 规则命中 ≥ 1 项，但 LLM judge 整体 `goalCompletion.status == "achieved"` | `auto_disagreement` | badcase | 规则与 LLM 判定相反，暴露评估器缺陷 |

### 13.3.4 TN 通道（auto_tn）

分层抽取高质量对话，作为回归基准正例，防止评测集被 badcase 灌水形成偏置：

| 规则 key | 条件 | 入池类型 | 说明 |
| --- | --- | --- | --- |
| `goal_achieved_high_score` | `goalCompletion.status == "achieved"` 且 `subjective.empathy.score >= 4` 且 `subjective.followUpEffectiveness.score >= 4` | goodcase | 高质量正例 |
| `clean_session_sampled` | 无任何 TP/FN/uncertainty/disagreement 规则命中，按配置抽样率（默认 5%）随机入池 | goodcase | 均衡抽样 |

**Holdout Split 规则（与 11.12 M1 联动，Post-MVP）**：Phase 2（P4+）起，TN 通道每次入池时，按 `capability_dimension` 分桶后做 stratified split，约 30% 的 TN case 被标记为 `holdout = true`（永远不对修复者暴露，只用于最终 validation gate）。每个 capability 桶的 holdout 数量最少保持 10 条；不足时优先补充 holdout，而非 seen set。**MVP 阶段此逻辑不激活，所有 case 默认 `holdout = false`。**

### 13.3.5 FP 通道（manual_fp）

FP 通道不由自动规则触发，由人工复审驱动：

- 当人工对某条 badcase 设置 `humanVerdict = "false_positive"` 时，系统将该 case 转为 `caseSetType = "goodcase"`，`source = "manual_fp"`，作为"系统误判的正例"入池。
- 这类 case 保留原 badcase 的触发规则记录，便于后续校准评估器灵敏度。
- `false_positive` 标记同时写入原始记录的 `metadata.false_positive`，防止同一 session 再次被 TP 通道自动入池。

---

**通用约束：**

- 所有阈值必须可调，写入 `dataset_admission_rules` 表（见 13.6）。
- 当多个规则命中同一 session 时，合并触发原因；最高严重度作为主严重度。
- 仅当 `useLlm = true` 且主观结果可用时，依赖主观维度的规则才生效；LLM 不可用时只跑客观与隐式规则（TP 规则）和 FN 通道规则。
- `synthesized` 和 `imported` 来源的 case 不参与 FN/TN 通道的抽样计算。
- `capability_dimension` 为可选字段（MVP 阶段不强制），有则写入候选与 case，无则留空；`failure_layer` **MVP 阶段不强制**，harness 层归因约束推迟至 Phase 2（P4+）。
- 若候选已携带 `capability_dimension`，TN 通道的 `clean_session_sampled` 按维度分层抽样；未填则统一归入默认分桶，不抛错。

## 13.4 候选生成

候选记录字段：

```json
{
  "candidateId": "ca_xxx",
  "evaluationRunId": "evr_xxx",
  "sessionId": "s1",
  "caseSetType": "badcase",
  "source": "auto_tp",
  "capabilityDimension": "tool_calling_correctness",
  "failureLayer": "L5_tool_selection",
  "attributionConfidence": 0.62,
  "triggeredRules": ["goal_failed", "low_empathy"],
  "severity": "high",
  "snapshot": {
    "transcript": [],
    "objective": {},
    "subjective": {},
    "riskTags": []
  },
  "normalizedTranscriptHash": "sha256:..."
}
```

`capabilityDimension` 与 `failureLayer` 详见 [`15-能力维度评测与归因.md`](./15-能力维度评测与归因.md)。**MVP 阶段两者均为可选**；`capabilityDimension` 有则填，`failureLayer` 推迟至 Phase 2 再强制。goodcase 候选的 `failureLayer` 必须为 NULL（保持现有约束）。

`source` 字段与触发规则的对应关系：

| source | 触发来源 |
| --- | --- |
| `auto_tp` | 13.3.1 任意规则命中 |
| `auto_fn` | 13.3.2 任意规则命中 |
| `auto_uncertainty` | 13.3.3 `judge_uncertainty` 命中 |
| `auto_disagreement` | 13.3.3 `rule_llm_disagreement` 命中 |
| `auto_tn` | 13.3.4 任意规则命中 |
| `manual_fp` | 人工操作触发，不经候选生成流程直接写入 |
| `synthesized` | 合成管线写入，不经候选生成流程 |
| `imported` | 外部导入工具写入，不经候选生成流程 |

候选先写入 `eval_case_candidates`（见 13.6），由后续规则判断是否落入正式 `eval_cases`。

## 13.5 去重策略

去重分两层：

1. **精确去重**：基于 `normalized_transcript_hash`。
   - 与现有 `eval_cases.normalized_transcript_hash` 命中则跳过；可选记录“同源样本”关联。
2. **近似去重**：基于轻量规则。MVP 不引入向量检索，仅采用：
   - 长度差小于 5%。
   - 用户消息归一化后字符差异率小于 10%。
   - 命中规则集合相同。
   - 所属客户 / 项目相同。
   命中近似去重时：
   - 默认拒绝入池，仅记录 `near_duplicate_of_case_id`。
   - 客户 PATCH `allowNearDuplicate=true` 时允许放行，并同步在 `eval_cases.metadata.near_duplicate_of` 中关联。

去重过程中遇到 `eval_cases.metadata.false_positive == true` 的旧记录，必须把候选丢弃，并记录 `triggered_rules + skipped_reason="false_positive_match"`。

## 13.6 数据模型扩展

新增 `dataset_admission_rules` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 规则 ID |
| `project_id` | 所属项目 |
| `rule_key` | 规则 key |
| `source` | 对应的 8 类 source 之一 |
| `case_set_type` | goodcase / badcase |
| `expression` | 规则表达式 JSON（如 `{"path": "subjective.empathy.score", "op": "<=", "value": 2}`） |
| `severity` | low / medium / high |
| `enabled` | 默认 true |
| `created_at`, `updated_at` | 审计 |

新增 `eval_case_candidates` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 候选 ID |
| `evaluation_run_id` | 来源 run |
| `session_id` | 来源 session |
| `case_set_type` | goodcase / badcase |
| `source` | 8 类 source 之一（见 13.1） |
| `capability_dimension` | 12 个能力维度之一（见 15 号文档） |
| `failure_layer` | L0–L8 之一；badcase 必填，goodcase 必须 NULL |
| `attribution_confidence` | 自动归因置信度 0–1 |
| `triggered_rules` | 命中规则 JSONB |
| `severity` | 综合严重度 |
| `snapshot_payload` | 候选 snapshot |
| `normalized_transcript_hash` | 文本哈希 |
| `decision` | `accepted / rejected / pending_review` |
| `decision_reason` | 例如 `auto_admitted / exact_duplicate / near_duplicate / false_positive_match / manual_rejected / low_attribution_confidence` |
| `created_at` | 时间 |

`eval_cases` 表新增以下字段：

- `source`：8 类来源（见 13.1）。
- `capability_dimension`：12 个能力维度之一，**MVP 阶段可选（nullable）**；Phase 2 起再追加 NOT NULL 约束。
- `failure_layer`：L0–L8 之一；**MVP 阶段不强制**，字段存在但无 check 约束；goodcase 必须为 NULL；Phase 2（P4+）起对 badcase 强制。
- `attribution_confidence` 与 `attribution_method`：归因置信度与方法（详见 15 号文档 15.7.1）；MVP 阶段可空。
- `metadata.false_positive` 与 `metadata.near_duplicate_of`：人工标注与近重复关联。
- `holdout`：`boolean, not null, default false`。Post-MVP（Phase 2）起用于泛化验证门禁（详见 [`11-测试与回归.md`](./11-测试与回归.md) 11.12 M1）；MVP 阶段字段存在，默认 false，不强制 holdout 分离逻辑。

兼容说明：读取旧数据时，`source = null` 或 `auto_admission` 映射为 `auto_tp`；`source = manual` 根据是否有 `manualOverrides[].type == "false_positive"` 映射为 `manual_fp` 或 `imported`。

## 13.7 API 行为

- `POST /api/eval-datasets/harvest-badcases`（已有）：保留入口，底层扩展为”按 5 个自动通道跑候选（TP/FN/uncertainty/disagreement/TN）→ 去重 → 落 `eval_cases`”。返回值新增 `acceptedBySource`（按 source 分组计数）、`skippedDuplicates / skippedFalsePositive / pendingReview` 计数。
- `GET /api/eval-datasets/cases`：新增过滤参数 `source=auto_tp|manual_fp|auto_fn|auto_tn|auto_uncertainty|auto_disagreement|synthesized|imported`，便于按通道复审。
- `PATCH /api/eval-datasets/cases/[caseId]`：`humanVerdict = “false_positive”` 时触发 FP 通道——自动将 case 转为 `caseSetType = “goodcase”`，`source = “manual_fp”`，并写入 `metadata.false_positive = true`，防止重复入池。
- `GET /api/eval-datasets/admission-rules`：列出当前生效规则，按 source 分组返回。
- `POST /api/eval-datasets/admission-rules`：创建规则，`source` 字段必填。
- `PATCH /api/eval-datasets/admission-rules/[id]`：更新阈值与启用状态。

| `imported` | 外部导入 |

- 候选列表（pending review）单独入口，显示触发规则、source 通道、严重度、相似 case 链接。
- `auto_fn / auto_uncertainty / auto_disagreement` 三类 case 默认进入 `pending_review`，在列表中以”待确认”状态提示，不直接参与回归统计。

## 13.9 可观测性

每次自动入池必须写入审计日志（或 `audit_logs` / `dataset_imports.metadata`），记录：

- `evaluationRunId`
- 候选总数
- 命中各规则的数量
- 精确去重数、近似去重数、误判跳过数
- 最终入池数

## 13.10 验收

1. 给定一份评估结果，`harvest-badcases` 输出与触发规则一一对应，每条 case 的 `source` 字段与命中通道一致，可重放。
2. 同一会话再次评估不会重复入池（精确哈希命中）。
3. 设置 `false_positive` 后再跑同样 run，TP 通道不会再次把同一 session 入池；FP 通道已将其转为 `manual_fp` goodcase。
4. `dataset_admission_rules` 启用/禁用立即生效，无需重启服务。
5. 当主观指标降级或不可用时，仅跑客观 + 隐式规则（TP 通道）和 FN 通道规则，不抛错；`auto_uncertainty` 和 `auto_disagreement` 规则自动跳过并记录 `skipped_reason = "llm_unavailable"`。
6. FN 通道：构造一个"系统判 OK 但用户重复提问 ≥ 2 次"的 session，`harvest-badcases` 返回值的 `acceptedBySource.auto_fn ≥ 1`。
7. `GET /api/eval-datasets/cases?source=auto_fn` 只返回 `source = auto_fn` 的 case。
8. 把一条 `auto_tp` badcase 标记为 `humanVerdict = "false_positive"` 后，该 case 的 `caseSetType` 变为 `goodcase`，`source` 变为 `manual_fp`，重新跑 harvest 时不会再被 TP 规则命中。
9. ⚠️ **Post-MVP**：TN 通道入池时，`holdout = true` 的 case 数量约为 seen 的 43%（保证 holdout ≥ 30%），且 `GET /api/eval-datasets/cases?holdout=true` 接口不对修复者暴露（需 `role=gate_reviewer` 权限）。MVP 阶段字段存在，默认 false，不实施 holdout 分离与权限控制。
10. ⚠️ **Post-MVP**：每个 capability 桶的 holdout count 可通过 `GET /api/eval-datasets/cases/counts?group=capability,holdout` 查询，不足 10 条时返回 `holdout_insufficient_capabilities` 告警字段。

## 13.11 瘦日志模式下的 source 操作定义

> 本节定义当对话日志为**瘦日志**（thin log）时，8 类 source 通道的实际可操作性与降级策略。
>
> **瘦日志定义**：只有 Q&A 对或 final_message 可用，没有工具调用链（tool_calls）、memory 命中记录、reasoning block。这是 MVP 阶段对话智能体评测的典型场景。

### 13.11.1 各通道可行性

| source 通道 | 瘦日志可行性 | 实际操作 |
| --- | --- | --- |
| `auto_tp` | ✓ 可用 | 依赖文本级信号：负向关键词 + 高严重度隐式信号 + goal_failed / recovery_failure 标签，足以触发 |
| `manual_fp` | ✓ 不受限 | 人工标注，不依赖日志结构 |
| `auto_fn` | ⚠️ FN-proxy（弱化） | 退化为文本可见代理信号：负向尾词、重复提问（≥2次）、最后一轮长度骤降（< 前均值 × 0.4）；工具调用层漏报不可见 |
| `auto_tn` | ⚠️ 需抽样配置 | 自动化程度低：须通过随机抽样比例或高分阈值（goal_achieved + 主观分 ≥ 0.8）筛选，不能全自动 |
| `auto_uncertainty` | ✓ 可用 | Judge confidence 区间 [0.4, 0.6] 判断，不依赖日志结构 |
| `auto_disagreement` | ⚠️ 校准专用 | **当前管线（fused）不能自动产生**。需双轨并行架构（规则链与 LLM judge 各跑一次，事后对比），MVP 阶段由校准员手动录入 |
| `synthesized` | ✓ 可用 | LLM 合成，不依赖真实日志结构 |
| `imported` | ✓ 可用 | 批量导入，做格式转换即可 |

### 13.11.2 failure_layer 默认约定

瘦日志无法判定 harness 层细节，`failure_layer` 字段按以下规则填写：

| 情况 | failure_layer 默认值 | 说明 |
| --- | --- | --- |
| 仅文本日志，无法归因 | `unknown` | Phase 2 启用 harness 后补全 |
| 确认为最终输出质量问题 | `L8`（Generation 层） | 文本级信号可支持此判断 |
| L0–L7 任一层 | 待 Phase 2 归因 | 瘦日志缺乏工具调用链、memory 记录，无法区分 |

`failure_layer = unknown` 的 case 不影响 capability_dimension 分类与入池，仅在 Phase 2 cross-capability guard 中才会被要求补全。

### 13.11.3 关于 auto_disagreement 的说明

`auto_disagreement` 代表"规则判定与 LLM judge 结论相反"的分歧案例，是 Calibration 的最高价值信号。但当前管线是 **fused** 模式（LLM 优先，规则只在 LLM 失败时作为兜底），两个评估器不会并行产生各自独立的结论。

**MVP 阶段操作方式**：

1. 校准员手动对比一批会话的规则命中情况与 LLM judge 结论。
2. 发现明显分歧的 case 手动标记 `source = auto_disagreement`，写入 `eval_case_candidates`。
3. 进入 `pending_review` 队列，由专人确认后入正式池。

**Phase 2 自动化路径**：引入双轨并行执行器，规则链与 LLM judge 独立运行，结论不一致时自动触发 `auto_disagreement` 入池逻辑。

### 13.11.4 实践建议

- **优先建立 auto_tp 通道**：文本级信号最稳定，MVP 应先打通此通道。
- **auto_fn 使用 proxy 信号时需在 UI 中标注来源**：告知用户该 case 的漏报判断基于文本代理，非 harness 确认。
- **auto_tn 配置固定抽样率**：建议初始值为每次评估随机抽取 5%–10% 的高质量会话，避免正例过多挤压 badcase 空间。
- **瘦日志场景不建议在早期强制填 failure_layer**：让字段保持 `unknown`，等 Phase 2 归因完善后统一回填，比强行填 `L8` 更准确。
