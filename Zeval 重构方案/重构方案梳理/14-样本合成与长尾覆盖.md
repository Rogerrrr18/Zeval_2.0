# 14-样本合成与长尾覆盖

业务长尾问题往往在真实日志中样本极少，难以单靠采集形成有效评测集。Zeval 提供 LLM 辅助的样本合成能力，让客户基于自己的场景描述生成长尾问题样本，作为评测集补充。

## 14.1 设计目标

1. 客户驱动：客户在自己的项目空间内描述长尾场景，由 Zeval 合成样本，不在公共空间共享。
2. 可解释：每条合成样本必须保留生成参数、模型、prompt 版本与质量校验结果。
3. 可控质量：合成样本经过 LLM 自校验后再入池，避免与目标问题不相关或重复。
4. 与评测集衔接：合成样本默认落入 `eval_cases.source = "synthesized"`，可被 sample batch 抽样、validation 回归引用。
5. 不污染真实数据：合成样本独立标识，不参与 baseline 与在线评测的统计基线。

## 14.2 输入与场景模板

合成请求关键输入：

```json
{
  "capabilityDimension": "knowledge_grounding",
  "targetFailureLayer": "L4_retrieval",
  "scenarioBrief": "B2B SaaS 客户支持，处理订阅暂停后退款的长尾问题",
  "longTailHints": [
    "用户已经退款被拒，但仍希望保留账户数据",
    "用户认为退款金额与实际扣费不符"
  ],
  "expectedFailures": ["off_topic", "preachy", "goal_partial"],
  "personas": [
    { "id": "frustrated_customer", "tone": "急躁", "language": "zh" }
  ],
  "session": {
    "minTurns": 6,
    "maxTurns": 14,
    "rolesPattern": "user-assistant alternating"
  },
  "count": 5,
  "labelDefaults": { "tags": ["billing", "long_tail"] }
}
```

`capabilityDimension` 必填，从 [`15-能力维度评测与归因.md`](./15-能力维度评测与归因.md) 的 12 个白名单值中选择。`targetFailureLayer` 可选，用于让合成 prompt 针对 harness 某一层注入失败模式（如 L4_retrieval 时刻意构造检索召回不准的样本）。

可选模板：客户可在项目空间维护 `synthesis_templates`（场景 → 默认 hints / personas / failures），减少重复输入。

## 14.3 生成流程

```
input → buildPromptContext → LLM 生成 candidate sessions → schema 校验
     → LLM self-check（与 longTailHints / expectedFailures 对齐）
     → 去重（normalized_transcript_hash + 与现有 eval_cases 比对）
     → 落库 eval_cases（source = synthesized，capability_dimension 来自请求参数）
```

约束：

- 生成 + 自校验各算一次 LLM 调用，必须分别记录 `judge_runs`，`stage` 取 `sample_synthesize` 与 `sample_synthesize_review`。
- 自校验输出 `{ alignmentScore, redundancy, qualityIssues[] }`。`alignmentScore < 阈值`（默认 0.7）的样本被丢弃，写入 `synthesis_runs` 失败原因。
- 不允许把客户真实对话直接喂入合成 prompt，仅可使用 redacted snippet 或抽象描述。
- 合成样本写入 `eval_cases` 时必须把请求中的 `capabilityDimension` 落到 `eval_cases.capability_dimension`；若样本被定位为 badcase 且请求带 `targetFailureLayer`，则 `failure_layer` 取该值并标 `attribution_method = 'manual'`（合成时人为指定，非实验归因）。

## 14.4 数据模型

新增 `synthesis_runs` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 合成运行 ID |
| `project_id` | 所属项目 |
| `request_payload` | 输入参数 JSON |
| `template_id` | 可选，引用 `synthesis_templates` |
| `model` | 实际调用模型 |
| `prompt_version` | 生成与自校验 prompt 版本 |
| `status` | `succeeded / partial / failed` |
| `requested_count` | 计划生成数 |
| `accepted_count` | 通过质量门入池数 |
| `rejected_count` | 校验失败 |
| `warnings` | 警告 |
| `created_at` | 时间 |

新增 `synthesized_samples` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 样本 ID |
| `synthesis_run_id` | 来源 run |
| `eval_case_id` | 对应 `eval_cases.id`（入池后填） |
| `transcript_payload` | 生成的 session |
| `alignment_score` | 自校验对齐分 |
| `redundancy_score` | 与已有 case 的重复度 |
| `judge_run_id` | 关联 LLM Judge 调用 |
| `metadata` | personas / longTailHints / expectedFailures 等 |

可选 `synthesis_templates` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 模板 ID |
| `project_id` | 所属项目 |
| `name` | 模板名 |
| `default_hints` | JSONB |
| `default_personas` | JSONB |
| `default_expected_failures` | JSONB |
| `created_at`, `updated_at` | 审计 |

## 14.5 API 行为

```json
{
  "synthesisRunId": "syn_xxx",
  "status": "succeeded",
  "acceptedCount": 4,
  "rejectedCount": 1,
  "samples": [
    {
      "synthesizedSampleId": "ss_xxx",

  "warnings": []

```

## 14.6 与评测集 / sample batch 的衔接

- 合成样本默认 `eval_cases.source = "synthesized"`，可被 `sample_batches` 抽样选中。
- sample batch 抽样时可指定 `includeSynthesized=true|false` 控制是否纳入。
- validation runs 必须显式记录是否包含合成样本，避免回归报告对比口径混淆。
- baseline 与在线评测不读取 `source = "synthesized"` 的样本，防止合成数据污染统计基线。

## 14.7 前端展示

- 案例池中合成样本显示 `合成` 徽章与对齐分。
- 提供合成入口：客户输入场景与 hints，预览生成结果后批量入池。
- 模板管理入口：常用场景一键复用。
- 文案不展示模型名 / prompt 版本，只展示业务语义（场景、对齐分、重复度）。

## 14.8 LLM 调用约束

- 默认使用 SiliconFlow 默认模型（详见 `8-LLM调用与降级.md`）。
- `SILICONFLOW_ENABLE_THINKING` 与 `ZEVAL_JUDGE_ENABLE_THINKING` 必须为 false，避免 thinking 内容污染合成输出。
- 合成 prompt 必须输出 JSON：包含 transcript、tags、expected failures。
- 自校验 prompt 必须输出 JSON：alignmentScore / redundancy / issues。
- 任一阶段失败：合成运行进入 `failed`，已生成的部分按 `partial` 入池或回滚（按客户配置）。

## 14.9 边界

- MVP 不做大规模批量合成（默认单次 ≤ 20 条，避免 LLM 成本失控）。
- 不在公共项目共享合成样本；合成样本仅在请求方 `project_id` 内可见。
- 不接受“近似真实用户”的合成需求，禁止合成可识别的特定人物对话。

## 14.10 验收

1. 任一合成请求都能写入 `synthesis_runs` 与 `synthesized_samples`，并关联到 `eval_cases`。
2. 任一合成样本都能从 case 列表追溯到 `synthesis_run_id`、`alignmentScore`、判定理由。
3. LLM 不可用时合成请求显式失败，并提示客户检查模型可用性。
4. `includeSynthesized=false` 时回归与 sample batch 不应出现合成样本。
5. `targetHoldout=true` 时合成的样本全部落入 `eval_cases.holdout = true`，且不出现在修复者可查的 seen set 接口中。

## 14.11 泛化探针合成（与 11.12 M3 联动）

**设计背景：** 自然入池的 badcase 场景分布往往集中在少数典型失败模式（如"退款场景下 instruction_following 失败"）。修复者容易只针对已知模式改善，而在相同能力的其他场景（预约、咨询、投诉）上仍然失败。泛化探针合成专门解决这个问题。

### 14.11.1 触发时机

每批新 badcase 入库完成后，系统检查每个 `capability_dimension` 桶的 `holdout_count < seen_count × 0.43`（即 holdout 比例低于 30%），则自动触发探针合成补充。

触发来源：
- 自动触发：由 `harvest-badcases` 完成后异步调用；
- 手动触发：`POST /api/eval-datasets/synthesize-probes` 指定 `capabilityDimension` 和 `sourceCase`。

### 14.11.2 合成策略

| 策略维度 | 要求 |
| --- | --- |
| 能力保持 | 探针必须与源 badcase 属于同一 `capability_dimension` |
| 场景变换 | 场景/角色/上下文必须与源 badcase 明显不同（如换行业、换诉求类型）|
| 失败层保持 | 探针的 `targetFailureLayer` 与源 badcase 一致，确保对照性 |
| 数量 | 默认每条源 badcase 生成 K=5 个探针，可配置 |
| 入池路径 | `source = "synthesized"`，`holdout = true`，`attribution_method = "manual"` |
| 可见性 | 探针 case 不出现在修复者的案例池视图中（`holdout=true` 过滤）|

### 14.11.3 合成请求扩展参数

在 14.2 标准输入基础上新增：

```json
{
  "targetHoldout": true,           // true → 合成结果全部落入 holdout set
  "sourceCase": {                  // 可选，提供源 badcase 作为模式参考
    "caseId": "case_xxx",
    "transcriptSnippet": "..."
  },
  "sceneVariations": [             // 场景变换指令（若未填则自动生成）
    "将场景从退款改为预约取消",
    "将用户身份从消费者改为企业采购员"
  ]
}
```

### 14.11.4 探针质量门

探针合成在标准 14.3 质量校验基础上额外检验：

- `scene_diversity_score`：与源 badcase 的场景相似度，**必须 < 0.6**（否则视为同一场景变体，拒绝）；
- `capability_alignment_score`：探针仍属于目标能力维度，**必须 ≥ 0.75**；
- 双重不通过的探针丢弃，写入 `synthesis_runs.rejected_count`，不重试（避免成本失控）。

### 14.11.5 API

| 路由 | 方法 | 说明 |
| --- | --- | --- |
| `/api/eval-datasets/synthesize` | POST | 已有接口，新增 `targetHoldout` + `sourceCase` + `sceneVariations` 参数 |
| `/api/eval-datasets/synthesize-probes` | POST | 专用探针合成接口，输入 `capabilityDimension + sourceCaseIds`，系统自动拉取场景变换指令 |
