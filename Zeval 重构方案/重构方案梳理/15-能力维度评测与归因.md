# 15-能力维度评测与归因

Zeval 不只是"对话质量评估工具"，目标是成为"Agent 优化决策系统"。本文档定义 Zeval 如何把用户语言（"我想优化多轮对话"）映射到工程语言（改 model / prompt / harness 哪一层），以及评测集如何按能力维度组织、按 6+2 类信号通道分桶、按 harness 层做故障归因。

本文档与 [13-案例池自动入池规则.md](./13-案例池自动入池规则.md)、[14-样本合成与长尾覆盖.md](./14-样本合成与长尾覆盖.md)、[12-调优包-Skill化交付.md](./12-调优包-Skill化交付.md) 共同构成"评测集 → 归因 → 调优"的完整闭环。

---

## 15.1 设计目标

> **MVP 范围说明**：P3 阶段（MVP）实现"能力维度可选标注 + 6+2 类信号通道"；harness 层归因（L0–L8）、Copilot 决策、对照实验为 **Phase 2（P4+）**，详见 [6-重构实施路线.md](./6-重构实施路线.md)。本文档同时覆盖 MVP 目标与 Phase 2 规划，各节目标标注了所属阶段。

1. **能力维度优先**（MVP）：评测集的主组织维度是 `capabilityDimension`（用户语言），不再是症状标签集合；MVP 阶段为可选标注。
2. **层级归因可解释**（Phase 2）：每条 badcase 显式归因到 harness 的具体层（L0–L8），归因结果可重放；MVP 不强制。
3. **对照实验为唯一真相**（Phase 2）：归因结论来自对照实验差异，不允许仅凭 LLM 单次判断给出"应该改 X"。
4. **Copilot 决策导向**（Phase 2）：Copilot 不只回答"哪里不好"，必须回答"先改什么、再改什么、性价比多少"。
5. **数据立方体**（Phase 2）：评测集是 `(capability × source × layer)` 三元组定位的立方体，三轴正交、可独立查询。

### 主参考

| 架构 | 核心抽象 | 入选理由 |
| --- | --- | --- |
| Claude Agent SDK / Tool Use | tool-use loop + skills（just-in-time）+ memory + computer use | Anthropic 官方、production-ready，skills 模型与本项目调优包一致 |

## 15.3 Canonical Agent Harness — 9 层抽象

每一层都满足"可独立替换、可独立测、可独立归因"的标准。

```
┌─────────────────────────────────────────────────────────────┐
│  L0  Model Substrate     模型底座（可路由 / 多厂商 / 参数）       │
├─────────────────────────────────────────────────────────────┤
│  L1  Input Understanding 意图解析 / slot filling / 输入校验     │
│  L2  Planning            任务分解 / TODO 树 / re-planning     │
│  L3  Memory & Context    短期 working / 长期 persistent /     │
│                          context window 压缩与召回             │
│  L4  Retrieval           RAG / KB query / multi-hop / rerank │
│  L5  Tool Selection      选工具 + 构造参数 + 参数校验            │
│  L6  Tool Execution      实际调用 / observation 处理 / 重试    │
│  L7  State & Control     状态机 / 循环终止 / handoff / 多 agent│
│  L8  Generation          最终合成 / 格式 / 风格 / safety filter │
└─────────────────────────────────────────────────────────────┘
```

每一层对应**可对照实验的"自变量"**，这是归因可执行的前提：

| 层 | 可对照的自变量 | 实验代价（相对） |
| --- | --- | --- |
| L0 | 换模型 / 调温度 / 调 top_p / 路由策略 | 高（重新跑全量） |
| L1 | 改 intent schema / 改 input validator | 低 |
| L2 | 改 plan prompt / 启用 plan-reflect / 改 max steps | 中 |
| L3 | 改 memory policy / 改 summarization / 改 truncation | 中 |
| L4 | 改 chunk size / 改 retriever / 改 rerank / 改 query rewrite | 中-高 |
| L5 | 改 tool description / 改 tool schema / 改候选集 | 低 |
| L6 | 改 retry policy / 改 timeout / 改 observation parser | 低 |
| L7 | 改 graph 拓扑 / 改终止条件 / 改 handoff 规则 | 高 |
| L8 | 改 system prompt / 改 output schema / 改 style guide | 低 |

`HarnessLayer` 枚举值（落到代码与表）：`L0_model / L1_input / L2_planning / L3_memory / L4_retrieval / L5_tool_selection / L6_tool_execution / L7_state / L8_generation`。

## 15.4 用户视角能力维度（12 个白名单）

能力维度是**用户语言**，不是 harness 语言。两者不能混，但要建映射。

| `capabilityDimension` | 中文 | 用户场景表述 |
| --- | --- | --- |
| `instruction_following` | 指令遵循 | "我让它做 A 它做了 B" |
| `multi_turn_coherence` | 多轮一致性 | "聊到第 5 轮就忘了" |
| `long_context_reasoning` | 长上下文推理 | "100 页 PDF 它抓不到关键" |
| `task_decomposition` | 任务分解 | "复杂任务它不会拆步骤" |
| `tool_calling_correctness` | 工具调用正确性 | "选错了工具 / 参数错" |
| `workflow_execution` | 工作流端到端 | "多步骤跑一半就丢" |
| `knowledge_grounding` | 知识忠实度 | "胡编乱造 / 引用对不上" |
| `safety_refusal` | 安全拒答 | "应拒答的没拒 / 不该拒的乱拒" |
| `style_format` | 风格与格式 | "输出格式 / tone 不对" |
| `latency_efficiency` | 效率 | "太慢 / token 太贵 / 调用次数多" |
| `self_correction` | 自我纠错 | "出错后不能恢复" |
| `human_handoff` | 人工交接 | "该转人工不转 / 转得太草率" |

新增能力维度需要走 PR 评审，不允许仅在前端临时加。

## 15.5 能力 ↔ Harness 层归因矩阵

每个能力都映射到**主因层 + 次因层**。这张表是 Copilot 做归因建议、设计对照实验的基石。

| 能力维度 | 主因层（先查） | 次因层（再查） | 典型对照实验 |
| --- | --- | --- | --- |
| `instruction_following` | L8 Generation, L0 Model | L1 Input | 同 prompt 跨模型 / 改 system prompt |
| `multi_turn_coherence` | L3 Memory | L7 State, L8 Generation | 改 summarization / 改 context window |
| `long_context_reasoning` | L3 Memory, L0 Model | L4 Retrieval | 换长上下文模型 / 拆 chunk + RAG |
| `task_decomposition` | L2 Planning, L0 Model | L7 State | 启用 plan-reflect / 改 max steps |
| `tool_calling_correctness` | L5 Tool Selection | L0 Model, L8 Generation | 改 tool description / 跨模型 |
| `workflow_execution` | L7 State, L2 Planning | L6 Tool Execution | 改终止条件 / 改 retry 策略 |
| `knowledge_grounding` | L4 Retrieval | L8 Generation | 改 chunk / 改 rerank / 加引用约束 |
| `safety_refusal` | L8 Generation, L0 Model | L1 Input | safety filter / refusal prompt |
| `style_format` | L8 Generation | L0 Model | 改 system prompt / 改 output schema |
| `latency_efficiency` | L7 State, L4 Retrieval | L0 Model, L2 Planning | 减步数 / 换更小模型 / cache |
| `self_correction` | L2 Planning, L7 State | L0 Model | 加 reflect / 加 verifier |
| `human_handoff` | L7 State, L1 Input | L8 Generation | 改 handoff 规则 / intent classifier |

**使用规则：**

1. 当某 capability 桶里 badcase 在某主因层占比 ≥ 60%，归因输出 `recommendedAction = "<layer>"`。
2. 主因层占比不显著（最高占比 < 40%）时，触发"跨层对照实验矩阵"，再按差异最大的轴下结论。
3. 归因报告必须列出对照实验配置（自变量列表 + 实验组数），不允许仅凭 LLM 单次判断给结论。

## 15.6 评测集 3D 立方体模型

### 15.6.1 三轴定义

```
        Z 轴：Harness 失败层（L0–L8，9 层）
        ↑
        │
        │
        └────────→ Y 轴：Source 信号通道（6+2 类，见 13 号文档）
       ╱
      ╱
     ↓
    X 轴：Capability 能力维度（12 个，见 15.4）
```

每条 case 是这个立方体里的一个点：

```ts
DatasetCaseRecord {
  capabilityDimension: "tool_calling_correctness",   // X 轴，必填
  source: "auto_fn",                                  // Y 轴（6+2 通道）
  failureLayer: "L5_tool_selection",                  // Z 轴，badcase 必填
  attributionConfidence: 0.78,                        // 归因置信度 0-1
}
```

### 15.6.2 X 轴 × Y 轴：每能力一组完整 6 类信号

**每个能力对应一组完整的 6+2 类信号**，不是合并、不是替代，是正交切片。

3 个硬理由：

1. **Calibration 必须按能力分桶**：判 `tool_calling_correctness` 的 judge 跟判 `multi_turn_coherence` 的 judge 是不同 prompt、不同阈值。混在一起算 → 校准信号被污染。
2. **优化决策按能力发起**：用户从来不会说"我想优化我的 agent"，而是"我想优化多轮对话"。Copilot 拉数据时必须 `WHERE capability = X` 后再看 6 类信号分布。
3. **检测规则本身 capability-aware**：`multi_turn_coherence` 的 FN 检测看 context 引用错误；`tool_calling_correctness` 的 FN 检测看用户复诉相同请求。规则不能共用。

### 15.6.3 通道与能力的耦合度

| 通道 | 是否 capability-specific | 说明 |
| --- | --- | --- |
| `auto_tp` | 强耦合 | 每个能力有自己的 TP 触发规则（沿用 13.3.1） |
| `auto_fn` | 强耦合 | 每个能力有自己的下游信号（沿用 13.3.2 + 按 capability 扩展） |
| `auto_uncertainty` | 弱耦合 | confidence ∈ [0.4, 0.6] 通用，但要按 capability 过滤展示 |
| `auto_disagreement` | 强耦合 | 不同 capability 的 judge 不同，分歧检测也不同 |
| `manual_fp` | 弱耦合 | 操作通用，但落到哪个 capability 桶必须标 |
| `auto_tn` | 弱耦合 | 抽样规则通用，但**必须按 capability 分层抽样**保证覆盖 |
| `synthesized` | 不耦合 | 但合成请求中必须指定 `capabilityDimension`（见 15.7） |
| `imported` | 不耦合 | 但导入时必须分类标 `capabilityDimension` |

### 15.6.4 X 轴 × Z 轴：每能力的失败层分布

每个 capability 桶内统计 `layerDistribution`，给 Copilot 用作归因建议输入：

```json
{
  "capabilityDimension": "tool_calling_correctness",
  "totalBadcases": 47,
  "layerDistribution": {
    "L5_tool_selection": 0.62,
    "L0_model": 0.18,
    "L8_generation": 0.20
  },
  "primarySuspect": "L5_tool_selection",
  "primarySuspectConfidence": 0.62
}
```

## 15.7 数据模型扩展

### 15.7.1 `eval_cases` 表新增字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `capability_dimension` | text, NOT NULL | 12 个白名单值之一 |
| `failure_layer` | text, NULL | L0–L8 之一；badcase 必填，goodcase 可空 |
| `attribution_confidence` | numeric(4,3) | 归因置信度 0-1 |
| `attribution_method` | text | `rule / llm_classifier / experiment / manual` 之一 |

约束：

- `capability_dimension`：**MVP 阶段可选（nullable）**；Phase 2 起追加 NOT NULL 约束。
- `failure_layer`：**MVP 阶段不强制**，goodcase 必须为 NULL；badcase 的 NOT NULL 约束推迟至 Phase 2（P4+）。
- 所有现有 case 在迁移时 `capability_dimension` 和 `failure_layer` 均可为 NULL，等候 Phase 2 重新归因。

### 15.7.2 新表：`capability_attributions`

记录每条 badcase 的归因决策与对照实验依据，方便 Copilot 复用历史归因结论。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | 主键 |
| `eval_case_id` | uuid FK | 关联 case |
| `capability_dimension` | text | 冗余，便于按 capability 查询 |
| `failure_layer` | text | 归因层 |
| `confidence` | numeric(4,3) | 置信度 |
| `method` | text | `rule / llm_classifier / experiment / manual` |
| `experiment_run_id` | uuid FK NULL | 来源对照实验（如有） |
| `evidence` | jsonb | 支撑证据列表 |
| `created_at` | timestamptz | 时间 |

### 15.7.3 新表：`experiment_routes`

候选实验路由配置，让 Copilot 根据归因结果发起对照实验。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid | 主键 |
| `project_id` | uuid FK | 所属项目 |
| `name` | text | 实验名 |
| `target_capability` | text | 针对的能力维度 |
| `target_layer` | text | 针对的 harness 层 |
| `axis` | text | 自变量轴名（如 `model` / `system_prompt` / `tool_description`） |
| `candidates` | jsonb | 候选值数组 |
| `eval_dataset_id` | uuid FK | 用于评测的 dataset |
| `status` | text | `draft / running / completed / archived` |
| `result_summary` | jsonb | 各候选的指标矩阵 |
| `created_at` | timestamptz | 时间 |

### 15.7.4 `eval_case_candidates` 表新增字段

候选记录同步增加 `capability_dimension` 与 `failure_layer`（badcase 时必填），保持与 `eval_cases` 一致。

## 15.8 Copilot 能力扩展

现有 6 个 skill 都是评估—诊断—修复内的，需要新增 **4 个优化决策导向的 skill**：

| Skill key | 输入 | 输出 |
| --- | --- | --- |
| `diagnose_capability_gap` | 用户痛点描述（自然语言） | 推断 `capabilityDimension` + 推荐评估指标集 |
| `compose_eval_dataset_for_capability` | `capabilityDimension` + scenario | 从案例池筛 + 合成补齐，组装成定向评测集 |
| `attribute_failure_to_layer` | 一条或一批 badcase | 层级归因 + `experimentSuggestion` |
| `suggest_optimization_path` | 归因结果 + 候选矩阵 | 性价比排序的优化路径（先调什么、再调什么） |

### 15.8.1 `diagnose_capability_gap` 行为

```
输入："我感觉 agent 多轮对话不太行"
  ↓
1. NLU 抽取关键词："多轮"
2. 候选 capability 排序：multi_turn_coherence (0.82) / long_context_reasoning (0.34) / self_correction (0.18)
3. 反问澄清："你是指上下文丢失，还是改变话题后无法回到主线？"
4. 最终输出：{ capability: "multi_turn_coherence", confidence: 0.82, clarification: "..." }
```

### 15.8.2 `attribute_failure_to_layer` 输出 schema

```json
{
  "failureLayer": "L5_tool_selection",
  "confidence": 0.78,
  "method": "rule | llm_classifier | experiment",
  "evidence": [
    "该 span 调用了 search_db 但应调用 search_kb",
    "工具描述歧义"
  ],
  "layerProbability": {
    "L0_model": 0.10,
    "L5_tool_selection": 0.55,
    "L8_generation": 0.20,
    "L4_retrieval": 0.10
  },
  "recommendedAction": "L5_tool_selection",
  "experimentSuggestion": {
    "type": "ab_test",
    "axis": "tool_description",
    "candidates": ["v1_current", "v2_disambiguated"],
    "estimatedCost": "low"
  }
}
```

### 15.8.3 `suggest_optimization_path` 输出示例

```json
{
  "capabilityDimension": "multi_turn_coherence",
  "rankedActions": [
    {
      "action": "L3_memory:enable_summarization",
      "expectedLift": 0.15,
      "cost": "low",
      "rationale": "73% bad cases 在 turn ≥ 5 的 context 截断，summarization 是最低成本试错"
    },
    {
      "action": "L0_model:switch_to_long_context_model",
      "expectedLift": 0.05,
      "cost": "high",
      "rationale": "模型层不是主因，且换模型成本高"
    }
  ]
}
```

## 15.9 候选实验路由（Model / Config Matrix）

实验路由让用户能跑：

```
同一组 eval cases × 候选维度矩阵
  ├─ Model: [GPT-4o, Claude 3.5, Qwen-72B, DeepSeek-V3]
  ├─ Temperature: [0.0, 0.3, 0.7]
  ├─ System Prompt: [v1, v2, v3]
  └─ Tool Schema: [strict, lenient]
   → 输出每个组合的 metrics 矩阵
   → Copilot 推荐性价比最高的组合
```

约束：

1. 实验必须绑定到一个 `capabilityDimension`，跑该 capability 的定向评测集，不混用。
2. 实验自变量必须落在 9 层 harness 之一，不允许"杂项"。
3. 多自变量组合时建议用拉丁方（Latin Square）减少组合爆炸；具体方法在工程实现时定。
4. 实验结果写入 `experiment_routes.result_summary`，并自动生成一份 `capability_attributions` 记录。

详细实现（包括 model gateway / config 注入 / 结果对比）属于 P5 阶段，本文档先冻结口径。

## 15.11 验收

**MVP（P3）验收：**

1. `eval_cases` 表接受 `capability_dimension` 为可选标注；有标注时可按维度过滤查询，无标注时不抛错。
2. 任一 capability 桶都能查到"6+2 类 source 各有多少条"，source 计数可按 capability 过滤（`?capability=X`）。
3. 合成样本和导入样本若已指定 `capabilityDimension` 则落库，未指定则留空（MVP 不强制）。
4. `failure_layer` 字段存在于 `eval_cases` 表，支持写入但无约束；goodcase 写入时必须为 NULL。

**Phase 2（P4+）验收（Post-MVP）：**

5. 任一 badcase 都能回答："属于哪个能力维度？失败在哪一层？归因依据是什么？"
6. 任一 capability 桶都能回答："6+2 类信号各有多少？层级分布如何？主疑似层是谁？"
7. Copilot 在用户说"我想优化 X 能力"后，能抽出 capabilityDimension → 拉对应桶 6 类信号 + 层级分布 → 给出归因 → 推荐优化路径（按性价比排序）。
8. 任一对照实验都能在 `experiment_routes` 中追溯到自变量、候选值、评测集、结果矩阵。
9. 前端不展示 `harness layer` 英文枚举值，必须按词典翻译为中文（如 L5 → "工具选择层"）。
10. **泛化验证**：任一 capability 桶的 validation 结论附带 `gen_coeff` 和 bootstrap 95% CI；`gen_coeff < 0.5` 或 `ci_95_lower ≤ 0` 时不通过（详见 [`11-测试与回归.md`](./11-测试与回归.md) 11.12）。
