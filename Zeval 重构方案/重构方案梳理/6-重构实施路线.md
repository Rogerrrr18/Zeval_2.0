# 6-重构实施路线

## P0：方案冻结

目标：把黑盒指标与数据流讲清楚，冻结 Supabase-only 目标方案。

交付：

- 本目录 PRD 文档。
- 主客观指标 XLSX。
- 指标计算 DAG。
- Supabase 目标 schema 草案。
- CRUD 复用映射。

验收：

- 用户能通过指标表理解每个指标的输入、计算时机、依赖和输出。
- 用户能通过 DAG 理解 LLM 何时介入。
- 后端目标不再出现 local-json/filesystem 作为正式方案。

## P1：评估投影迁移

目标：先让一次评估运行完整落入 Supabase。

范围：

- `evaluation_runs`
- `sessions`
- `message_turns`
- `topic_segments`
- `objective_signals`
- `subjective_signals`
- `risk_tags`
- `evidence_spans`
- `suggestions`

验收：

- `POST /api/evaluate` 保持响应不变。
- Supabase 中可查到 run、topic、指标、证据、建议。
- LLM 失败时 `judge_failures` 或等价字段可观察。

## P2：Baseline 与在线评测迁移

目标：替换 workbench baseline filesystem store。

范围：

- `baselines`
- `baseline_runs`
- `online_eval_runs`
- `replay_turns`
- `run_comparisons`

验收：

- 首页保存 baseline 后可在在线评测页按客户和时间发现。
- 在线评测可对比 baseline 与当前 run。
- 对比图表可追溯到两次 `evaluation_run_id`。

## P3：评测集 + 自动入池 + 回归迁移 + 能力维度落库（MVP）

目标：把 goodcase / badcase、sample batch、validation run 全部落库；同步上线"自动入池"能力，让评估完成后能自动产出候选 case，详见 [`13-案例池自动入池规则.md`](./13-案例池自动入池规则.md)；同步上线**评测集按能力维度可选标注**的最小闭环，让 case 可选携带 `capability_dimension`。**harness 失败层归因（`failure_layer` / L0–L8）为 Post-MVP，推迟至 Phase 2（P4+）。**

范围：

- `eval_cases`（含 `source` 8 类 / `capability_dimension` **可选** / `failure_layer` **字段保留但 MVP 不强制** / `attribution_confidence` / `attribution_method` / `metadata.false_positive` / `metadata.near_duplicate_of`）
- `eval_case_baselines`
- `eval_case_candidates`（可选携带 `capability_dimension`；`failure_layer` 为 Post-MVP）
- `dataset_admission_rules`
- `sample_batches`
- `sample_batch_cases`
- `validation_runs`
- `validation_results`

API：

- `/api/eval-datasets/cases`、`/api/eval-datasets/cases/[id]`（GET 支持 `source` 过滤；PATCH 支持 `metadata.false_positive`）
- `/api/eval-datasets/candidates`
- `/api/eval-datasets/admission-rules`、`/api/eval-datasets/admission-rules/[id]`
- `/api/eval-datasets/sample-batches`（POST 支持 `includeSynthesized`）
- `/api/eval-datasets/harvest-badcases`（行为升级为：跑规则 → 候选 → 去重 → 升级为 case）
- `/api/validation-runs`、`/api/validation-runs/[id]`

验收：

- case 去重 hash 在 Supabase 唯一约束中生效；近似去重落入候选并附 `near_duplicate_of_case_id`。
- sample batch 可按 goodcase / badcase 分层抽样，并按需排除合成样本。
- validation run 可复盘每个 case 的通过/失败原因。
- 一次评估完成后，可在 Supabase 中查到 `eval_case_candidates`，并按规则自动晋升或停留在 `pending_review`。
- 标记为 `false_positive` 的旧 case，再次评估同一 session 不会被自动入池。
- `dataset_admission_rules` 启用 / 禁用立即生效。
- `capability_dimension` 为可选字段，有则写入，无则留空；`failure_layer` 字段存在于 schema 但 MVP 不做 check 约束，`failure_layer_required_for_badcase` 约束推迟至 Phase 2。
- `GET /api/eval-datasets/cases?source=auto_fn` 能按 source 过滤；`?capability=multi_turn_coherence` 过滤在 case 已填写该字段时有效。

## P4：调优包 Skill 化 + 长尾合成 + Agent 迁移 + 泛化验证机制 ⚠️（Post-MVP 规划）

目标：

1. 把 remediation package、agent run、jobs 从文件 / JSON 队列迁入 Supabase。
2. 调优包升级为 Skill 文件夹交付，详见 [`12-调优包-Skill化交付.md`](./12-调优包-Skill化交付.md)。
3. 上线长尾样本合成能力，详见 [`14-样本合成与长尾覆盖.md`](./14-样本合成与长尾覆盖.md)。
4. **上线泛化验证机制（M1–M5）**，确保修复改善具有举一反三的泛化性，详见 [`11-测试与回归.md`](./11-测试与回归.md) 11.12。

范围：

- `remediation_packages`（含 `skill_version` / `skill_artifact_uri` / `skill_metadata`）
- `remediation_artifacts`（含 `skill_md / metadata_json / prompt_md / script_md` 等 kind）
- `synthesis_templates`
- `synthesis_runs`
- `synthesized_samples`
- `agent_runs`
- `jobs`
- **`eval_cases` 新增 `holdout boolean` 字段**（M1 Seen/Holdout 分离）
- **`validation_generalization_reports` 新表**（M4/M5 泛化 + bootstrap CI）
- `acceptance_gate` schema 新增 `min_n_per_capability` 字段（M2）

API：

- `/api/remediation-packages`、`/api/remediation-packages/[id]`
- `/api/remediation-packages/[id]/skill`
- `/api/remediation-packages/[id]/skill/download`
- `/api/remediation-packages/[id]/skill/regenerate`
- `/api/eval-datasets/synthesize`（新增 `targetHoldout` / `sourceCase` / `sceneVariations` 参数，M3）
- `/api/eval-datasets/synthesize-probes`（M3 专用探针合成接口）
- `/api/eval-datasets/synthesis-runs`、`/api/eval-datasets/synthesis-runs/[id]`
- `/api/eval-datasets/synthesis-templates`、`/api/eval-datasets/synthesis-templates/[id]`
- `/api/eval-datasets/cases/counts`（新增 `?group=capability,holdout` 参数）
- `/api/agent-runs`、`/api/agent-runs/[id]`
- `/api/jobs`、`/api/jobs/[id]`、`/api/jobs/run`
- `/api/validation-runs/[id]/generalization`（查询泛化验证报告）

验收：

- 调优包可从 badcase 生成并导出为 Skill 文件夹（zip / tar），结构与 12 号文档一致。
- Skill 在 Claude Code / Codex / Cursor 中通过中文或英文 trigger 触发后，能按 `actions` 顺序执行并报告 acceptance-gate 结果。
- 客户多次下载同一 `packageId` 的 Skill 不破坏服务端状态，接口幂等。
- 长尾合成接口可写入 `synthesis_runs / synthesized_samples`，合成样本默认 `eval_cases.source = "synthesized"`，对齐分低于阈值的样本自动拒绝并落 `rejected_count`。
- baseline 与在线评测不读取 `source = "synthesized"` 的样本。
- Agent 执行状态可查询、可重试、可审计；jobs 不再依赖本地 JSON 文件。
- **泛化验证（M1–M5 全部验收）**：
  - `eval_cases.holdout = true` 的 case 不暴露给修复者；每个 capability 桶 holdout count ≥ 10。
  - N < 10 的桶只输出 `INSUFFICIENT_EVIDENCE`，不产出 pass/fail。
  - `npm run generalization:ci` 跑通，结果写入 `validation_generalization_reports`。
  - `gen_coeff < 0.5` 或 `ci_95_lower ≤ 0` 时整包 fail。
  - 合成探针（`targetHoldout=true`）全部落入 holdout，不可在修复者视图中出现。

## P5：能力归因 + Copilot 决策 + 实验路由 ⚠️（Post-MVP 规划）

目标：把 P3 已落的 `capability_dimension / failure_layer` 升级为完整的"归因 + 决策 + 实验"闭环，详见 [`15-能力维度评测与归因.md`](./15-能力维度评测与归因.md)。

范围：

- `capability_attributions` 表上线，记录每条 badcase 的层级归因决策与依据。
- `experiment_routes` 表上线，支持候选实验路由（model / prompt / tool schema 多自变量矩阵）。
- Copilot 新增 4 个 skill：`diagnose_capability_gap` / `compose_eval_dataset_for_capability` / `attribute_failure_to_layer` / `suggest_optimization_path`。
- 调优包 metadata 增加 `capabilityDimension / targetFailureLayer / experimentRouteId`。

API：

- `/api/copilot/diagnose-capability` POST — 用户痛点描述 → `capabilityDimension` 推断。
- `/api/copilot/attribute-failure` POST — 单条或批量 badcase → 层级归因输出。
- `/api/copilot/suggest-optimization-path` POST — 归因结果 + 候选矩阵 → 性价比排序的优化路径。
- `/api/experiments` GET / POST、`/api/experiments/[id]` GET / PATCH。
- `/api/eval-datasets/cases` GET 增加 `capability` 过滤参数。

验收：

- 用户在 Copilot 输入"我想优化多轮对话"，Copilot 能：抽出 `capabilityDimension` → 拉对应桶的 6 类信号 + 层级分布 → 给出归因 → 推荐优化路径，全链路可重放。
- 任一对照实验都能在 `experiment_routes` 中追溯到自变量、候选值、评测集、结果矩阵。
- 任一 badcase 的归因都能在 `capability_attributions` 中查到 method / confidence / evidence / experiment_run_id。
- 调优包导出时 metadata 中三个新字段必填，缺失则导出失败。

## 风险与控制

| 风险 | 控制 |
| --- | --- |
| 一次性迁移范围过大 | 先迁移 evaluation projection，再迁移业务闭环 |
| 指标口径漂移 | 以 XLSX + metric registry 双重约束 |
| LLM 思考输出污染 JSON | 默认模型 `Qwen/Qwen3.5-27B`，`ZEVAL_JUDGE_ENABLE_THINKING` 与 `SILICONFLOW_ENABLE_THINKING` 必须同时为 `false`，并记录 parse failure |
| 历史文件数据丢失 | 迁移前做只读归档，迁移脚本幂等 |
| 自动入池误判扩散 | `false_positive` 标记必须被自动入池流程尊重；候选默认 `pending_review` |
| 长尾合成污染基线 | 合成样本独立 `source`，baseline / 在线评测自动排除 |
| 自动归因低置信 | `attribution_confidence < 0.4` 的候选默认 `pending_review`，等待对照实验或人工确认；不允许直接入正式 `eval_cases` |
| 实验组合爆炸 | 多自变量时建议拉丁方设计，单次最多 3 个自变量；`experiment_routes.candidates` 上限默认 12 |
| Skill 包内泄露密钥或 PII | Skill 文件中不得包含 API Key / DB 密码；证据片段最小披露 |
| 页面变复杂 | 内部逻辑只进文档，不进用户页面 |
| 小样本误判泛化 | N < 10 的 capability 桶严禁产出 pass/fail；`insufficient_evidence_policy = "block"` 为默认；PM 不允许以"样本不足"为由人工覆盖 gate 结论 |
| Seen/Holdout 污染 | `holdout=true` 的 case 必须在 API 层过滤（需 `role=gate_reviewer` 权限）；修复者的案例池视图默认只显示 `holdout=false` |
| 合成探针过于相似 | 探针 `scene_diversity_score < 0.6` 时强制拒绝，不重试；避免生成"换几个词"的伪多样性探针 |

## 非目标

- 本轮不做登录系统和复杂权限 UI。
- 本轮不做大规模任务编排平台。
- 本轮不把 Scenario 固化为首场景。
- 本轮不保留本地兼容方案作为正式架构分支。
- 本轮不在 Skill 包里做交互式向导，Skill 仅以"读 → 跑 → 报告"线性结构交付。
- 本轮不用合成样本去更新对外的统计基线指标。
- **MVP 阶段不做 harness 失败层（L0–L8）归因与强制约束**，`failure_layer` 字段保留于 schema，待 Phase 2（P4+）再实施层级归因、DB check 约束与 Copilot 归因决策。
