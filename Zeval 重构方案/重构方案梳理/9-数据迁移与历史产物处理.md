# 9-数据迁移与历史产物处理

重构后 Zeval 后端只使用 Supabase。本文档定义如何把当前散落在 filesystem / `local-json` / 旧 Postgres 兼容表中的历史产物，一次性迁移到新表结构，以及之后的处理原则。

## 9.1 迁移目标

| 现有产物 | 来源路径 / 适配器 | 目标 Supabase 表 |
| --- | --- | --- |
| 评估投影 | `local-json-database`，部分 `zerore_records` | `evaluation_runs / topic_segments / objective_signals / subjective_signals / risk_tags / evidence_spans / suggestions` |
| baseline 快照 | `mock-chatlog/baselines/*` | `baselines / baseline_runs` |
| 评测集 case | `src/eval-datasets/storage/file-system-dataset-store.ts` | `eval_cases / eval_case_baselines` |
| sample batch | 同上 | `sample_batches / sample_batch_cases` |
| 调优包 | `src/remediation/file-system-package-store.ts` | `remediation_packages / remediation_artifacts` |
| validation 运行 | `src/validation/file-system-validation-run-store.ts` | `validation_runs / validation_results` |
| agent 运行 | `src/agent-runs/file-system-agent-run-store.ts` | `agent_runs` |
| jobs 队列 | 本地 JSON | `jobs` |
| traces | 内存环缓冲 | 不迁移，重新启动后丢弃（与决策一致：MVP 仅保留接口） |

## 9.2 迁移原则

1. 一次性迁移：迁移脚本运行成功后，旧文件冻结为只读；不再回写任何业务数据到文件。
2. 幂等：迁移脚本必须可重复运行，重复执行不产生重复行（依靠业务唯一键 + Supabase 唯一约束）。
3. 可追溯：迁移过程中保留旧 ID 与新 ID 的映射，写入审计或日志。
4. 不阻塞：若某条历史记录字段缺失或解析失败，跳过并记录到迁移报告，不阻断整体迁移。
5. 不要补救脏数据：旧字段不一致时按目标表 schema 默认值填充，并在 `metadata.import_warnings` 中记录原因。

## 9.3 迁移步骤建议

```text
1. 准备目标表：执行 supabase-target-schema.sql。
2. 读取旧 filesystem store / local-json：
   - 评估投影：扫描 zerore_records 与 local json 文件。
   - baseline：扫描 mock-chatlog/baselines。
   - 评测集 / 调优包 / validation：扫描各自目录。
3. 字段映射：
   - workspaceId → projectId（保持值不变，仅改字段名）。
   - 旧时间戳保持原值或解析为 ISO 8601。
4. 写入 Supabase：
   - 优先按主键或业务唯一键 upsert。
   - 关联表（如 sample_batch_cases）等父表写入完成后再插入。
5. 验证：
   - 抽样比对：旧 vs 新关键字段 hash 一致。
   - 计数校验：每张表迁移条数与源条数差值不大于阈值。
6. 切流：
   - 新代码强制 ZEVAL_DATABASE_ADAPTER=postgres。
   - 删除 filesystem store 实例化代码，保留迁移脚本入口。
```

## 9.4 关键字段映射

### 9.4.1 baseline

| 旧文件字段 | 新表字段 |
| --- | --- |
| 文件名中的 `customerId/runId` | `baselines.customer_id`、`baseline_runs.id` |
| `rawRows.json` | `baseline_runs.raw_rows_artifact_uri` 或 `snapshot_payload.rawRows` |
| `summary.json` | `baseline_runs.snapshot_payload.summary` |

### 9.4.2 eval_cases

| 旧字段 | 新字段 |
| --- | --- |
| `caseId` | `eval_cases.id` |
| `caseSetType` | `eval_cases.case_set_type` |
| `transcript` | `eval_cases.transcript_payload` |
| `normalizedTranscriptHash` | `eval_cases.normalized_transcript_hash` |
| `labels` | `eval_cases.labels` |
| `sourceRunId` | `eval_cases.source_evaluation_run_id` |

### 9.4.3 evaluation 投影

旧 `zerore_records` 中的 JSONB 通过类型字段定位到目标表，并按 `evaluation_run_id` 串联。

## 9.5 旧适配器去留

| 适配器 | 处理 |
| --- | --- |
| `src/db/local-json-database.ts` | 重构后不再作为正式运行路径，可保留作为迁移源读取器 |
| `src/db/postgres-database.ts` | 保留，但目标只连 Supabase；删除 `zerore_records` 相关写路径 |
| `FileSystemDatasetStore / FileSystemWorkbenchBaselineStore / FileSystemPackageStore / FileSystemValidationRunStore / FileSystemAgentRunStore` | 重构后停用；保留 import-only 模式给迁移脚本读取 |
| `src/queue/index.ts` JSON 队列 | 替换为 Supabase `jobs` 表 + 简单 worker |

## 9.6 迁移完成后的清单

- 新代码所有 store 仅依赖 Supabase。
- `mock-chatlog/baselines/*` 标记为 deprecated，不再写入。
- `.env.local` 删除 `ZEVAL_DATABASE_ADAPTER=local-json`。
- 删除 `DATASET_STORE_PROVIDER=filesystem` 等老开关，改为只读迁移参数。
- README 中 “Project Structure” 不再展示 filesystem store 模块。

## 9.7 失败回滚

- 迁移前对 Supabase 表做一次空表快照。
- 若关键校验失败，回滚目标表到空表快照，并修正脚本后重跑。
- 不在生产环境中长时间双写：避免数据漂移。

## 9.8 后续维护

- 迁移完成后，所有新增功能直接面向 Supabase 表。
- 历史 PRD 中描述的 `workspace_id`、`zerore_records`、`local-json` 仅出现在 `../原始prd参考/` 与本文件中作为历史背景，不进入新代码。
- 一次性迁移脚本作为只读工具保留在 `scripts/migrations/`，下个里程碑再决定是否归档。
