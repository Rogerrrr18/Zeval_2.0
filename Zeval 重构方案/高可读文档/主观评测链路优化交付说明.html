上传至主观评测链路优化：交付说明

本页面面向协作复盘：说明从对话日志进入评估链路开始，到主观指标含 LLM Judge 完成之间， 我们实际实现了哪些优化，并逐条回答协作文档提出的并发、汇总、降级与可观测问题。

已实现 topic 切分 LLM Judge 有界并发 全局并发池 降级状态可观测 PR 已提交

一、范围与边界

本轮处理的链路

只处理上传后的评估主链路：raw chatlog 进入 normalize、topic 切分、分段情绪补全、 客观指标、主观指标、LLM Judge、汇总与响应 metadata。

本轮不处理的内容

不处理在线回放范式、调优包、中转站、供应商多活、持久化熔断状态，也不实现当前链路中不存在的 auto_disagreement 双轨评估冲突。

核心判断： 当前系统是 fused pipeline：客观补全先产生中间层，主观指标再引用这些中间层。 它不是规则轨与 LLM 轨并行裁决同一 case 的双轨架构，因此“评估器冲突”只能作为 Post-MVP 预留。

二、链路现状说明

rawRows -> normalizeRawRows -> buildTopicSegments -> topic_continuity_review? [LLM, 长间隔时触发] -> scoreTopicSegmentEmotions -> segment_emotion_baseline? [LLM, 每个 topic segment 可触发] -> buildObjectiveMetrics [规则] -> buildSubjectiveMetrics -> buildImplicitSignals [规则] -> buildGoalCompletions -> goal_completion_judge? [LLM, unclear 才触发] -> buildRecoveryTraces -> recovery_trace_strategy? [LLM, completed trace 才触发] -> subjective_dimension_judge? [LLM, session 级四维 Judge] -> aggregation / charts / suggestions / response meta
阶段 代码入口 LLM stage 并发与失败行为
normalize src/pipeline/normalize.ts 无 本地规则，输出稳定 turnIndex、timestampMs、activeHour。
topic 切分 src/pipeline/segmenter.ts topic_continuity_review 单 session 内顺序切分；跨 session 使用 ZEVAL_JUDGE_TOPIC_CONCURRENCY 有界并发； LLM 请求进入全局池。强调用失败会让 parse stage 失败。
分段情绪补全 src/pipeline/emotion.ts segment_emotion_baseline 使用 ZEVAL_JUDGE_SEGMENT_CONCURRENCY 有界并发； 单 segment LLM 失败回退规则情绪基线。
目标完成 src/pipeline/goalCompletion.ts goal_completion_judge 规则硬信号优先，只有 unclear 才调 LLM；多 session 使用 ZEVAL_JUDGE_GOAL_CONCURRENCY 有界并发。
恢复轨迹 src/pipeline/recoveryTrace.ts recovery_trace_strategy 依赖 goal completion；仅 completed trace 调 LLM 总结修复策略； 使用 ZEVAL_JUDGE_RECOVERY_CONCURRENCY 有界并发。
四维主观 Judge src/pipeline/subjectiveMetrics.ts subjective_dimension_judge session 级单次调用，transcript 内包含多个 topic segment； 多 session 使用 ZEVAL_JUDGE_SESSION_CONCURRENCY 有界并发。
全局 LLM 请求池 src/lib/siliconflow.ts 所有 LLM stage 所有 LLM 请求共享 ZEVAL_JUDGE_GLOBAL_CONCURRENCY，并记录 queuedMs、durationMs、attempts、errorClass、degradedReason。

三、我们做了什么

1. topic 与 emotion 补全

完成按 session 顺序构建 topic；长间隔处可触发 continuity LLM。 分段情绪从无界 Promise.all 改为有界并发。

2. 主观指标并发

goal completion、recovery trace、四维 Judge 都改为 session 级有界并发。 四维 Judge 与 goal→recovery 链并行启动。

3. 全局并发池

所有 SiliconFlow / OpenAI-compatible 请求共享进程内全局池， 避免多个阶段叠加后打爆供应商限流。

4. 降级与状态机

新增 meta.runState、meta.stageStatuses、 meta.llmJudge，区分完整成功、部分降级和硬失败。

5. 聚合解释性

新增 dimensionBreakdowns 与 aggregation， 保留原有四维全局摘要，同时给出 session 级明细。

6. 重试与熔断

已实现重试、指数退避、错误分类、stage-local 轻量熔断； 熔断命中沿用原有降级或抛错策略。

四、明确回答 4.1：并发与队列

问题 1：分段情绪 LLM 与 session 级主观 Judge 应共享全局并发池还是分阶段独立 limit？

答案：共享全局并发池，同时保留阶段级 limit。 阶段级 limit 控制某一类任务的排队形态；全局池控制实际打到模型网关的请求总数。 这样既避免 segment emotion 抢占全部资源，也避免多个阶段并发叠加后超过供应商限流。
设计选择 为什么这样设计 放弃了什么 风险与缓解
阶段级 limit + 全局池 阶段级 limit 让 topic、emotion、goal、recovery、dimension 各自可控； 全局池保证实际供应商请求不会被多个阶段叠加打爆。 放弃“每个阶段独立放开并发”的最低延迟幻想，因为它会把限流风险转移给供应商 API。 队列可能增加单个请求等待时间；用 queuedMs、avgQueuedMs 暴露排队压力， 后续可按数据调大或调小各阶段 limit。
跨 session 并发，单 session 内顺序 topic 切分中 active segment 依赖前一轮判断，单 session 内并发会破坏切分语义； 不同 session 互不依赖，因此可以并发。 放弃“每条消息都并行切分”的极端速度方案。 单个超长 session 仍会成为局部慢点；后续如遇到超长日志，可再做 session 内窗口化切分。
配置 作用
ZEVAL_JUDGE_GLOBAL_CONCURRENCY 所有 LLM 请求共享的供应商侧总并发上限。
ZEVAL_JUDGE_TOPIC_CONCURRENCY 跨 session topic continuity review 的并发上限；单 session 内仍顺序。
ZEVAL_JUDGE_SEGMENT_CONCURRENCY segment emotion baseline 的阶段内并发上限。
ZEVAL_JUDGE_SESSION_CONCURRENCY 四维 subjective dimension judge 的 session 并发上限。
ZEVAL_JUDGE_GOAL_CONCURRENCY / ZEVAL_JUDGE_RECOVERY_CONCURRENCY 目标完成与恢复轨迹 LLM 阶段的并发上限。

问题 2：哪些阶段适合异步化，哪些必须阻塞在单次评估响应内？SLA 与可观测字段怎么定义？

答案：MVP 中，生成当次评估结果所必需的阶段继续阻塞；可复算、可后补、可投影的阶段异步化。 当前同步 / SSE / asyncMode 都复用同一条 pipeline；projection 与结果保存失败只进入 warnings，不阻断主响应。
设计选择 为什么这样设计 没有选择的方案 权衡结果
主结果链路阻塞，投影和保存非阻塞 用户点击评估后需要拿到同一份完整报告；normalize、topic、emotion、objective、subjective 都是报告内容本身。 projection 和保存只是后续检索、落库、复盘能力，不应该让一次评估因为写库失败而整体失败。 全链路都异步化，用户只拿 jobId；或者所有落库失败都让请求 500。 当前体验更直接：小样本同步可用，大样本可用 asyncMode；写入失败通过 warnings 和 runState=degraded 告知。
SSE / asyncMode 复用同一 pipeline 避免三套入口出现不同评估口径；同步 JSON、SSE、async job 只是在交互形态上不同。 为 async job 单独实现一套离线逻辑。 牺牲一些针对离线批量的极致吞吐，但换来口径一致和更低维护成本。
阶段 是否阻塞单次响应 是否适合异步化 理由
normalize / topic segment 是 否 后续 enriched rows、图表和主观 Judge 都依赖它。
segment emotion 是 部分可后置 当前情绪曲线、恢复判断和四维 Judge 输入依赖情绪分。
objective metrics / implicit signals 是 否 本地规则成本低，是主观指标和建议生成基础。
goal completion / recovery trace / subjective dimensions 是 批量场景可 job 工作台需要当次看到主观结论；大批量评测可走 asyncMode。
evaluation projection / result persistence 否 是 失败已进入 warnings，不阻断 200 / SSE result / job success。
badcase 入池与后续数据集维护 否 是 候选入池可异步重试，避免拖慢用户侧反馈。

SLA 建议

  • 同步工作台小样本目标 30-60 秒返回。
  • 超过阈值建议走 asyncMode。
  • SSE 必须持续发 stage running / done / failed。

可观测字段

  • stage：status / durationMs / degradedReason
  • LLM：queuedMs / durationMs / attempts / errorClass
  • job：queuedAt / startedAt / finishedAt / lastError

问题 3：是否应按 segment 拆分四维 Judge，还是保留 session 级单次调用？

答案：MVP 保留 session 级单次调用。 四维 Judge 的输入会把同一 session 下的多个 topic segment 拼入 transcript， 这样保留上下文完整性、调用次数更低、API 契约稳定。只有出现长会话 token 超限或产品明确需要“每个 topic 的四维评分”时，再引入 segment 级 Judge。
方案 优点 风险
session 级 Judge（当前） 上下文完整；调用次数低；聚合简单；契约稳定。 长 session token 较高；失败粒度较粗。
segment 级 Judge 失败粒度细；可局部重试；可按 topic 展示。 调用数显著增加；跨 topic 上下文丢失；聚合语义复杂。
权衡结论： 本轮优化的目标是“上传后尽快得到稳定可解释的主观指标”，不是把每个 topic 都变成独立主观报告。 因此四维 Judge 保持 session 级更符合当前产品语义；segment 级结果先通过 dimensionBreakdowns.topicSegmentIds 建立关联，给未来升级留位置。

五、明确回答 4.2：数据汇总与语义

问题 4：多 session、多 segment 的四维聚合是否满足业务可解释性？如何兼容 API 与前端？

答案：当前聚合可作为 MVP 全局摘要，但不能当作严格统计结论；因此已补充 session 级明细和聚合口径 metadata。 原有 subjectiveMetrics.dimensions 保持不变，前端不需要立刻改； 新增 dimensionBreakdowns 和 aggregation 支持展开解释和后续前端升级。
字段 现在怎么做 业务含义
dimensions 保留全局四维摘要。 保证现有 API 与前端展示兼容。
dimensionBreakdowns 按 session 输出原始四维 review、weight、source、succeeded、topicSegmentIds。 让长 session 权重、失败回退、证据来源可解释。
aggregation 声明 session_row_weighted_average、session_row_count、证据合并限制。 让消费方知道这是加权近似摘要，不是人工标注真值。
设计选择 为什么这样设计 放弃了什么 风险与缓解
保留全局摘要，追加 session 明细 当前前端和 API 消费方已经依赖 dimensions； 直接改成多 session / 多 segment 结构会破坏展示与下游契约。 放弃“这次直接重做主观指标数据结构”的方案。 全局摘要仍会隐藏少数 session 的异常；通过 dimensionBreakdowns 暴露原始 session 级结果， 让前端后续可以展开查看。
按 session 行数加权 行数越多代表该 session 在上传日志中占比越高，作为 MVP 权重容易解释，也不需要额外业务字段。 放弃按用户价值、case 难度、人工标签置信度加权，因为当前瘦日志没有这些稳定字段。 长 session 可能放大权重；通过 aggregation.weightBasis 明示口径， 并保留 breakdown 给后续改成 capped weight 或 badcase-priority weight。
reason / evidence 近似合并 全局摘要需要短文本，不能把每个 session 的证据全部塞进主卡片。 放弃在摘要层展示完整证据链。 摘要可能解释不完整；完整证据留在 dimensionBreakdowns.dimensions 中， 摘要层只承担快速浏览。

问题 5：隐式信号、客观指标与主观 Judge 的因果顺序是否应显式化？

答案：应该，而且已经显式化。 文档中写明阶段顺序；运行时响应新增 meta.stageStatuses 与 meta.llmJudge，可看到每个 stage 的状态、耗时、LLM 请求统计和降级原因。
客观补全：normalize -> topic -> emotion -> objective metrics 主观输入：implicit signals + enriched rows + goal/recovery context 主观输出：goalCompletions + recoveryTraces + dimensions 汇总输出：aggregation + dimensionBreakdowns + meta.runState + meta.llmJudge

六、明确回答 4.3：报错、降级与产品一致性

问题 6：当前降级路径对用户可见字段有什么影响？

场景 HTTP / Job 行为 用户可见字段
useLlm=false, judgeRequired=false 评估成功 subjectiveMetrics.status=degraded；meta.llmJudge.enabled=false；warnings 有主观降级提示。
useLlm=false, judgeRequired=true 入口 400；pipeline 也保护性抛错 返回 error，不产出评估结果。
单 session LLM 失败且 judgeRequired=false 评估成功 对应 session 回退规则结果；subjectiveMetrics.status=degraded；meta.llmJudge.failedRequests > 0。
LLM 失败且 judgeRequired=true 硬失败 同步 JSON 500；SSE type=error；async job failed。
projection / result persistence 失败 主评估成功 meta.warnings 追加原因；meta.runState=degraded；不阻断 200 / SSE result / job success。

问题 7：API 是否需要区分“部分降级仍可 200”和“硬失败 5xx”？如何落在响应 JSON？

答案：需要，且已落在 meta。 HTTP 契约保持兼容：可用结果继续 200，硬失败才 5xx；细粒度状态通过 meta.runState 与 meta.stageStatuses 表达。
设计选择 为什么这样设计 放弃了什么 风险与缓解
部分降级仍返回 200 用户上传后最需要的是可用评估结果；如果主链路已经完成，只因某个 LLM session 或 projection 失败就 500， 会让用户失去已有的客观指标、规则结果和图表。 放弃“只要任何非核心步骤失败就整体失败”的强一致接口。 200 可能掩盖质量下降；用 runState=degraded、warnings 和 stageStatuses.degradedReason 明确告诉前端和用户。
judgeRequired=true 时硬失败 有些使用方要求 LLM Judge 是强依赖，这时规则 fallback 不能冒充主观评测完成。 放弃“一律降级成功”的宽松策略。 强依赖模式可用性更低；通过请求参数让调用方自己选择严格性。
{ "meta": { "runState": "ready | degraded | failed", "stageStatuses": [ { "stage": "subjective", "status": "ready | degraded | failed", "durationMs": 1234, "degradedReason": "llm_judge_failed_requests" } ], "llmJudge": { "totalRequests": 12, "failedRequests": 1, "stages": [ { "stage": "subjective_dimension_judge", "avgQueuedMs": 30, "avgDurationMs": 1800 } ] } } }

问题 8：429、超时、JSON 解析失败如何重试、熔断、退避？MVP 边界是什么？

答案：已实现重试、退避、错误分类和进程内 stage-local 轻量熔断。 408 / 409 / 425 / 429 / 5xx / timeout / network 会按 ZEVAL_JUDGE_RETRY_ATTEMPTS 重试； 延迟使用指数退避加 jitter；JSON 解析失败、无有效内容等非瞬态错误不重试。
能力 当前实现 MVP 边界
重试 ZEVAL_JUDGE_RETRY_ATTEMPTS，默认 3。 只对可恢复错误重试。
退避 指数退避 + jitter。 避免瞬时限流继续打爆供应商。
错误分类 rate_limited / timeout / provider_4xx / provider_5xx / invalid_response / network / circuit_open 进入 meta.llmJudge.recentRequests，便于排障。
熔断 ZEVAL_JUDGE_CIRCUIT_BREAKER_FAILURES + ZEVAL_JUDGE_CIRCUIT_BREAKER_COOLDOWN_MS 当前为进程内 stage-local；不做跨实例持久化熔断。
供应商多活 未做 Post-MVP，避免本轮引入更大运维面。
熔断权衡： 本轮选择进程内 stage-local 熔断，而不是持久化全局熔断。原因是 MVP 先解决“同一进程内连续失败还继续排队请求”的资源浪费； 持久化熔断需要跨实例共享状态、供应商健康检查和恢复策略，运维复杂度更高。当前风险是多实例部署时每个实例会独立判断熔断； 这被列为 Post-MVP，后续可以基于 judge_runs 或集中式状态存储升级。
