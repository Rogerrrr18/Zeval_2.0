▸ v2.1 改动说明（2026-05-10）

本次迭代在原有"评估 + 复盘 + 调优"主链路基础上，新增三处系统性改动。MVP 聚焦对话智能体评测主链路（P1–P3）；harness 层归因、Copilot 决策、泛化验证为 Phase 2（P4+） 规划。

① 能力维度评测体系（新增 15 号文档）— MVP：维度可选标注 / Phase 2：harness 归因

评测集主维度从"症状标签"切换为 12 个能力维度（instruction_following / multi_turn_coherence / tool_calling_correctness 等）。MVP 阶段 capability_dimension 为可选标注，支持按维度过滤案例池。Phase 2（P4+） 再启用 9 层 Canonical Harness（L0 模型底座 → L8 最终合成）的强制归因，届时 Copilot 新增 4 个决策 skill，实现「痛点描述 → 能力定向 → 故障层归因 → 优化路径推荐」全链路。

② 评测集 6+2 类信号通道（完善 13 号文档）

案例池不再只收"确认坏的对话"，按混淆矩阵四象限建立 8 类 source 通道：auto_tp（真坏）/ manual_fp（假阳性转正例）/ auto_fn（系统漏报）/ auto_tn（金标正例）/ auto_uncertainty（边界）/ auto_disagreement（评估器分歧）/ synthesized / imported。每个能力维度独立维护这组 8 类信号（不共用桶），是 Calibration 与归因决策的最小可信单元。

⚠️ 两项实现约束：
① auto_disagreement（评估器分歧）不由当前 fused 流水线自然产生。现行管线是"LLM 优先、规则兜底"的单轨串联，规则结果与 LLM 结果无法独立对比。要产生真正的分歧信号，需要双轨并行（规则链与 LLM judge 同时跑、事后对比），当前标记为校准专用通道，MVP 阶段手动录入。
② 瘦日志场景（仅有 Q&A 对 / final_message，无工具调用链、无 memory 命中记录）下，auto_fn 退化为 FN-proxy（文本可见信号：负向尾词、重复提问、长度骤降），harness 层 L4–L6 往往无法判定，failure_layer 默认为 unknown 或 L8。

③ 泛化验证机制 M1–M5（完善 11 号文档 11.12 节）— Phase 2（P4+）规划，MVP 不实施

原有 validation 只能证明"在见过的 badcase 上改善了"，无法保证能力泛化（举一反三）。M1–M5 填补这个空缺（Phase 2 P4+ 上线）：M1 Seen/Holdout 强制分离（30% holdout 修复者不可见，gen_coeff 衡量泛化度）；M2 最小样本量门禁（N < 10 只输出 INSUFFICIENT_EVIDENCE）；M3 合成变体探针（同能力跨场景变体注入 holdout）；M4 跨能力回归 Guard（防止修好 A 搞坏 B）；M5 Paired Bootstrap CI（ci_95_lower ≤ 0 不通过门禁）。相关表/字段已在 schema 中预留，MVP 阶段不写入数据。

Zeval 是把对话日志变成可解释质量指标的评估工作台

它接收 AI 对话日志，做字段补全与主题切分，再用规则与 LLM Judge 给出客观与主观指标，并把不达预期的会话（以及假阳性误判、漏报、边界案例）沉淀成多来源的评测集，最终驱动 Agent 真正变好。本页帮新人在不读源码的前提下理解项目背景、当前能力、为什么要重构以及目标态。

1. 这是个什么项目

Zeval 是面向 AI 对话产品的「评估 + 复盘 + 调优」工作台。它处理上传或接入的对话日志，输出可解释的指标、证据片段、可执行优化建议、badcase 与回归用例，并支持与基线（baseline）对比、在线回放、调优包生成。

评估

客观指标 + LLM 主观评审 + 隐式风险信号，给出统一结构化结果。

复盘

基线沉淀、在线回放对比、badcase 收获，让"为什么变差"可追溯。

调优

调优包（issue-brief / spec / badcases / acceptance-gate）驱动 Agent 修复并回归验证。

6. 为什么要重构

  1. 指标黑盒：用户难以追溯每个指标的输入、计算时机、依赖、降级行为和落点。
  2. 存储分裂：filesystem store / local-json / zerore_records 桥表 / postgres 适配并存，难以演进。
  3. 多套场景假设：历史 PRD 同时存在"B2B 客服优先""培训陪练优先""场景无关"三种判断。
  4. 多套指标表并存，易漂移。
  5. 主观评估失败时降级行为不一致，前端会出现规则降级伪装成 LLM 结果。
  6. baseline / 在线评测 / 回归 链路依赖文件路径，多客户多版本难以横向对比。

场景策略

保持当前 AGENTS.md 口径，先做场景无关通用评估框架，Scenario 作为可插拔扩展。

API 策略

历史 PRD 与配置

历史文档全文归档到 原始prd参考/，配置项只保留变量结构与用途模板，真实 key 不进入文档。

8. 保留的核心功能

  • 补全层：RawChatlogRow → NormalizedChatlogRow → EnrichedChatlogRow 中间层思想保留。
  • 主客观指标：客观稳定可算；主观由 LLM Judge 产出 score / reason / evidence / confidence。
  • 隐式推断：兴趣下降、理解障碍、情绪恢复失败信号保留，并要求按 run / session / segment 可追溯。
  • baseline 与在线评测：保留"工作台生成 baseline → 在线回放 → 多指标对比"闭环。
  • 评测集：goodcase / badcase、去重 hash、sample batch 抽样、回归验证能力保留。
  • 建议契约：所有建议保持"问题 → 影响 → 行动"模式，并显式绑定触发指标 key。

15. 前端展示约束

  • 不展示算法名、prompt stage、模块路径；用户看到的是业务语义。
  • 主观指标必须显示来源：LLM 评审 规则降级 不可用。
  • 建议卡片按"问题 → 影响 → 行动"展示，绑定触发指标的中文名。
  • 跨页跳转后状态保留（基于 sessionStorage 或后端 run）。
  • 从 UI 移除"启用 LLM 主观评估"复选框。

16. 测试与回归

必备脚本（MVP）

tsc --noEmit、npm run lint、npm run smoke:e2e:*、calibration:ci、db:smoke:evaluate-projection；generalization:ci 为 Phase 2 脚本。

客观回归

固定 fixture 的 snapshot：客观指标输出哈希稳定，改动需要解释。

主观回归

calibration gold set，分数误差与 reason 关键词一致率，阈值建议 0.85。

投影回归

一次 evaluate 必须能在 Supabase 中看到 runs / segments / signals / evidence 完整写入。

泛化验证（M1–M5）⚠️ Phase 2

Seen/Holdout 分离 + Bootstrap CI + 跨能力 Guard。N < 10 不产出 pass/fail 只输出 INSUFFICIENT_EVIDENCE。结果写入 validation_generalization_reports。MVP 阶段表/字段预留，不激活。

在线回放比对的统计局限：当前回放只保留用户原始话术，不重生成后续轮次，是一种"前缀重放"。比对结论仅适用于「同一用户话术下新版本回复的改善幅度」，不能模拟真实多轮发散。MVP 阶段比对结论附带 N（样本量），小样本时需在 UI 明确提示。Bootstrap CI 与 gen_coeff 门禁为 Phase 2（P4+） 机制。

17.2 案例池自动入池规则

评估完成后，按五条采集通道自动识别值得入池的会话，写入 eval_case_candidates，去重后晋升为正式 eval_cases。每条 case 必须显式标注 source，共 8 类：

source通道触发方式入池类型
auto_tpTP — 真坏案例规则 / 主观指标命中（goal_failed、low_empathy、interest_decline、recovery_failure 等）badcase
auto_fnFN — 漏报系统判 OK，但下游信号异常（负向尾、重复提问、长度骤降）badcase
auto_uncertainty边界案例Judge confidence ∈ [0.4, 0.6]，默认 pending_reviewbadcase
auto_disagreement评估器分歧规则命中 ≥ 1 项，但 LLM judge 整体判为达标badcase
auto_tnTN — 金标正例高质量会话抽样（目标达成 + 共情 ≥ 4，或按配置比例随机抽取）goodcase
manual_fpFP — 人工复审人工把 badcase 标记为假阳性 → 自动转为正例对照组goodcase
synthesized合成LLM 辅助生成，不进入 baseline 统计任意
imported外部导入批量导入历史案例任意
  • 去重：精确哈希 + 近似（长度差 / 字符差 / 命中规则相同）。
  • 假阳性（FP）：人工标记后 case 转为 source=manual_fp goodcase，同一 session 不再被 TP 通道重复入池。
  • auto_fn / auto_uncertainty / auto_disagreement 三类默认 pending_review，不直接参与回归统计，等待人工确认。

详见 13-案例池自动入池规则.md。

17.4 能力维度评测与归因

Zeval 不只是"对话质量评估工具"，目标是"Agent 优化决策系统"。把用户语言（"我想优化多轮对话"）映射到工程语言（改 model / prompt / harness 哪一层），靠两张表 + 一个数据立方体。

3D 立方体：评测集是 capability × source × layer 的点

capabilityDimension（X 轴，12 个能力）× source（Y 轴，6+2 类）× failureLayer（Z 轴，L0–L8）

每条 case 落到立方体的一个点：{ capability, source, failureLayer }。每个能力桶内独立维护完整的 6+2 类信号（不共用），按能力分桶 calibration、按能力发起优化决策、按能力定向检测规则。

Canonical Agent Harness — 9 层抽象

收敛自 Claude Agent SDK / LangGraph / OpenAI Assistants v2 / Agentic RAG 四套主参考架构。每一层都有"可独立替换 / 测 / 归因"的特征。

层含义典型自变量
L0Model Substrate 模型底座换模型 / 调温度 / 路由策略
L1Input Understanding 意图解析改 intent schema / validator
L2Planning 任务分解改 plan prompt / plan-reflect
L3Memory & Context 记忆与上下文改 summarization / truncation
L4Retrieval 检索改 chunk / rerank / query rewrite

17.5 泛化验证机制（M1–M5）⚠️ Phase 2（P4+）规划，MVP 不实施

当前 offline_eval 只能证明"在见过的 badcase 上改善了"，无法证明能力本身泛化。M1–M5 机制从根本上解决这个问题，确保修复的改善具有举一反三的泛化性。本节为 Phase 2 规划，相关字段与表已在 schema 中预留，MVP（P1–P3）阶段不激活。

机制核心思路关键判定条件阶段
M1 Seen / Holdout 分离 每个 capability 桶 30% case 标 holdout=true，修复者不可见，只在最终 gate 时读取 gen_coeff = holdout改善率 / seen改善率，≥ 0.8 通过，< 0.5 拒绝 P4
M2 最小样本量门禁 每个 capability 桶 N < 10 时不产出 pass/fail 输出 INSUFFICIENT_EVIDENCE，不允许 PM 人工覆盖 P4
M3 合成变体探针 每批 badcase 入库后自动合成 K=5 个同能力、不同场景探针，全部落入 holdout 探针 scene_diversity_score < 0.6 强制拒绝，capability_alignment_score ≥ 0.75 P4
M4 跨维度回归 Guard 修复通过 seen_set 后，追加跑全量 12 维 capability holdout 检查 任一 capability delta ≤ -0.03 → 整包 fail；防止修好 A 搞坏 B P4
M5 Paired Bootstrap CI Case-level delta 不报点估计，报 95% bootstrap CI（2000 次有放回抽样） ci_95_lower ≤ 0 → TRENDING_NOT_SIGNIFICANT，不通过门禁 P4

新表 validation_generalization_reports

每次 validation run 完成后写入，包含：per-capability gen_coeff、bootstrap CI、cross_capability_guard 结果、insufficient_evidence 列表、overall_verdict。

新字段 eval_cases.holdout 新表 validation_generalization_reports 新接口 /api/eval-datasets/synthesize-probes 新接口 /api/validation-runs/[id]/generalization

17.6 LLM 默认模型与 thinking 关闭

默认模型 Qwen/Qwen3.5-27B 是思考型模型，必须显式同时关闭以下两个开关，否则模型只回 reasoning_content，content 为空，导致整条评估链路 JSON 解析失败。 
ZEVAL_JUDGE_ENABLE_THINKING=false
SILICONFLOW_ENABLE_THINKING=false
切换到非思考模型时仍要保留两个开关为 false，避免误用思考模型。

17.7 评估管线架构说明与信号通道可行性

本节汇总 v2.1 可行性审查（v2.1补充-评估器分歧与8类信号可行性）的核心结论，防止设计文档与实现产生认知偏差。

17.7.1 管线执行顺序（Enrich → Objective → Subjective）

当前管线是顺序串联，三个阶段按序执行，不是"主观 = 客观加权派生"：

01
Enrich
字段补全 · 主题切分
规则信号提取
→
02
Objective
客观指标计算
（依赖 Enrich 输出）
→
03
Subjective
LLM Judge 评审
（依赖 Enrich 输出，
不依赖客观指标分值）

主观阶段（LLM Judge）与客观阶段共享 Enrich 信号，但主观分数不从客观分值二次推导，两者并列而非派生。

17.7.2 Fused 管线 vs. 双轨并行

当前实现是 fused（融合） 模式：LLM judge 优先，LLM 失败时规则兜底。这个结构无法自动产生"评估器分歧"信号：

模式结构能否产生 auto_disagreement
Fused（当前） LLM 优先 → 规则兜底，单一结果输出 不能 规则仅在 LLM 失败时运行
双轨并行（未来） 规则链与 LLM judge 独立运行，事后对比结论 可以 结论不一致 → auto_disagreement

结论：auto_disagreement 在 MVP 阶段为校准专用通道，由校准员手动录入；自动产生需 Phase 2 引入双轨并行架构。

17.7.3 Badcase 采集逻辑

Badcase 入池不依赖单一断言，而是多信号启发式组合判断（src/eval-datasets/harvest-badcases.ts）：

  • 负向关键词命中（客户流失意向、投诉升级等）
  • 高严重度隐式信号（兴趣下降、服务拒绝等）
  • 目标失败标签 / 恢复失败标签

三类信号任一触发均可入池（OR 逻辑），不需要全部命中。

17.7.4 瘦日志下 8 类信号通道可行性

瘦日志定义：只有 Q&A 对或 final_message，没有工具调用链、memory 命中记录、reasoning block。这在 MVP 典型场景（对话智能体评测）中很常见。

source 通道瘦日志可行性说明
auto_tp✓ 可用负向关键词 + 文本级隐式信号足够触发
manual_fp✓ 不受限人工标注，不依赖日志结构
auto_fn⚠️ 弱化退化为 FN-proxy（文本负向尾、重复提问、长度骤降），工具调用漏报不可见
auto_tn⚠️ 需抽样金标正例需通过随机抽样或高分阈值筛选，无法全自动
auto_uncertainty✓ 可用Judge confidence 区间判断，不依赖日志结构
auto_disagreement⚠️ 校准专用需双轨并行架构，MVP 阶段手动录入（见 17.7.2）
synthesized✓ 可用LLM 合成，不依赖真实日志
imported✓ 可用批量导入，格式转换即可

瘦日志下 failure_layer 默认约定：L4–L6（检索/工具选择/工具执行）通常无法从文本日志判定，failure_layer 字段默认置 unknown，若确认是最终输出问题可标 L8。Phase 2 启用 harness 层归因后再补全。

完整分析见 v2.1补充-评估器分歧与8类信号可行性.html；瘦日志 source 操作定义见 13-案例池自动入池规则.md §13.11。
