1. 评估流水线顺序与数据依赖

参考实现中，一次评估在 enrich 之后依次计算客观指标与主观指标（主观步骤内部再构造隐式信号、goal completion、recovery 等）。主观侧的隐式信号来自 enrich 后的会话行，而不是把上一阶段输出的 ObjectiveMetrics 聚合数值当作输入再「算出」主观分数。

业务场景模板（scenario KPI）会在更高一层把 objective 与 subjective 的 resolved 数值按权重合成；这与「主观四维完全由客观加权得到」不是同一条链路。更准确的说法是：共用解析结果与会话结构信号，再在 scenario 层按需加权组合。

2. 为何当前代码几乎不产生「规则 vs LLM」分歧事件

2.1 主观四维 Judge

LLM 审稿输入中会附带同一套隐式信号摘要；规则降级路径（buildRuleBasedDimensions）与 LLM 共享同源线索。实现上是「LLM 优先，缺字段再回落规则」，不会并行保留两套完整评分再比对，因此没有内置的「规则说坏、LLM 说好」检测器。

2.2 Goal completion

规则路径仅在结果为 unclear 时才会调用 LLM；若规则已给出 achieved / failed / partial，则不会再跑 LLM Judge。于是也不存在「同一 session 上规则终局 vs LLM 终局」的官方对峙分支。

关于 auto_disagreement：在数据集类型定义里，该枚举值的注释语义为「规则裁决与 LLM 裁决不一致」。但在参考实现的 bad case 落库路径中，默认写入的是 auto_tp，工程内未发现根据「双评估器不一致」自动赋值 auto_disagreement 的逻辑。它更接近数据契约中为校准预留的通道，而非当前 fused pipeline 的自然产物。

3. bad case 实际更接近哪种触发逻辑

话题级 harvesting 大量使用：用户话术中的负向关键词、响应间隔与 topic 切换等启发式、以及severity 为 high 的隐式信号（与主文档中「隐式 / 规则命中」的直觉一致）。后续再与 goal / recovery、重复提问等标签合并形成 bad case 资产。

这与「只有隐式负向关键词才会断言」不完全等同：实现里是多路信号叠加，关键词与 high severity 隐式信号均为常见触发源。

4. 8 类 source 与混淆矩阵：在重构中是否可行

结论：作为产品与数据模型设计，可行；作为「现有单次评估自动推导」，不完整。

维度 说明
混淆矩阵 需要明确定义「系统判好坏」与「参考标准」（金标、人工裁定、影子裁判等）。没有参考轴则 TP/TN/FP/FN 无法操作化。
auto_disagreement 若要落地，需要双轨评估或离线校准流程（例如 shadow rule judge、扩展指标中同时保留 rule 与 LLM 两套结果并定义冲突阈值、或由人在校准批次上给出锚标签）。不是单靠「先客观后主观」就会自动涌现。
8 类通道 × 能力维度 工程上可映射为「能力维度 × source」索引；需配套入池规则、候审状态（如 pending_review）与 golden 集对齐流程。类型层已在参考仓库的数据契约中出现同类枚举，与重构文档方向一致。

5. 与主文档表述如何对齐（建议）

  • 将 auto_disagreement 写明为：校准与评测可靠性专用通道，依赖后续实现的「双评估器不一致」检测或人工裁定，而非当前 fused pipeline 的默认输出。
  • 在描述「主客观关系」时，避免写成「主观分数完全由上游客观加权计算」；改为强调 enrich / 隐式信号 / scenario 加权合成等分层关系，以免读者推导出不存在的「单一总分对峙」。
  • 若线上仅能采集问答对 / final_message 级日志，请在 PRD 中单列瘦日志模式下各 source 的可操作定义；详见本文 §6。

6. 瘦日志（仅问答对 / final_message）下的 8 类 source

典型瘦日志仅包含用户话术与助手最终可见回复（如 final_message），缺少工具调用链、记忆命中、执行反馈、推理块等结构化字段。下面的结论与主文档 v2.1 的「6+2」通道兼容：保留枚举作为案例血缘与入池策略，同时对可操作含义做降级说明。

6.1 总体判断

  • 8 类 source 仍适合作为数据契约：表达案例如何进入池子（自动真坏、人工纠偏、漏报候选、金标正例、边界、双评估器不一致、合成、导入），与是否有完整 OTEL 式追踪正交。
  • 受瘦日志约束的主要是「漏报定义」与「能力 / 层归因」：auto_fn 只能覆盖「文本与行为代理能看到的漏报」；tool_calling_correctness 等与 trace 强相关的维度无法单靠 final_message 客观核验。

6.2 按通道的可操作性（瘦日志）

source 瘦日志下 备注
auto_tp 可用 依赖文本启发式、Judge、goal/recovery 类可从 transcript 推断的标签；「真坏」强度随证据厚度变化。
manual_fp 不受限 人工复审与血缘标注，与观测栈无关。
auto_fn 弱化 仅能捕捉文本可见的「系统判 OK 但仍异常」（重复问、负向尾词、异常收尾等）。工具静默失败、检索错但未写入回复等 → 不可见；建议在 PRD 中单列为「FN-proxy」语义。
auto_tn 可用需抽检 可按表层规则 + Judge 高分抽样正例；「金标」置信度低于全链路可观测场景，宜提高人工抽检或标记为弱 TN。
auto_uncertainty 可用 依赖 Judge 输出 confidence，与 tool trace 无关。
auto_disagreement 可用（若双轨） 需并行保留规则轨与 LLM 轨再比对；与是否有推理块无关。
synthesized 不受限 合成侧可自行携带结构化字段；统计上与 baseline 隔离的主文档口径不变。
imported 不受限 外部导入；字段完整度由来源决定。

实践顺序建议：先用 transcript + Judge 跑通 auto_tp、auto_uncertainty、manual_fp / imported / synthesized；将 auto_fn 限制为「文本可证伪」条件；auto_tn 配抽检；auto_disagreement 待双轨评估接线后再放量。

文档层面建议：在 Zeval 重构方案/重构方案梳理/13-案例池自动入池规则.md 增加小节「瘦日志模式下的 source 操作定义」，明确 auto_fn 的 proxy 边界，以及 failureLayer 在缺少 trace 时的默认值（如 unknown 或倾向 L8）。
