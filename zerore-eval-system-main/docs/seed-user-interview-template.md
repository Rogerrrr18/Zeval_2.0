# Zeval 种子用户访谈指南

> 时长：60 分钟（弹性 ±10 分钟）
> 形式：腾讯会议 / 飞书 / 线下，建议录屏 + 实时记录
> 输出：访谈纪要 + 假设验证表 + 行动项

---

## 0. 访谈前准备（30 分钟）

### 0.1 选人标准（同时满足 ≥ 3 条）
- 团队正在做面向终端的 AI 应用（客服 / 座舱 / RAG / Agent / 内容生成等）
- 有真实生产对话日志（≥ 1 万 turns / 月）
- 团队里有人正在「手动看 case / 写 prompt 调优」
- 已用过或考虑用过：LangSmith / Langfuse / Confident AI / DeepEval / Braintrust 任一
- 决策角色：AI 负责人 / 算法 / 产品负责人 / CTO

### 0.2 资料预读
- 受访者公司主营业务、最近的产品/技术博客
- 他们在用的 LLM 厂商、是否有自研模型
- 公开渠道能搜到的他们的 AI 产品演示

### 0.3 现场环境
- 浏览器打开 Zeval（最好已有跑过的 demo run）
- 准备好 1 份**他们行业**的 mock 数据集（提前 30 分钟塞进 `mock-chatlog/raw-data/`）
- 调料包样例 1 份（让他们看 Skill 文件夹）

---

## 1. 开场（5 分钟）

> "今天主要是想了解一下你们团队当前怎么衡量 AI 应用的质量、目前最痛的地方是什么。
> 我会演示一下 Zeval，但前 20 分钟更想听你们的现状，避免我先入为主。
> 全程不会推销，访谈结束后我们整理出几条你们最关心的事情，再决定要不要做下一步。"

**自报家门 → 介绍 Zeval 一句话定位**：

> "Zeval 想做的是：给 AI 应用团队一套**自动化的评测 + 调优闭环**。
> 上传日志 → 自动判定 good/bad case → 生成 agent 可读的调优包 → 跑回放验证 —— 整套不需要人工标注。"

---

## 2. 现状摸底（15 分钟）

### 2.1 团队结构 & AI 应用形态
- 你们目前在做的 AI 产品大概是什么样？终端用户是谁？
- 多少人在做？算法 / 产品 / 工程的比例大概？
- 上线多久？日活 / 日对话量大概什么量级？

> **在测的假设**：① 是否在我们的目标客户画像内（B 端 + 真实流量 ≥ 1 万 turns/天）；② 团队结构能否消化我们的产品（至少要有 1 个能写 prompt 的工程师）

### 2.2 当前怎么知道"AI 跑得好不好"
- 你们现在怎么判断这周/这版的 AI 比上一版好？
- 有没有定期人工抽检？谁做、多久一次、抽多少？
- **关键问题**：上次发现一个明显问题，是怎么发现的？是用户投诉来的、内部抽检来的、还是有指标告警？

> **在测的假设**：用户对"评测"的心智强度。如果他们说"基本靠用户投诉"，说明评测刚需未被启动；如果说"我们每周看 100 个 case"，说明已有评测习惯 → 我们能加速。

### 2.3 现在用的工具
- 你们试过哪些评测平台？为什么没继续用 / 用得不爽？
- 内部有没有自己写的脚本 / dashboard？
- 算 prompt 改了之后的效果，怎么算的？人肉对比还是有自动化？

> **在测的假设**：① 竞品认知度；② 已有工具的具体痛点（拿来对比我们的差异化）；③ 是否会被"再多接一个工具"的成本劝退

---

## 3. 痛点深挖（10 分钟）

按对方刚才提到的最痛的事情，挑 1-2 件深挖。常见钩子：

### 3.1 "case 太多看不过来" 钩
- 上次抽了 100 个 case，多久看完的？看完后做了什么？
- 看完之后能不能给到工程团队一个具体的"改这里"的建议？还是停留在"感觉某些场景不太行"？

### 3.2 "上线后才发现退步" 钩
- 你们有 baseline 概念吗？版本之间是怎么比的？
- 出过让你印象深的"线上比测试集差很多"的情况吗？

### 3.3 "长尾覆盖不到" 钩
- 测试集是怎么来的？多久更新一次？
- 用户实际行为里，多少比例的 case 是你们测试集没有覆盖到的？怎么知道的？

### 3.4 "工具调用错" 钩（agent 类客户专用）
- agent 调错工具 / 参数填错的频率？怎么发现的？
- 当前怎么判定一次工具调用算成功 / 失败？

> **在测的假设**：哪条钩拿到最强反应，就是我们 GTM 时主推的那条。

---

## 4. 产品演示（15 分钟）

### 4.1 演示原则
- **3 分钟之内必须出第一个结果**，否则收尾换演示
- 用提前准备好的他们行业的数据，不要用通用 demo
- **每展示一屏，停下来问一句**："你看到这屏第一反应是什么？"

### 4.2 演示路径（按重要性排）

**① Chat 入口（3 分钟）**
- 在 `/chat` 里说："帮我看下这批日志，告诉我哪里需要优化"
- 让他们看 plan → tool_call → final 的流式过程
- **关键观察**：他们会不会主动问"它怎么知道这是 bad case 的？"
  - 如果问 → 进入 4.3 隐式信号层介绍
  - 如果不问 → 跳过技术细节，直接看结论

**② 隐式信号 + 自动标注（4 分钟）**
- 进案例池，展示"为什么这条被判为 bad"的信号清单
- 强调：**不需要人工标注**，纯规则 + 客观指标 + 关键词
- **关键问题**："如果让你看 100 条 case，你会希望系统帮你做哪些预处理后再给你看？"

**③ 调优包 / Skill 文件夹（4 分钟）**
- 打开生成的 Skill 文件夹，给他们看 SKILL.md
- 演示一下"喂给 Claude Code / Codex 让它自动改 prompt"的流程
- **关键问题**："你们现在改 prompt 是工程改还是产品改？这种格式他们会接受吗？"

**④ 基线对比（4 分钟）**
- 展示 online-eval 里基线对照
- **关键问题**："如果上线前能跑一遍这个，你愿意花多少时间设置？"

### 4.3 不要演示的东西
- ❌ trace 观测（已决策删除）
- ❌ 样本合成功能（项目制对接，不在标准产品里）
- ❌ SDK / CLI 细节（除非他们主动问）

---

## 5. 关键决策点验证（8 分钟）

直接抛出我们正在做的关键产品决策，让他们投票：

### 5.1 case 切分粒度
> "我们正在把案例切分从 session 粒度改为 topic 粒度。比如同一个对话，前半段用户满意、后半段不满意，会被切成 2 个 case。
> 你们更倾向哪种？为什么？"

> **在测的假设**：topic 切分是否真的更符合用户直觉

### 5.2 自动标注 vs 人工标注
> "我们决定**完全不要人工标注**，全用规则 + 关键词 + 客观指标自动判定。
> 你能接受吗？哪些场景你担心规则会判错？"

> **在测的假设**：纯自动标注的接受度

### 5.3 调优包形态
> "我们交付的是一个 skill 文件夹（SKILL.md + reference 目录），你们直接喂给 Claude Code / Codex 让 agent 改代码。
> 这种交付形式你们工程团队会用吗？还是你们更想要一份 PDF / Confluence 文档？"

> **在测的假设**：Claude Code / Codex 心智渗透率

### 5.4 双模式
> "我们考虑做两种界面：**产品模式**（PM/CEO 看，只有几个核心数字 + 一句话洞察）和 **工程模式**（算法看，全量参数）。
> 你们团队是哪一种为主？两种你都要吗？"

> **在测的假设**：是否值得做双模式

---

## 6. 商业信号（5 分钟）

> "假设这个产品成熟了，覆盖了你们刚才说的所有痛点 ——"

### 6.1 价值锚定
- 你们目前为这件事**付出的成本**大概是什么？（人 × 时间 / 月）
- 如果这个产品能省下其中 50%，对你们值多少钱？

### 6.2 决策路径
- 如果你们要采购，谁拍板？需要走什么流程？
- 一般 SaaS 工具的预算是放在哪个 BU / 哪个老板？

### 6.3 试用意愿
- 如果我们提供 1 个月免费试点（含数据接入支持），你们能投入多少时间一起跑？
- 数据敏感度多高？私有化部署是必须吗？

### 6.4 不会买的原因
- 如果最后没买，最可能的原因是什么？

> **在测的假设**：① 是否能做成付费客户；② 私有化部署是否是绕不开的；③ 决策链长度

---

## 7. 收尾（5 分钟）

### 7.1 让他们给我们排序
> "回顾一下今天看到的几个能力：
> ① Chat 自动跑评估 + 给结论
> ② 自动 bad case 切分（不需人工标注）
> ③ Skill 文件夹形式的调优包
> ④ 基线回放对比
> ⑤ 隐式信号推断层
> 
> 如果只能保留 2 个，你保留哪 2 个？为什么？"

> **在测的假设**：哪些能力是核心，哪些是 nice-to-have

### 7.2 转介绍
- 你身边还有谁也在解决类似问题？方便引荐吗？

### 7.3 后续承诺
- 我们下周整理一份会议纪要 + 我们的 roadmap，发给你
- 如果你们愿意试点，我们可以帮你接入 SDK，1 周内出第一份评估报告

## 附录：常见反对意见 & 应对话术

| 反对意见 | 应对（不要驳，先认） |
|---|---|
| "我们已经在用 LangSmith / Langfuse 了" | "他们偏 trace 观测，我们偏评测结论 + 调优闭环。能不能并存看看？我们 1 周可以接入" |
| "我们数据不能出公司" | "支持私有化（Docker / on-prem），评估全部本地跑，可以演示一下部署文档" |
| "case 自动标注不准吧" | "我们提供「人工覆盖」按钮，你可以随时翻案。我们目标是让你只看 5%，不是 0 看" |
| "我们没专人做评测" | "正好，我们的 Chat 入口就是给非工程师用的，PM 也能看懂结论" |
| "现在没预算" | "试点免费一个月，跑完拿到 ROI 数据再决定要不要走采购" |
