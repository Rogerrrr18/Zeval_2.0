# 1-评测集构建

## 1. 文档目的

本文定义 Zeval 下一阶段新增能力：`评测集构建`。

目标是把当前“对 chatlog 做评估”的能力，升级为“自动抽取案例、沉淀评测集、为产品迭代建立 baseline，并持续追踪优化效果”的闭环能力。

本文重点回答 5 个问题：

1. 为什么要构建评测集
2. 什么样的 topic 片段应该进入评测集
3. 如何给 topic 片段打分并划入 `goodcase` / `badcase`
4. 后续如何用评测集验证产品迭代效果
5. 什么才算真正的优化成功（success）

## 2. 需求背景

当前项目已经具备：

- 对原始 chatlog 按 `session / topic segment` 做切分
- 对 topic 片段计算客观指标、主观指标、情绪分、隐式信号
- 输出图表与优化建议

但目前结果仍以“单次分析”为主，缺少一层更稳定的产品验证机制：

- 不能自动沉淀高质量/低质量案例
- 不能形成长期复用的评测基线
- 不能用统一标准验证产品优化是否真实有效

因此，团队新增决策是：

1. 构建 `评测集案例抽取功能`
2. 定义 `topic 片段评分机制`
3. 自动形成 `goodcase` 与 `badcase` 评测集
4. 以“加入评测集时的得分”作为 `baseline`
5. 设计一套基于评测集的跑分方案，用于验证交互产品优化效果

## 3. 新决策分析

### 3.1 这次新增的不是“一个新指标”，而是一层新能力

本次新增能力的本质是：

从“分析工具”升级为“评测基础设施”。

原来系统做的是：

- 对一批对话做评估
- 输出指标和建议

新增后系统还要做：

- 从评估结果中抽取可复用案例
- 为案例建立稳定分数与风险标签
- 将案例沉淀为评测集
- 在后续产品版本中重复使用这些案例做回归验证

### 3.2 topic 片段是最合适的评测单元

评测单元不建议直接用整段 `session`，而应采用 `topic segment`。

原因：

- 一个 session 往往包含多个主题
- 不同主题的目标和情绪性质不同
- 整段 session 过大，不利于准确定位问题
- topic segment 更适合作为“可复用案例”

结论：

- `topic segment` 是评测集构建的最小评测单元

### 3.3 success 不能只看总分提升

团队当前的判断是正确的：

优化效果不应只定义为“新版本分数更高”。

对于 `badcase`，更重要的 success 信号包括：

- 是否更早识别到风险
- 是否降低兴趣下降风险
- 是否降低理解障碍风险
- 是否降低说教感
- 是否让情绪恢复更快

因此，success 需要是多维定义，而不是单一总分。

## 4. 需求目标

本阶段目标：

1. 为每个 `topic segment` 计算一个 `0-100` 的案例分
2. 基于阈值自动划入：
   - `goodcase` 评测集
   - `badcase` 评测集
3. 将该片段加入评测集时的分数保存为 `baseline`
4. 在后续产品版本中，用同一批评测集做统一跑分
5. 输出：
   - 总体优化分
   - goodcase 保持率
   - badcase 修复率
   - 风险发现提前量
   - 风险下降情况

## 5. 核心对象定义

### 5.1 评测单元

`CaseUnit = topic segment`

每个评测单元至少包含：

- `caseId`
- `sessionId`
- `topicSegmentId`
- `topicLabel`
- `topicSummary`
- `segmentTranscript`
- `objectiveMetricsSnapshot`
- `subjectiveMetricsSnapshot`
- `signalSnapshot`
- `segmentScore`
- `baselineVersion`
- `caseSetType`

### 5.2 评测集类型

#### `goodcase`

高质量案例，代表当前产品做得好的片段，作用是：

- 检查新版本是否破坏已有优势
- 识别回归问题
- 沉淀最佳实践模式

#### `badcase`

低质量案例，代表当前产品表现差的片段，作用是：

- 检查新版本是否真实修复问题
- 识别风险是否更早被发现
- 验证是否降低负面信号

## 6. topic 片段评分机制

## 6.1 设计原则

topic 片段评分必须满足：

- 分数区间统一为 `0-100`
- 同时融合客观指标与主观指标
- 风险信号要直接影响分数
- 结构尽量稳定，可持续复用
- 便于 PM 和算法团队解释

### 6.2 总体结构

建议将案例总分拆为 3 个子分：

1. `objectiveScore`（客观质量分）
2. `subjectiveScore`（主观体验分）
3. `riskPenaltyScore`（风险惩罚分）

最终公式建议：

```text
caseScore =
  0.35 * objectiveScore
  + 0.45 * subjectiveScore
  + 0.20 * riskAdjustedScore
```

其中：

```text
riskAdjustedScore = 100 - riskPenaltyScore
```

说明：

- 主观体验分权重最高，因为当前项目的核心价值在“情绪体验 + 对话质量”
- 客观分负责提供结构稳定性
- 风险惩罚负责防止“总分高但有严重风险”的片段混入 goodcase

## 6.3 客观质量分 objectiveScore

建议第一阶段由以下 3 类指标组成：

### 6.3.1 对话轮数分 turnDepthScore

目标：避免过短的片段被误判为高质量。

建议公式：

```text
if turnCount <= 2: turnDepthScore = 20
else if turnCount <= 4: turnDepthScore = 45
else if turnCount <= 7: turnDepthScore = 70
else: turnDepthScore = 90
```

### 6.3.2 对话长度分 contentLengthScore

目标：避免“极短回复片段”被高估，同时避免无效冗长。

建议使用用户与 AI 平均长度共同计算：

```text
contentLengthScore =
  clamp(
    50
    + userLengthBonus
    + assistantLengthBonus
    - excessiveLengthPenalty,
    0,
    100
  )
```

第一阶段规则建议：

- 用户平均长度适中：加分
- AI 平均长度适中：加分
- AI 过长：扣分
- 双方都极短：扣分

### 6.3.3 主题稳定分 topicStabilityScore

目标：判断该片段内是否过于跳题。

建议公式：

```text
topicStabilityScore =
  clamp(100 - topicSwitchCount * 25, 0, 100)
```

### 6.3.4 objectiveScore 聚合

```text
objectiveScore =
  0.35 * turnDepthScore
  + 0.35 * contentLengthScore
  + 0.30 * topicStabilityScore
```

## 6.4 主观体验分 subjectiveScore

建议由以下 4 项构成：

- `emotionSegmentScore`（片段情绪得分）
- `empathyScore`（共情得分）
- `offTopicScore`（答非所问得分）
- `preachinessScore`（非说教得分）

### 6.4.1 情绪得分 emotionSegmentScore

直接使用当前项目已有的 `emotionScore`：

```text
emotionSegmentScore = emotionScore
```

说明：

- 消极情绪天然会拉低片段总分
- 负向但被成功修复的片段，仍可通过恢复能力获得一定补偿

### 6.4.2 共情得分 empathyScoreNormalized

当前系统的 `empathyScore` 为 `1-5`，统一映射到 `0-100`：

```text
empathyScoreNormalized = empathyScore * 20
```

### 6.4.3 答非所问得分 offTopicScoreNormalized

当前系统的 `offTopicOrIgnoreRisk` 本质上是“越高越好”的质量分，同样做映射：

```text
offTopicScoreNormalized = offTopicOrIgnoreRisk * 20
```

### 6.4.4 非说教得分 preachinessScoreNormalized

当前系统的 `preachinessRisk` 实际是“说教感控制得越好分越高”，统一映射：

```text
preachinessScoreNormalized = preachinessRisk * 20
```

### 6.4.5 subjectiveScore 聚合

```text
subjectiveScore =
  0.40 * emotionSegmentScore
  + 0.25 * empathyScoreNormalized
  + 0.20 * offTopicScoreNormalized
  + 0.15 * preachinessScoreNormalized
```

## 6.5 风险惩罚分 riskPenaltyScore

建议使用隐式信号层直接形成风险惩罚。

输入信号：

- `interestDeclineRisk`
- `understandingBarrierRisk`
- `emotionRecoveryFailureRisk`

建议公式：

```text
riskPenaltyScore =
  100 * (
    0.35 * interestDeclineRisk
    + 0.30 * understandingBarrierRisk
    + 0.35 * emotionRecoveryFailureRisk
  )
```

然后：

```text
riskAdjustedScore = 100 - riskPenaltyScore
```

### 6.5.1 一票否决规则

建议增加以下规则，避免明显危险片段混入 `goodcase`：

```text
if emotionRecoveryFailureRisk >= 0.8:
  不允许进入 goodcase

if understandingBarrierRisk >= 0.8:
  不允许进入 goodcase

if emotionScore <= 35:
  不允许进入 goodcase
```

## 7. goodcase / badcase 自动入库规则

### 7.1 阈值规则

建议第一阶段采用简单阈值：

```text
if caseScore >= 75:
  进入 goodcase 候选集

if caseScore <= 45:
  进入 badcase 候选集

if 45 < caseScore < 75:
  不自动入库，保留为普通样本
```

### 7.2 入库补充约束

为了避免评测集偏斜，建议增加如下约束：

- 每个主题类型都要有样本
- 每个风险类型都要有覆盖
- goodcase 与 badcase 都要保留不同长度区间样本
- 同质片段去重，避免评测集过于重复

### 7.2.1 重复过滤机制

评测集必须具备重复过滤能力，否则会出现：

- 同一段对话被重复导入
- 高相似片段大量挤占评测集
- 跑分结果被少数模式放大

因此，新增案例导入时需要做两层去重：

#### 第一层：强唯一约束

若以下字段完全一致，则视为同一案例，禁止重复入库：

```text
sourceFile
+ sessionId
+ topicSegmentId
+ normalizedTranscriptHash
```

其中：

- `normalizedTranscriptHash` 表示对 topic 片段文本做标准化后的哈希
- 标准化建议包括：
  - 去除连续空白
  - 统一标点与大小写
  - 去除无意义系统包装字段

#### 第二层：高相似去重

即使 `topicSegmentId` 不同，也可能是语义高度重复的案例，因此建议增加近似去重：

```text
duplicateScore =
  0.50 * transcriptSimilarity
  + 0.20 * topicLabelSimilarity
  + 0.15 * metricSimilarity
  + 0.15 * signalSimilarity
```

规则建议：

```text
if duplicateScore >= 0.92:
  判定为重复案例，不入库

if 0.82 <= duplicateScore < 0.92:
  标记为疑似重复，进入人工确认
```

说明：

- `transcriptSimilarity`：片段文本相似度
- `topicLabelSimilarity`：主题标签是否一致
- `metricSimilarity`：关键指标是否非常接近
- `signalSimilarity`：隐式信号结构是否高度一致

#### 第三层：评测集配额约束

即使不重复，也要避免同一模式过多：

- 单一主题类型不能超过评测集总量的某个比例
- 同一风险类型不能过度集中
- 相似模板对话应限制最大入库数量

结论：

- 新增案例不能只靠 `caseId` 去重
- 必须同时做“文本哈希去重 + 高相似去重 + 配额控制”

### 7.3 baseline 定义

每个案例首次入评测集时，要把以下内容固定下来：

- `baselineCaseScore`
- `baselineObjectiveScore`
- `baselineSubjectiveScore`
- `baselineRiskPenaltyScore`
- `baselineSignals`
- `baselineGeneratedAt`
- `baselineProductVersion`

说明：

- baseline 不应随系统重新跑分自动覆盖
- baseline 是后续所有“效果提升”计算的基线

## 7.4 评测集目录结构与存储方案

当前阶段不建议直接上 MySQL 这类需要部署和配置的数据库。

推荐先采用：

- `文件系统目录`
- `JSON` 作为案例主存储
- `JSONL` 作为批量跑分结果存储
- `CSV` 作为 PM/运营视角的导出总表

### 7.4.1 推荐目录结构

```text
eval-datasets/
├─ README.md
├─ schema/
│  ├─ case.schema.json
│  ├─ baseline.schema.json
│  ├─ run-result.schema.json
│  └─ sample-batch.schema.json
├─ goodcase/
│  ├─ manifest.json
│  └─ cases/
│     └─ .gitkeep
├─ badcase/
│  ├─ manifest.json
│  └─ cases/
│     └─ .gitkeep
├─ runs/
│  └─ .gitkeep
├─ samples/
│  └─ .gitkeep
└─ indexes/
   ├─ cases.csv
   └─ latest-baselines.csv
```

### 7.4.2 为什么这样设计

`JSON` 适合存储：

- transcript
- baseline 快照
- signal 快照
- 多维指标
- 标签与版本信息

`JSONL` 适合存储：

- 一次批量跑分中的逐案例结果
- 便于追加写入
- 便于后续分析和导出

`CSV` 适合存储：

- 供 PM / 运营筛选查看的总索引

## 7.5 热插拔存储设计

虽然当前阶段采用文件系统，但后续大概率会迁移到数据库，因此必须从一开始就把存储做成热插拔。

### 7.5.1 设计原则

业务层不能直接依赖“本地文件路径”。

统一抽象为 `DatasetStore`：

```text
DatasetStore
├─ createCase()
├─ updateCase()
├─ getCaseById()
├─ searchCases()
├─ checkDuplicate()
├─ saveRunResult()
├─ createSampleBatch()
└─ listSampleBatches()
```

### 7.5.2 第一阶段适配器

第一阶段先实现：

- `FileSystemDatasetStore`

后续可平滑增加：

- `SQLiteDatasetStore`
- `PostgresDatasetStore`
- `RemoteApiDatasetStore`

### 7.5.3 切换原则

业务逻辑层统一依赖接口：

```text
CaseExtractor -> DatasetStore
EvaluationRunner -> DatasetStore
SampleBatchBuilder -> DatasetStore
```

而不是直接读写：

- 本地 `JSON`
- 本地 `CSV`
- 某个数据库表

这样未来切到数据库时，只需要替换适配器，不需要重写上层逻辑。

## 8. 如何用静态 topic 片段验证产品迭代效果

这是当前最关键的困惑点。

结论先说：

**不能只靠“静态旧对话片段整体重评分”来验证新版本效果。**

原因：

- 静态历史对话本身不会变化
- 如果只是重新给旧片段打分，得到的仍然是旧系统行为结果
- 这无法代表“新版本产品会怎么回复”

因此，单纯对静态历史片段重跑评估，不足以验证迭代效果。

## 8.1 三种候选方案分析

### 方案 A：固定用户 query，直接比较新旧单轮回复

做法：

- 从 badcase / goodcase 片段里抽取一个关键用户 query
- 新版本产品对同一 query 生成回复
- 将新回复与旧回复进行对比打分

优点：

- 简单
- 成本低
- 易于自动化

缺点：

- 只验证“单轮回复”
- 无法覆盖多轮演化
- 不能很好验证风险是否更早被发现

### 方案 B：前缀回放 + 下一轮关键回复比较

做法：

- 保留该 topic segment 在关键节点之前的对话前缀
- 让新版本产品在相同上下文下继续生成下一轮回复
- 将新回复与旧回复做维度比较

优点：

- 比单 query 更贴近真实场景
- 能保留上下文影响
- 能检查是否更好接住用户情绪和主题

缺点：

- 仍然主要验证“下一步响应”
- 对长期效果覆盖有限

### 方案 C：前缀回放 + 局部多轮模拟

做法：

- 给定历史前缀
- 固定一个或多个关键用户后续输入
- 让新版本产品连续生成 2-4 轮回复
- 再对这段新生成片段进行完整评估

优点：

- 更接近真实交互效果
- 可以检查风险是否更早被发现
- 可以检查情绪是否恢复得更快

缺点：

- 成本更高
- 评测设计更复杂

## 8.2 推荐方案

建议采用 **“双层评测方案”**：

### 第一层：回归稳定层

使用 `前缀回放 + 下一轮关键回复比较`

目标：

- 快速验证每次版本更新是否明显变好或变坏
- 适合作为日常回归评测

### 第二层：效果验证层

使用 `前缀回放 + 局部多轮模拟`

目标：

- 检查新版本是否真的改善了交互效果
- 特别适合 badcase 修复验证

结论：

- 不推荐只做单轮 query 对比
- 也不推荐只对静态旧片段重评分
- 推荐做“前缀回放”的半静态评测

## 8.3 客户产品接入方式

为了让客户更新后的交互产品能够快速接入评测，推荐采用“回复函数 API 化”的方式。

### 8.3.1 接入思路

客户提供一个 HTTP API，评测系统不关心客户产品内部实现，只把它当作“回复函数”调用。

输入：

- 历史前缀
- 当前用户 query
- 可选 metadata

输出：

- 当前版本产品生成的回复
- 可选调试信息

评测系统只负责：

1. 从评测集中取样
2. 组织历史前缀
3. 调客户 API 获取新回复
4. 将新回复与 baseline 对比
5. 输出分数与 success 结果

### 8.3.2 这样做的好处

- 客户改完产品后，不需要把内部逻辑迁入评测系统
- 评测系统可以统一适配不同客户
- 后续从 mock API 切到真实客户 API 成本很低

### 8.3.3 当前阶段建议

由于现在没有真实客户，当前阶段建议在仓库中提供：

1. 一个 `mock customer api`
   - 模拟客户产品的回复接口
2. 一个 `SiliconFlow` 示例接入 API
   - 直接作为“临时客户产品”使用

这样后续联调时：

- 没有客户时，用 `SiliconFlow` 示例 API 跑通
- 有客户时，只需把调用地址替换成客户 API

## 8.4 临时评测集抽样方案

在真实测试中，不建议每次都跑完整评测集，而应支持从全量评测集中随机抽样出一份“临时评测集”。

例如：

- `50 goodcase`
- `50 badcase`

### 8.4.1 抽样目标

临时评测集必须同时满足：

- goodcase / badcase 比例可控
- 主题类型尽量均衡
- 风险类型尽量覆盖
- 不重复抽到高度相似样本

### 8.4.2 推荐抽样规则

```text
sampleBatch =
  stratifiedRandomSample(
    by = caseSetType + topicLabel + riskTag,
    target = 50 goodcase + 50 badcase
  )
```

### 8.4.3 抽样结果需要固化

每次抽样后必须把本次临时评测集固定保存，避免同一轮版本对比时样本漂移：

- `sampleBatchId`
- 抽样时间
- 抽样规则
- 抽中的 caseId 列表
- 评测目标版本

说明：

- 同一轮版本比较必须使用同一份 `sampleBatch`
- 不能边跑边重新随机，否则结果不可比

## 9. 评测集跑分方案

## 9.1 跑分输入

对每个评测案例，保留：

- 历史前缀
- 关键用户 query
- baseline 旧回复
- baseline 旧分数
- baseline 旧风险信号

新版本上线前，跑分流程如下：

1. 读取评测集案例
2. 用历史前缀回放上下文
3. 用固定用户 query 触发新版本产品回复
4. 生成新回复或局部多轮新片段
5. 对新结果重新跑评估
6. 与 baseline 对比

## 9.2 单案例跑分输出

每个案例输出以下结果：

- `baselineCaseScore`
- `currentCaseScore`
- `scoreDelta`
- `baselineSignals`
- `currentSignals`
- `signalDelta`
- `isImproved`
- `isRegressed`
- `judgeReason`

## 9.3 评测集总体输出

建议输出以下总览指标：

- `overallScoreDelta`
- `goodcaseKeepRate`
- `goodcaseRegressionRate`
- `badcaseRepairRate`
- `badcaseMajorRepairRate`
- `riskReductionRate`
- `earlyDetectionGain`

## 10. success 指标定义

### 10.1 总体 success

整体优化 success 不应只看总分提升，建议定义为复合成功：

```text
success =
  总体分提升
  + badcase 修复率达标
  + goodcase 无明显回归
  + 风险信号下降
```

### 10.2 goodcase success

对 goodcase，更关注“是否守住原有优势”。

建议指标：

- `goodcaseKeepRate >= 85%`
- `goodcaseRegressionRate <= 10%`
- 高质量案例中不得出现严重新风险

### 10.3 badcase success

对 badcase，更关注“是否修复问题”。

建议 success 指标：

1. `caseScore` 是否提升
2. `emotionScore` 是否提升
3. `understandingBarrierRisk` 是否下降
4. `interestDeclineRisk` 是否下降
5. `emotionRecoveryFailureRisk` 是否下降
6. 风险是否更早被发现
7. 回复内容是否更少说教、更贴题、更有共情

### 10.4 风险提前发现指标 earlyDetectionGain

这是 badcase 的关键新增 success 指标。

目标：不是等问题变严重才识别，而是在更早阶段发现。

建议定义：

```text
earlyDetectionGain =
  baselineRiskFirstTurn - currentRiskFirstTurn
```

解释：

- 若新版本在更早轮次就识别到风险，则该值为正
- 对 badcase 来说，这是重要提升信号

### 10.5 推荐 success 判定规则

建议第一阶段使用以下综合判定：

```text
一个 badcase 被判定为修复成功，当且仅当：

1. currentCaseScore - baselineCaseScore >= 8
2. understandingBarrierRisk / interestDeclineRisk / emotionRecoveryFailureRisk 至少下降 1 项
3. 不出现新的高风险信号
```

```text
一次版本迭代被判定为成功，当同时满足：

1. overallScoreDelta > 0
2. badcaseRepairRate >= 60%
3. goodcaseKeepRate >= 85%
4. high severity 风险总量下降
```

## 11. 功能需求清单

### 11.1 案例抽取

- 支持按 `topic segment` 自动抽取候选案例
- 支持按阈值自动归类为 `goodcase / badcase`
- 支持人工确认与去重
- 支持文本哈希去重
- 支持高相似疑似重复识别

### 11.2 baseline 存储

- 支持保存案例加入评测集时的 baseline 分数
- 支持保存 baseline 对话片段、风险信号与版本号
- baseline 不可被重复跑分自动覆盖

### 11.3 评测集管理

- 支持查看 goodcase 列表
- 支持查看 badcase 列表
- 支持按主题、风险类型、版本筛选
- 支持案例移入/移出评测集
- 支持构建临时评测集 sample batch
- 支持固定 sample batch 结果用于版本对比

### 11.4 跑分能力

- 支持按版本批量跑评测集
- 支持输出案例级对比结果
- 支持输出评测集级汇总结果
- 支持调用客户侧 API 作为回复函数
- 支持 mock customer api 与 SiliconFlow 示例 API 两种接入

### 11.5 success 报告

- 输出总体分变化
- 输出 goodcase 保持情况

## 12. MVP 实施建议

建议按两阶段落地：

### 第一阶段

先实现：

- topic segment 案例打分
- goodcase / badcase 自动归类
- baseline 固化
- 单轮或单步前缀回放跑分
- 文件系统版 DatasetStore
- 去重与 sample batch 抽样
- mock customer api 联调

### 第二阶段

再扩展：

- 局部多轮模拟
- 风险提前发现统计
- 更复杂的版本对比报告
- 人工标注与案例运营后台

## 13. 结论

本次新增需求的价值，不只是“再多一份分析报告”，而是把评估系统升级为一套可持续迭代的产品验证机制。

最终形成的闭环应为：

```text
原始对话
-> topic 切分
-> 片段评分
-> goodcase / badcase 入库
-> baseline 固化
-> 新版本回放跑分
-> success 判断
-> 驱动下一轮优化
```

对 PM 来说，这层能力的核心价值是：

- 可以持续追踪“产品是否真的变好”
- 可以知道“变好的是哪里、变差的是哪里”
- 可以让优化目标从主观感受，变成可复用、可追踪、可汇报的评测体系
