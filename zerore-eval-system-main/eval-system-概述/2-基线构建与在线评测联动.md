# 2 · 基线构建与在线评测联动

本文档说明：**首页工作台评估结果如何固化为基线**、**在线评测页如何加载基线并与新回放结果对比**，以及 **customerId + 评测时间（runId / createdAt）** 在联调中的职责边界。实现以当前 MVP 代码为准（`mock-chatlog/baselines/` + `/api/workbench-baselines` + `/online-eval` + `/api/online-eval/replay`）。

---

## 1. 目标与范围

- **基线（baseline）**：在一次「完整评估」之后，将 **原始对话 `rawRows`** 与 **当次 `evaluate` 全量结果** 绑定存档，作为后续「同一话术脚本」下的对照参照。
- **在线评测**：在 **不改变 user 话术顺序与内容** 的前提下，逐条用 **客户 HTTP 回复接口**（与 `mock-customer-api` 的 `POST /reply` 契约一致）替换 **assistant** 内容，再跑同一套 `enrich → objective → subjective → charts` 流水线。
- **对比**：在线页展示 **主观维度、客观核心、情绪曲线、隐式信号** 等多图对比（基线 vs 回放后）。

不在本文范围：生产级多租户隔离、SSRF 全面治理、评测集配额引擎（仍以 `1-评测集构建.md` 为长期参照）。

---

## 2. 首页侧：基线如何产生

### 2.1 用户操作路径

1. 在「对话评估工作台」上传 `CSV/JSON/TXT/MD`，完成 ingest，得到 `rawRows`。
2. 点击「开始评估」，调用 `POST /api/evaluate`，得到 `EvaluateResponse`（含 `runId`、`objectiveMetrics`、`subjectiveMetrics`、`charts` 等）。
3. 填写 **客户 ID（customerId）**（默认 `default`），点击 **「保存工作台基线」**。
4. 前端调用 `POST /api/workbench-baselines`，写入快照文件。

### 2.2 持久化结构

- 根目录：`mock-chatlog/baselines/`
- 目录名：`sanitizeCustomerId(customerId)`  
  - 若 customerId 仅含 `[a-zA-Z0-9_-]`，则目录名与输入一致（截断长度）。  
  - 否则使用 **短哈希目录名**（避免中文/特殊字符无法建目录或碰撞），**快照 JSON 内仍保存原始 `customerId` 字符串** 供展示。
- 文件名：`sanitizeRunIdForFile(evaluate.runId).json`
- 单文件内容（`WorkbenchBaselineSnapshot`）核心字段：
  - `schemaVersion: 1`
  - `customerId`, `runId`, `createdAt`, `label?`, `sourceFileName?`
  - `evaluate`: 完整 `EvaluateResponse`（与首页下载 JSON 一致）
  - `rawRows`: **必须与本次评估同源**，供在线回放逐条替换 assistant。

### 2.3 为何必须存 `rawRows`

在线回放逻辑依赖 **稳定的 user 侧话术时间线**。若仅存指标、不存原文，则无法保证与客户新接口在「同一输入分布」下对比，违背「前缀/同源回放」类评测意图（参见 `1-评测集构建.md` 中 baseline 与跑分对照思路）。

### 2.4 与 eval-datasets 中 baseline.json 的区别

| 机制 | 用途 |
|------|------|
| `mock-chatlog/baselines/*`（工作台快照） | **联动手台 ↔ 在线评测**；按 customerId 索引；含 rawRows + 全量 evaluate。 |
| `eval-datasets/.../baseline.json`（数据集案例） | 评测集治理、版本跑分、sample batch 等 **结构化案例库** 能力，与工作台快照互补。 |

MVP 阶段优先打通前者；后者可在后续把「工作台保存」一键同步为 `DatasetStore.saveBaseline`。

## 3. 在线评测页：如何发现基线并对照

### 3.1 发现基线

1. 用户输入 **customerId**（与保存时一致；若曾用中文名，仍输入同一字符串，服务端会映射到同一哈希目录）。
2. 点击 **「刷新基线列表」**：`GET /api/workbench-baselines/{customerId}`，返回按 **文件修改时间倒序** 的索引（含 `runId`、`createdAt`、`label`、`sourceFileName`）。
3. 下拉选择一条 **runId**（即一次评测时间维度的锚点；`createdAt` 为 ISO 时间，便于人读）。

**区分客户与时间的建议用法**

- **customerId**：租户/客户/产品线等 **稳定主体**，用于目录聚合。
- **runId + createdAt**：一次评估 **时间维度** 锚点；同一 customerId 下可保留多条历史，用于对比「上周基线 vs 本周新接口」。

### 3.2 执行在线回放

1. 可选填写 **客户回复 API 基址**（会自动拼 `/reply`；与 `mock-customer-api/siliconflow-customer-api.mjs` 对齐）。
2. 留空则读取环境变量 **`SILICONFLOW_CUSTOMER_API_URL`**，再回退 **`http://127.0.0.1:4200`**，即 **默认 SiliconFlow 模拟通道**。
3. 调用 `POST /api/online-eval/replay`，请求体带 `baselineRef: { customerId, runId }`。
4. 服务端：
   - 读取快照 → `rawRows` + `baselineEvaluate`
   - `replayAssistantRowsWithHttpApi` 逐条 HTTP 替换 assistant
   - `runEvaluatePipeline` 得到新的 `evaluate`
5. 响应同时返回 **`baselineEvaluate` + `evaluate`**，前端绘制多图对比。

### 3.3 临时评测集（约 20 条）

- 常量：`TEMP_EVAL_SAMPLE_GOODCASE_TARGET = 10`、`TEMP_EVAL_SAMPLE_BADCASE_TARGET = 10`（合计约 20）。
- `POST /api/eval-datasets/sample-batches`：**不足 20 条仍允许落盘**，并在 `warnings` 中说明（适配 mock 案例池较小时期）。
- 该能力与「工作台基线回放」并行：前者服务 **eval-datasets 案例库跑批**，后者服务 **同源话术 + 新回复函数** 的交互效果验证。

## 4. 回复 HTTP 契约（客户接入约定）

请求：

```http
POST {base}/reply
Content-Type: application/json

{
  "messages": [ { "role": "user"|"assistant"|"system", "content": "..." } ],
  "userQuery": "当前轮用户话术（与 messages 最后一轮 user 对齐）"
}
```

响应：

```json
{ "reply": "assistant 应返回的正文", "provider": "optional", "model": "optional", "latencyMs": 0 }
```

错误时返回非 2xx 或 `{ "error": "..." }`，在线链路会中断并提示（便于观测，不吞错）。

---

## 5. 风险与后续增强

- **SSRF**：服务端会 `fetch` 用户填写 URL，仅限 MVP 内网演示；生产需域名白名单、超时、鉴权与配额。
- **一致性**：若客户接口改变 userQuery 语义，会导致与基线「输入分布」偏离；应在客户侧保持与 mock 相同的契约测试。
- **大文件**：基线 JSON 含 `evaluate.enrichedRows` 时体积较大；后续可拆分为「指标快照 + 原文分文件」。
