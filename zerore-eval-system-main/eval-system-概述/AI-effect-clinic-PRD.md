1. 产品概述

1.1 产品定位

AffectBench （AI Effect Bench）是一个面向 AI 交互产品创始人的效果诊断与优化顾问。

不是平台，不是 SDK，不是认证体系。是一个让创始人一周内看到效果、愿意付费的小工具。

1.2 目标用户（访谈）

AI 陪伴产品创始人（如 Replika、Character.ai 类产品）

独立开发者（有 AI 对话产品，想优化留存）

AI 社群/孵化器成员（需要效果评估支持）

1.3 核心价值

一周内知道为什么用户流失

拿到 3 条具体到 prompt 级别的优化建议

用数据向投资人证明产品效果

1.4 MVP 目标

开发周期：1 周（两人 vibe coding）

代码量：~2000 行（前端 200 行 + 后端 800 行 + 评估逻辑 1000 行）

成本：$5/月（Vercel 免费 + Railway $5/月）

定价：$299/次（诊断版）

收入目标：3 个月内 20 个客户 = $6,000

2. 产品架构

2.1 核心流程

用户上传对话日志/sessions/tools_call/skill_usage/……（json/txt/csv） → 后端评估 → 生成报告 → 用户看结果（Web2：从埋点到指标 / Agent：从input到output）

2.2 技术架构图

用户浏览器 (Next.js) → Vercel → Railway (FastAPI) → 评估逻辑

2.3 模块划分

前端模块：上传页面 + 结果展示页面

后端模块：文件上传 API + 评估 API + 结果查询 API

评估模块：6 大维度 30+ 指标计算 + 优化建议生成

3. 功能清单（MVP）

3.1 前端功能

1. 上传区 - 拖拽上传 JSON/CSV 文件（~50 行代码）

2. 进度条 - 显示评估进度 0-100%（~30 行代码）

6. 评估逻辑（核心）

6.1 时间维度指标（4 个）

6.1.1 对话深度分布

计算方法：统计 1 轮/3 轮/5 轮+/10 轮 + 对话占比

代码示例：depths = [len(c.messages) for c in convs]

业务洞察：用户在哪一轮大量流失？找到流失高峰轮次

6.1.2 响应时间间隔

计算方法：用户两次消息之间的时间差（秒）

代码示例：gaps = [t[i+1]-t[i] for i in range(len(t)-1)]

业务洞察：用户思考时间越长，可能越困惑或犹豫

6.1.3 对话时长

计算方法：从开场到结束的总时长（分钟）

代码示例：duration = (last_msg.time - first_msg.time) / 60

业务洞察：过短（<1 分钟）= 没吸引力，过长（>30 分钟）= 效率低

6.1.4 活跃时段热力图

计算方法：按小时统计对话数量

代码示例：hours = [msg.hour for msg in messages]

业务洞察：用户什么时间最需要陪伴？优化运营时间

6.2 情绪维度指标（5 个）

6.2.1 情绪曲线

计算方法：每轮对话的情感分数连线

代码示例：scores = [sentiment(m.content) for m in messages]

业务洞察：情绪是上升还是下降？整体趋势如何

可视化：折线图（Chart.js Line Chart）

6.2.2 情绪波动幅度

计算方法：最高分 - 最低分

代码示例：volatility = max(scores) - min(scores)

业务洞察：波动太大 = 不稳定，用户体验不一致

6.2.3 情绪拐点

计算方法：情绪从正转负/从负转正的节点

代码示例：for i in range(1, len-1): if scores[i-1]<scores[i]>scores[i+1]: peak

业务洞察：哪轮对话改变了用户心情？做对/做错了什么

6.2.4 峰值情绪时刻

计算方法：情绪最高/最低的那轮对话

代码示例：peak_idx = scores.index(max(scores))

业务洞察：做对了什么（峰值）/做错了什么（谷值）

6.2.5 情绪恢复时间

计算方法：负面情绪持续到恢复中性的轮次

代码示例：for i in range(start, len): if scores[i] > 0: return i-start

业务洞察：共情能力够不够快？恢复越短越好

6.3 交互断点指标（5 个）

6.3.1 断点位置

计算方法：用户最后一条消息的轮次

代码示例：positions = [len(c.messages) for c in convs]

业务洞察：大部分用户在第几轮离开？

可视化：热力图（Chart.js Heatmap）

6.3.2 断点前情绪

计算方法：最后一轮的情绪分数

代码示例：last_emotion = sentiment(last_message)

业务洞察：是带着负面情绪离开的吗？

6.3.3 断点前 AI 行为

计算方法：最后一轮 AI 的回应类型

代码示例：last_ai_msg = messages[-2] if messages[-1].role==user else messages[-1]

业务洞察：最后一轮 AI 说了什么导致离开？

6.3.4 沉默时长

计算方法：用户最后一次回复到现在的时长

代码示例：silence = now - last_message.time

业务洞察：是暂时离开还是永久流失？

6.3.5 断点聚类

计算方法：统计所有断点位置的分布

代码示例：from collections import Counter; Counter(positions)

业务洞察：找到"死亡轮次"（如第 3 轮）

6.4 对话质量指标（6 个）

6.4.1 话题切换频率

计算方法：话题变化的次数 / 总轮次

业务洞察：切换太快 = 没深度，太慢 = 无聊

6.4.2 用户主动提问比例

计算方法：用户提问次数 / 用户消息总数

业务洞察：用户好奇 = 投入度高

6.4.3 AI 主导 vs 用户主导

计算方法：AI 发起话题数 / 用户发起话题数

业务洞察：1:1 最理想，AI 太多 = 说教

6.4.4 用户输入长度趋势

计算方法：每轮用户消息字数的变化

业务洞察：越来越短 = 失去兴趣

6.4.5 AI 回应长度

计算方法：AI 消息平均字数

业务洞察：太长 = 压迫感，太短 = 敷衍

6.4.6 共情语句密度

计算方法：包含共情词的回应比例

代码示例：empathy_words = ["我理解", "我明白", "我懂"]

业务洞察：共情能力够不够

6.5 用户参与度指标（5 个）

6.5.1 个人信息分享度

计算方法：用户分享个人信息的轮次占比

业务洞察：愿意分享 = 信任建立

6.5.2 表情 emoji 使用率

计算方法：包含表情的消息比例

业务洞察：情绪投入度

6.5.3 用户重复问题次数

计算方法：相同/类似问题出现的次数

业务洞察：AI 没听懂？用户不耐烦？

6.5.4 用户困惑表达

计算方法：什么意思、不懂等词频

业务洞察：AI 表达不清晰

6.5.5 24 小时返回率

计算方法：24 小时内再次对话的比例

业务洞察：真正的留存指标

6.6 问题诊断指标（5 个）

6.6.1 说教检测

计算方法：AI 消息包含应该、必须等词的频率

代码示例：preachy = ["应该", "必须", "你要"]

业务洞察：说教太多 = 用户反感

可视化：雷达图（Chart.js Radar）

6.6.2 无视检测

计算方法：用户提问后 AI 转移话题的比例

业务洞察：AI 没理解用户？

6.6.3 重复检测

计算方法：AI 重复相同回应的次数

业务洞察：模型多样性不够

6.6.4 最佳实践对话

计算方法：高留存对话的共同特征

业务洞察：复制成功模式

6.6.5 负面模式识别

计算方法：低留存对话的共同特征

业务洞察：避免失败模式
