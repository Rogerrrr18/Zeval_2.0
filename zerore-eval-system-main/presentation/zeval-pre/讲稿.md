# Zeval Pre 讲稿

目标听众：founder、技术朋友  
建议时长：8-10 分钟  
核心记忆点：Zeval 把真实对话里的 bad case 和 golden case，变成 AI Agent 持续进化的样本。

## 1. 封面

大家好，我今天想讲一下我们最近在做的 Zeval。

先用一句话概括：我们希望把 AI Agent 产品里的真实对话，变成下一版体验持续进化的样本。

为什么做这个？因为 Agent 产品已经过了只看 demo 的阶段。它开始进入客服、运营、内部工具、陪伴、销售这些真实场景。一旦进入真实场景，大家很快会遇到一个问题：产品上线后，质量到底怎么持续变好？

这里面有两类样本都很重要。一类是 bad case，它告诉我们哪里有风险。另一类是 golden case，它告诉我们什么样的交互值得保留、复用和继续增强。

Zeval 想解决的就是这个问题：让团队从真实对话里持续提炼产品进步的方向。

## 2. AI Agent 正在进入真实业务

过去做 AI 产品，一个 demo 效果好，大家就会很兴奋。

但 Agent 真正进入业务之后，情况会复杂很多。用户会在多轮对话里完成任务，也会表达情绪，还会不断试探这个 Agent 到底靠不靠谱。

失败也不一定是系统报错。它可能是答非所问，可能是语气太硬，可能是用户情绪没有被接住，也可能是用户慢慢不想聊了。

所以发版风险也变大了。Prompt 改一下，模型换一下，工具链调一下，某些场景可能变好，另一些场景可能悄悄变差。

## 3. Founder 的痛点

从 founder 视角看，最痛的是不确定。

团队每天可能都在看 chatlog。看完以后，大家会有很多感觉：这条不太好，那条好像有问题，这版应该比上一版强一点。

但这些感觉很难沉淀下来。用户到底在哪一轮掉了？哪个回答让体验开始变差？这类问题通常很难说清楚。

然后团队开始改 prompt。改完以后再挑几条样例看一下。如果看起来顺了，就准备发版。

这个过程太靠经验。今天踩过的坑，下次发版还可能再踩一遍。

## 4. 技术侧的痛点

技术朋友可能会更关心另一个问题：eval 到底能不能帮我们真的修产品？

很多 eval 可以给分数。比如平均分 82，或者某个维度 4.1 分。这个信息有用，但还不够。

我们更想知道的是：哪一轮触发了风险？证据是哪句话？为什么判断这里有共情问题？这个问题应该改 prompt，改 policy，改 orchestration，还是改代码？

还有一个更现实的问题：修完以后，怎么证明真的比 baseline 好？

如果没有这个证明，大家最后还是会回到凭感觉发版。

## 5. 一个真实失败形态

这页是一个简化例子。

用户说自己焦虑，最近睡不着。Agent 其实有回答，也给了建议。但它没有接住用户当下的状态。

用户继续说：“我知道，但我现在就是停不下来地想事情。”这时候 Agent 说：“你应该减少负面思考，建立积极心态。”

这句话看起来没错，但体验上会很糟。用户最后只回了一句：“算了。”

这类失败非常常见。表面上 Agent 有回答，实际上对话已经开始失控。我们要同时看回答本身、情绪恢复、用户追问和目标推进。

## 6. 我们的判断

Zeval 的核心判断很简单：AI Agent 产品需要一条体验进化闭环。

这条闭环包括五步：上传真实对话，评估质量信号，提取证据，生成优化任务，再用回放验证。

这样一来，bad case 不会只停留在复盘里，golden case 也不会只停留在“这条不错”的感觉里。它们都会进入下一次发版前的验证链路。

换句话说，真实对话本身就是很好的产品样本。关键是有没有工具把它沉淀下来。

## 7. Zeval 怎么解决

Zeval 的链路大概是这样。

第一步，接收原始对话。现在支持 CSV、JSON、TXT、MD。

第二步，做解析和标准化。系统会把原始 chatlog 变成统一结构，按 session 和 topic 做整理。

第三步，做质量评估。客观指标用规则算，主观指标由 LLM Judge 输出结构化结果，比如 score、reason、evidence、confidence。

第四步，输出图表、证据、建议、bad case 和 golden case。

第五步，高优先级问题和高质量样本会被编译成优化任务。里面有 issue brief、cases、remediation spec 和 acceptance gate。后面可以用 baseline 和 replay 判断这次改动有没有真的提升体验。

## 8. Workbench

这是我们的工作台。

它的作用很直接：让团队先把真实对话跑起来。

早期不需要复杂接入。拿一批历史 chatlog，上传进去，就能看到标准化结果、评估指标、图表和建议。

这一步对我们也很重要。因为客户不会先关心架构有多完整，他会先关心：你能不能从我的真实对话里找出有价值的问题。

## 9. Bad Case Pool

第二个核心页面是 bad case pool。

我们希望每一次失败都能留下来。

一个 bad case 里应该有证据、有风险信号，也要允许人去修正判断。后面这些 case 可以做去重、抽样、复盘，也可以放进固定回归集合。

这样产品越迭代，测试资产会越多。团队不会每次都从零开始看 chatlog。

## 10. Remediation Package

第三个关键是调优包。

发现问题只是第一步。更重要的是把问题变成可以执行的任务。

我们的调优包里会包含 issue brief、badcases、remediation spec 和 acceptance gate。

它可以交给工程师，也可以交给 Codex 或 Claude Code 这类 Coding Agent。它们可以根据这些材料去改 prompt、policy、orchestration，必要时也可以改代码。

所以 Zeval 连接的是两件事：前面是质量发现，后面是工程修复。

## 11. 当前支持的接入方式

目前我们支持四种接入方式。

第一种是文件上传。CSV、JSON、TXT、MD 都可以，适合演示、客户样例和第一批数据验证。

第二种是评估 API，`POST /api/evaluate`。适合接入内部系统，或者做发版前检查。

第三种是 trace ingest，`POST /api/traces/ingest`。它可以把线上 Agent 的运行轨迹写进系统，后面用于归档和回放。

第四种是 SDK、CLI 和 Agent package。这个方向主要是把评估结果接进工程流程，让修复任务可以继续往 Codex、Claude Code 这类工具里走。

也就是说，Zeval 可以很轻地开始，也可以逐步接到持续回归和 release gate 里。

## 12. 结尾

最后讲一下我们为什么觉得这件事有意义。

当 Agent 成为产品核心体验以后，每一段真实对话都很珍贵。

bad case 告诉我们用户在哪里卡住，哪里没有被理解，哪里任务没有完成。golden case 告诉我们什么样的追问更聪明，什么样的回应更自然，什么样的交互更值得复制。

我们正在定义 Agent 时代的质量标准。这个标准不只看分数，也看真实对话里能不能持续发现风险、提炼好案例，并把这些样本接到产品、工程和发版验证里。
