## 一、研究背景与核心痛点

### 背景

zerore 团队正在构建一套针对多轮对话 Agent 的 eval system。与传统 benchmark 预先构建数据集不同，zerore 的做法是从历史对话数据中挖掘 session 并切分为评测单元。这种基于真实历史数据的评测方式更贴近生产环境，但也引入了两个核心痛点。

### 痛点 1：固定多轮 User Query 回放导致「错位」

- **传统做法：** 固定多轮 user query → 让新 Agent 逐轮回复。
- **问题：** Agent 优化后可能提前解决问题，但 user query 仍在回放，导致后续 query 与 Agent 的上下文不匹配。例如：原始对话中用户第 3 轮才提供关键信息，但优化后的 Agent 在第 1 轮就引导用户提供了，此时第 2–3 轮的原始 query 对 Agent 来说就是噪声。

### 痛点 2：SimUser + LLM-as-a-Judge 不可复现、对话递归

- **替代方案：** 引入模拟用户（SimUser）和 LLM 裁判。
- **问题：**
  1. SimUser 在长交互中人格不一致，容易与 Agent 产生无限递归对话；
  2. LLM-as-judge 在多轮场景下评判结果不稳定，run-to-run 方差大；
  3. 整体流程不可复现，每次评测结果可能不同。

### 数据来源差异

| 对比项 | 传统 benchmark | zerore 做法 |
|--------|------------------|-------------|
| 数据 | 预先构建数据集（如 MultiWOZ、TaskMaster），让 Agent 跑分 | 从真实历史对话中挖掘 session，以 session 为评测单元 |
| 优势 | 标准化 | 覆盖真实用户意图分布和长尾场景 |
| 挑战 | — | session 内 topic 可能跳跃（从 topic2 切回 topic1），切分困难，必须以完整 session 为评测单元 |

## 三、论文详细梳理（按相关度排序）

### RAG-DIVE 动态多轮对话评测 (2604.16310)

- **相关度：** ⭐⭐⭐⭐⭐ | **发表时间：** 2026-01 | **GitHub：** 待确认  
- **核心问题：** 静态多轮数据集评测的缺陷：预定义数据集只能做单向固定查询，无法捕捉 Agent 在真实交互中的动态适应能力。  
- **解决方案：** 提出动态多轮评测方法，让对话根据 Agent 回复动态演化，而非固定回放。通过动态检索和回复生成，模拟真实多轮对话的自适应特性。  
- **关键发现：** 固定 user query 回放会限制评测捕捉 Agent 的自适应能力，导致评测结果与实际部署表现存在 gap。  
- **对 zerore 的启示：** 历史 session 不应逐字回放，而应提取关键意图 / 目标，让 SimUser 根据这些意图动态生成 query。

### 从离线日志学习主动式 LLM (2510.25441)

- **相关度：** ⭐⭐⭐⭐⭐ | **发表时间：** 2025-10 | **GitHub：** 待确认  
- **核心问题：** 当前范式要么只优化单轮指标，要么依赖脆弱且昂贵的用户模拟器，存在现实鸿沟。  
- **解决方案：** 提出 Learn-to-Ask 框架，直接从离线日志中学习主动式对话 Agent，无需用户模拟器。通过行为克隆 + 离线 RL，从真实交互中学习策略。  
- **关键发现：** 从真实历史对话中学习，避免了模拟用户的不真实性，且能学到真实的用户意图分布。  
- **对 zerore 的启示：** 可以用历史 session 训练一个轻量级的 SimUser，让它学习真实用户的提问模式，而非用 LLM 实时生成。

### 多轮对话回放评测推荐 Agent (2603.12564)

- **相关度：** ⭐⭐⭐⭐⭐ | **发表时间：** 2026-03 | **GitHub：** 待确认  
- **核心问题：** 提出评测盲区（evaluation blindness）：Agent 消费错误工具数据时推荐不当产品，但标准质量指标仍接近完美。  
- **解决方案：** 回放 23 轮金融咨询对话到 8 个 LLM，通过多轮回放暴露单轮评测看不到的行为漂移。  
- **关键发现：** 多轮回放评测能发现 Agent 在连续交互中的行为漂移，这是固定 benchmark 做不到的。但同时也发现：Agent 优化后可能改变对话路径。  
- **对 zerore 的启示：** 多轮回放本身有价值，但需要处理 Agent 改变对话路径的问题。可以引入状态检查点机制。

### 多轮场景 LLM 裁判可靠性框架 (2604.27727)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2026-04 | **GitHub：** 待确认  
- **核心问题：** LLM-as-judge 在多轮交互中的评判可靠性、跨模型可比性和可解释性不足。  
- **解决方案：** 提出 rubric 驱动的 LLM-as-judge 框架，通过细粒度评分准则和多次采样投票提升多轮场景裁判可靠性。  
- **关键发现：** 通过结构化评分准则 + 多次采样，LLM 裁判的可复现性可提升约 40%。  
- **对 zerore 的启示：** 裁判不可复现的核心原因是评分准则不够结构化。引入 rubric + 投票机制可显著改善。

### MEMO 多轮多 Agent 评测方差问题 (2603.09022)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2026-03 | **GitHub：** 待确认  
- **核心问题：** 多轮多 Agent 评测存在显著的 run-to-run 方差：早期小偏差在轮次间累积，被多 Agent 耦合放大，导致评测不可复现。  
- **解决方案：** 记忆增强的上下文优化：在每轮交互后注入记忆锚点，减少早期偏差的累积效应。  
- **关键发现：** 多 Agent 交互的方差是系统性问题，不加以控制则评测结果基本不可比。  
- **对 zerore 的启示：** SimUser + Agent 的双边交互方差大，可以固定一方（SimUser 用确定性策略）或引入记忆锚点减少累积偏差。

### MUSE 多领域用户模拟器 (2604.13828)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2026-04 | **GitHub：** 待确认  
- **核心问题：** 现有用户模拟器依赖浅层用户画像、长交互中人格不一致、局限于英语单领域。  
- **解决方案：** 自进化用户画像 + 评分准则对齐的多领域用户模拟框架。用户画像在交互中动态更新，保持人格一致性。  
- **关键发现：** 自进化画像能显著改善长交互中的人格一致性，但递归问题仍需额外控制。  
- **对 zerore 的启示：** SimUser 需要人格画像机制，且需要递归检测 + 终止策略。

### ATOD 代理式任务对话评测框架 (2601.11854)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2026-01 | **GitHub：** 待确认  
- **核心问题：** 现有 benchmark 缺乏对 agentic 行为（协调交错目标、长程上下文、主动异步执行）的系统评测支持。  
- **解决方案：** 评测单元从单轮扩展到完整 session，支持多目标、多工具、长对话场景的端到端评测。  
- **关键发现：** session 级评测能更好地反映 Agent 的真实能力，但需要新的评测指标设计。  
- **对 zerore 的启示：** 以 session 为评测单元是正确方向，需要设计 session 级别的评测指标而非 turn 级别。

### LLM 强化学习回放策略综述 (2605.02913)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2026-04 | **GitHub：** 待确认  
- **核心问题：** 综述性文章，覆盖 rollout（轨迹采样）策略的全面调研。  
- **解决方案：** 系统分类：生成（Generate）、过滤（Filter）、控制（Control）、回放（Replay）四大策略。  
- **关键发现：** 回放策略在样本效率和评测真实性之间有独特优势，但需要处理分布偏移问题。  
- **对 zerore 的启示：** 回放策略的方法论全景图，适合理解整体技术路线。

### 真实客服对话离线到在线评测 (2510.22143)

- **相关度：** ⭐⭐⭐⭐ | **发表时间：** 2025-10 | **GitHub：** 待确认  
- **核心问题：** 离线评测指标和在线实际表现之间存在 gap，需要闭环验证。  
- **解决方案：** 从真实客服对话中学习，构建 benchmark 到 optimization 的闭环。  
- **关键发现：** 离线评测的改进不一定能转化为在线性能提升，需要持续 A/B 验证。  
- **对 zerore 的启示：** 回放评测的结果需要定期用在线 A/B 测试验证，确保离线指标的有效性。

### ClawMark 多轮多日 Agent 基准 (2604.23781)

- **相关度：** ⭐⭐⭐ | **发表时间：** 2026-04 | **GitHub：** 待确认  
- **核心问题：** 现有 benchmark 无法捕捉 Agent 在多天工作流中环境独立变化的场景。  
- **解决方案：** 多轮、多日、多模态的活世界基准，环境会独立于 Agent 变化。  
- **关键发现：** 评测单元从单 session 扩展到多日，考虑环境动态变化。  
- **对 zerore 的启示：** 如果 session 跨多天，需要考虑环境状态变化的影响。

### 对抗式模拟用户评测 Agent (2605.07138)

- **相关度：** ⭐⭐⭐ | **发表时间：** 2026-05 | **GitHub：** 待确认  
- **核心问题：** 现有 benchmark 假设用户合作诚实，但真实交互中用户会 gaslight、升级情绪、施压。  
- **解决方案：** 对抗式模拟用户，测试 Agent 在极端交互下的鲁棒性。  
- **关键发现：** 模拟用户不应只模拟正常行为，还应模拟对抗性行为来测试 Agent 边界。  
- **对 zerore 的启示：** SimUser 可以设计多种模式（正常 / 对抗 / 混乱），覆盖更多测试场景。

### DuCCAE 对话系统离线评测框架 (2603.19248)

- **相关度：** ⭐⭐⭐ | **发表时间：** 2026-02 | **GitHub：** 待确认  
- **核心问题：** 生产环境对话系统面临响应速度和长程任务能力的权衡。  
- **解决方案：** 混合引擎架构，在 Du-Interact 数据集上离线评测 + 百度搜索大规模在线评估。  
- **关键发现：** 离线 benchmark + 在线 A/B 测试结合是生产环境的可行实践。  
- **对 zerore 的启示：** 回放评测可以作为离线基准，但需要在线验证。

## 五、两种可行的历史数据回放评测方法

基于上述文献调研，结合 zerore 从历史 session 挖掘评测数据的实际需求，提出以下两种评测方法。

**两种方法的核心设计原则：**

1. 保留历史数据的真实性；  
2. 允许 Agent 改变对话路径（不强制逐轮匹配）；  
3. 保证评测的可复现性（确定性或低方差）。

---

### 方法一：基于状态锚定的因果回放评测

#### 1. 方法概述

从历史 session 中提取「状态转换链」作为评测锚点，而非逐字回放 user query。每个状态锚点代表对话中的一个关键进展（如：用户提出需求 → Agent 确认理解 → Agent 调用工具 → 用户反馈结果）。评测时，SimUser 从锚点中提取用户意图进行回放，Agent 在每个锚点处被检查是否达成了对应的状态转换。如果 Agent 提前达成目标状态，则跳过中间锚点；如果偏离因果链，则标记错误。

#### 2. 核心流程

**步骤 1：历史 session 标注**

对每个历史 session，标注状态转换链：

`S₀ → (user_intent₁, agent_action₁) → S₁ → (user_intent₂, agent_action₂) → S₂ → … → Sₙ`

其中 `Sᵢ` 是对话状态（如：需求已明确、工具已调用、结果已确认），`user_intent` 是该轮用户的核心意图摘要，`agent_action` 是该轮 Agent 的关键动作（如：调用 XX 工具、返回 XX 结果）。

**步骤 2：构建评测任务**

每个 session 生成一个评测任务，包含：

- 初始状态 `S₀` 和初始 `user_intent`  
- 目标状态 `Sₙ`（session 的最终目标）  
- 中间锚点列表：`[(S₁, intent₁), (S₂, intent₂), …, (Sₙ₋₁, intentₙ₋₁)]`  
- 因果依赖图：哪些锚点必须在哪些锚点之前达成  

**步骤 3：回放执行**

- SimUser 使用锚点中的 `user_intent` 生成自然语言 query（而非逐字回放）  
- Agent 回复后，LLM 裁判检查：  
  - 是否达成了当前锚点对应的状态？（是 → 前进，否 → 标记偏差）  
  - 是否提前跳过了多个锚点？（是 → 记录「提前解决」，不扣分）  
  - 是否偏离了因果链？（是 → 标记严重错误）  
- 当 Agent 达成目标状态 `Sₙ` 或达到最大轮次时终止  

**步骤 4：评分计算**

- **状态覆盖率：** 达成的锚点数 / 总锚点数（权重 80%）  
- **因果合规率：** 未偏离因果链的轮次 / 总轮次（权重 10%）  
- **效率分：** 实际轮次 / 原始 session 轮次（小于 1 加分，大于 1 扣分，权重 10%）  
- **总分** = 状态覆盖率 × 0.8 + 因果合规率 × 0.1 + 效率分 × 0.1  

#### 3. 可复现性保证

- **SimUser 确定性：** 每个 `user_intent` 对应一个固定的 prompt 模板 + 意图摘要，SimUser 使用低 temperature（≤ 0.3）生成 query，保证 query 生成可复现  
- **裁判结构化：** LLM 裁判使用 rubric 驱动的评分准则（参考论文 2604.27727），每次评判使用 3 次采样 + 多数投票，降低裁判方差  
- **锚点标注标准化：** 状态锚点的标注规则预先定义（如：工具调用、结果确认、需求澄清各为一类锚点），不同标注者对同一 session 的锚点一致性大于 90%  
- **随机种子固定：** 所有 LLM 调用固定 random seed，评测结果可 100% 复现  

#### 4. 优势与局限

**优势**

- 允许 Agent 改变对话路径，解决了「提前解决但 query 还在」的问题  
- 保留历史数据的真实性（意图来自真实 session）  
- 因果链检查确保评测的逻辑严密性  
- 可复现性高（确定性 SimUser + 结构化裁判）  

**局限**

- 需要人工标注状态锚点，标注成本较高（可通过半自动标注降低）  
- 对 topic 跳跃的 session，因果链可能复杂（但仍以 session 为单位，符合需求）  
- SimUser 生成的 query 可能与原始 query 有差异（但意图一致，不影响评测有效性）

### 方法二：动态意图驱动的回放评测（在线评估 / 可复现性主推方向）

#### 1. 方法概述

从历史 session 中提取「意图序列」作为评测核心，将 user query 抽象为意图列表。评测时，SimUser 按意图顺序与 Agent 交互，但每个意图的达成判定是**动态的**：当 Agent 的回复已经覆盖了当前意图时，SimUser 自动切换到下一个意图；当 Agent 未能覆盖当前意图时，SimUser 会尝试用不同方式追问（最多 N 次）；当 Agent 完全偏离时，触发「偏离检测」并标记错误。

#### 2. 核心流程

**步骤 1：意图提取**

对每个历史 session，用 LLM 提取意图序列：

`Intent = [intent₁, intent₂, …, intentₖ]`

每个 `intent` 包含：

- 意图描述（自然语言）  
- 原始 query 示例（1–3 条）  
- 达成条件（Agent 回复满足什么条件算达成）  
- 依赖关系（哪些 intent 必须在哪些之前达成）  

**步骤 2：构建评测任务**

- 初始 prompt：session 的背景信息（如对话主题、用户身份等）  
- 意图序列：按时间顺序排列的 intent 列表  
- 当前意图指针：初始指向 `intent₁`  
- 追问计数器：每个 intent 最多追问 N 次（默认 3 次）  

**步骤 3：动态回放**

循环执行：

1. SimUser 根据当前意图指针，从 intent 的原始 query 示例中选择或生成 query  
2. Agent 回复  
3. 裁判检查 Agent 回复是否覆盖了当前意图：  
   - **是** → 意图指针前进到下一个 intent  
   - **否** → 追问计数器 +1，若小于 N 则 SimUser 换方式追问，否则标记「未达成」并前进  
4. **偏离检测：** 若 Agent 回复与当前意图完全无关，标记「偏离错误」  
5. **终止条件：** 所有意图处理完毕 或 达到最大轮次  

**步骤 4：评分计算**

- **意图达成率：** 成功达成的意图数 / 总意图数（权重 50%）  
- **追问效率：** 总追问次数 / 总意图数（越少越好，权重 20%）  
- **偏离率：** 偏离错误的轮次数 / 总轮次（权重 20%）  
- **轮次效率：** 实际轮次 / 原始 session 轮次（小于 1 加分，权重 10%）  

#### 3. 可复现性保证（论文检索后的在线评估设计要点）

- **意图提取确定性：** 使用固定 prompt + 低 temperature 提取意图，同一 session 多次提取结果一致  
- **SimUser 模板化：** 每个 intent 对应一个 query 生成模板，SimUser 仅在模板内变化，不自由发挥  
- **裁判 rubric 化：** 意图达成判定使用结构化 rubric（覆盖 / 部分覆盖 / 未覆盖三档），降低裁判主观性  
- **偏离检测规则化：** 使用 embedding 相似度 + 关键词匹配检测偏离，而非纯 LLM 判断，保证确定性  
- **随机种子固定：** 所有 LLM 调用固定 seed  

#### 4. 优势与局限

**优势**

- 实现简单，不需要复杂的因果链标注  
- 天然支持 Agent 提前解决（意图指针自动前进）  
- 追问机制模拟了真实对话中的「用户不满意」场景  
- 偏离检测保证了评测的严格性  

**局限**

- 意图提取质量依赖 LLM，可能丢失细微意图  
- 对 topic 跳跃的 session，意图序列可能不够准确  
- 追问机制可能产生不自然的对话（SimUser 反复追问同一意图）

## 六、方法对比与实施建议

| 维度 | 方法一：状态锚定因果回放 | 方法二：动态意图驱动回放 |
|------|--------------------------|--------------------------|
| 实现复杂度 | 中（需标注状态锚点和因果链） | 低（只需提取意图序列） |
| 标注成本 | 高（需定义状态转换） | 低（LLM 自动提取意图） |
| 可复现性 | 高（锚点确定性 + 结构化裁判） | 中高（意图确定性 + rubric 裁判） |
| 评测严格度 | 高（因果链检查） | 中（意图覆盖检查） |
| Agent 路径自由度 | 高（允许跳过锚点） | 高（允许提前达成） |
| 对 topic 跳跃的鲁棒性 | 中（因果链可能复杂） | 高（意图序列天然有序） |
| 适合场景 | 工具调用型对话、任务完成型对话 | 信息查询型对话、多轮澄清型对话 |
| 首期实施周期 | 2–3 周（含标注工具开发） | 1–2 周（LLM 提取 + 模板 SimUser） |

### 实施建议

建议**分两阶段**实施：

1. **第一阶段（1–2 周）：** 实施方案二（动态意图驱动回放）  
   - 快速验证历史数据回放评测的可行性  
   - 用 50–100 个历史 session 做试点  
   - 收集裁判评分与人工评分的一致性数据  
   - 迭代优化意图提取 prompt 和 rubric  

2. **第二阶段（2–3 周）：** 在方法二基础上，对关键场景引入方法一的状态锚定机制  
   - 对工具调用型 session，标注状态锚点  
   - 引入因果链检查，提升评测严格度  
   - 建立离线评测指标与在线 A/B 测试的关联验证  

### 核心结论

两种方法都能保证可复现性的核心在于：

- 用**确定性的意图 / 状态锚点**替代自由对话；  
- 用**结构化的 rubric 裁判**替代自由评判；  
- 用**固定的 random seed**替代随机采样。  

历史数据的真实性通过意图 / 状态的**原始提取**来保证，而非通过逐字回放。
