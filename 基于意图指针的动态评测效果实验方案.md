# 基于意图指针的动态评测效果实验方案

**版本：** v1.0（已冻结首期决策：单臂 + 原始 session 基线雷达 + SiliconFlow + 三分类 Judge）  
**性质：** **可行性试验**（pilot）——不前置统计假设，以能否稳定跑通流水线、Schema 校验通过率、指标可解释性、人工抽检一致性为**验收导向**。

**目标：** 在受控、可复现的前提下，**试跑**「意图指针 + 可回填上下文 + SimUser-as-Judge」闭环；验证抽取 Schema、锁版修订、轮次预算与 SiliconFlow 待测通道是否可工程化落地，并为后续规模化提供实证记录。

---

## 1. 试验目的与验收导向

### 1.1 目的（要回答的问题）

- 流水线能否在 **10 条 session** 上端到端跑通：CSV → Schema 化抽取 → 动态回放 → 日志与指标落盘。  
- **预定义 JSON Schema** 的抽取结果是否稳定可校验、可人工修订并**锁版**后再跑评测。  
- **SimUser 单意图轮次预算**（`B_i = ceil(2·n_i)` 等）是否在实际调用中可防拖死、日志可审计。  
- **偏离判定**在仅做 LLM **三分类**时，方差与人工抽检是否可接受（作为经验结论记录，不作假设检验）。

### 1.2 验收导向（不作 H0/H1 假设）

| 维度 | 首期记录方式 |
|------|----------------|
| 可跑通 | 10 条中 `pipeline_success` 条数 / 10；失败原因分类统计 |
| Schema | 首轮自动抽取校验通过率；锁版后二次校验 100% |
| 可理解 | 每 session 保留 Judge `rationale` + `evidence_quote` 抽检表 |
| 可比 | 与 **§7.4** 基线向量同轴雷达对比（定性为主） |

## 2. 数据输入：10 组 Session 与 CSV

### 2.1 规模与形态

- **数量：** 10 组多轮对话 session（建议每条 session 独立 `session_id`，组内多轮以多行表示）。  
- **载体：** 单个 **CSV** 文件作为唯一输入源（UTF-8，建议带表头）。

### 2.2 CSV 最小列集（建议）

| 列名 | 必填 | 说明 |
|------|------|------|
| `session_id` | 是 | 会话标识，同一 session 多行相同 |
| `turn_index` | 是 | 单调递增，便于对齐「历史轮次」 |
| `role` | 是 | `user` / `assistant`（若仅有 assistant 最终可见，需约定如何表示 user 侧） |
| `content` | 是 | 该轮文本 |
| `timestamp` | 否 | ISO8601，便于统计响应间隔（与 Zeval 客观指标可选对齐） |

**首期约定：** CSV **实际列名与样例文件稍后补录**；在样例发布前，实现侧以「最小可解析列集」为开发假设（见上表），最终以锁版 CSV 说明为准。若无 `tool_result` / `metadata`，可回填项完全依赖 §4 从纯文本推断。

### 2.3 数据质量门禁

- 每个 `session_id` 至少包含 **2 个 user 轮次**（否则多轮动态评测意义有限）。  
- `turn_index` 在 session 内唯一且有序。  
- 实验前脚本校验：无空 `content`、角色合法、session 数 = 10。

## 3. 流水线总览

```
CSV 读入
  → 校验与规范化（session 切分、排序）
  → 【一次模型调用】按 **JSON Schema** 输出 intent_sequence + refillables（§4）
  → **人工修订（若需要）并锁版** `*_locked.json`；正式动态回放**仅消费锁版**（§4.5 / §9）
  → 按 session 实例化：意图指针状态机 + 可回填 prompt 块（§5–§6）
  → 循环：SimUser 生成 user → 待测 Agent → SimUser-as-Judge 判定（§5）
  → 汇总指标（§7）+ 基线 B vs 动态 D 雷达（§7.4）+ 可选 Zeval 客观/主观跑分（§7.3）
  → 实验日志与可复现包（seed、prompt 版本、模型名）
```

## 4. 第一步：单次模型调用 — 意图序列 + 可回填信息（预定义 Schema）

对 **每个 session**，在载入 CSV 并拼好 transcript 后，执行 **恰好一次**「整理型」模型调用，输出 **两组结构化信息**。**必须先**在仓库中固化 `extract_session.schema.json`（或等价 JSON Schema）及 `prompt_version`：**模型只负责填空，不负责发明字段**，从而同时满足：

1. **可复现：** 同一 transcript + 同一 schema 版本 + 同一 seed → 输出可被校验、diff、回放；后续 SimUser / Judge 只消费结构化结果，减少自由文本漂移。  
2. **任务驱动不再重复执行：** 对「查状态、查单号」等已在历史中闭环的信息，通过 **可回填项** 显式写入「已知结论」，由待测 Agent **据已知事实作答**，评测聚焦对话策略与解释能力，而非再次调用外部执行链。

### 4.0 业务侧概念结构与工程键名映射

与产品/标注同学对齐时，可采用下列**中文概念数组**（与你们口头 schema 一致）；落库与校验时映射为 **英文键名**（下表）。

**意图序列（概念示意）：**

```text
[
  { 意图序号: 1, 意图内容: "查询订单状态", 轮次长度: 2 },
  { 意图序号: 2, 意图内容: "……", 轮次长度: … }
]
```

**可回填项（概念示意）：**

```text
[
  { 回填序号: 1, 触发条件: "提出查询订单状态", 回填参考: "进行中" },
  { 回填序号: 2, … }
]
```

| 概念字段（中文） | 工程键名（建议） | 用途 |
|------------------|------------------|------|
| 意图序号 | `intent_index`（或 `intent_id`） | 指针顺序、日志对齐 |
| 意图内容 | `intent_text`（可与 `description` 合并） | SimUser 生成与 Judge rubric 锚点 |
| 轮次长度 | `turn_span_user_turns` | **优先作为** §5 中 `n_i` 的来源（历史该意图下用户发起轮数）；与 `historical_span` 二选一或互相校验 |
| 回填序号 | `refill_index` | 列表顺序与审计 |
| 触发条件 | `trigger_condition` | 与某类用户意图或 `intent_index` 对齐，用于决定何时把该回填注入 system 前缀（通常：进入对应意图或首轮即注入） |
| 回填参考 | `refill_reference` | 写入「已知事实」短文本；可衍生 `injection_text` |

### 4.1 顶层 JSON 形态（一次响应、两组数组）

根对象建议包含 `schema_version`、`session_id` 及两个数组字段，便于 CI 校验：

```json
{
  "schema_version": "2026-02-intent-refill-v1",
  "session_id": "sess_001",
  "intent_sequence": [
    {
      "intent_index": 1,
      "intent_text": "查询订单状态",
      "turn_span_user_turns": 2,
      "example_user_queries": ["帮我查一下订单12345的状态"],
      "success_criteria": "助手明确给出该订单状态（进行中/已完成/已取消等）或等价结论",
      "depends_on": [],
      "historical_span": { "start_turn_index": 1, "end_turn_index": 4 }
    }
  ],
  "refillables": [
    {
      "refill_index": 1,
      "trigger_condition": "用户提出查询订单状态且历史中已给出状态",
      "refill_reference": "订单12345：进行中",
      "key": "order_12345_status",
      "source_turn_index": 3,
      "confidence": "high",
      "injection_text": "【会话已知事实】用户已查询订单12345；历史对话中已确认状态为「进行中」。请在不再调用外部订单系统的前提下，基于该事实回答用户后续问题。"
    }
  ]
}
```

> `historical_span` 可与 `turn_span_user_turns` 同时存在：**优先用 `turn_span_user_turns` 作为 `n_i`**；若与 span 内统计的 user 行数差异大于 1，则打标 `n_i_conflict=true` 供人工复核。

### 4.2 输出 A：`intent_sequence` 字段说明（与上表一致）

| 字段 | 说明 |
|------|------|
| `intent_index` / `intent_id` | 稳定序号，从 1 或 0 起均可，**全 session 唯一、单调**即可 |
| `intent_text` / `description` | 意图自然语言描述（可与「意图内容」合并为一字段） |
| `turn_span_user_turns` | **轮次长度**：对应概念字段，供 §5 直接作为 `n_i` |
| `example_user_queries` | 1–3 条来自历史的典型 user 表述（可原文摘录） |
| `success_criteria` | 判定「达成」的 rubric 要点（供 Judge 使用） |
| `depends_on` | 前置 `intent_index` 列表，无则 `[]` |
| `historical_span` | 可选：`turn_index` 闭区间，用于校验或与 `turn_span_user_turns` 交叉验证 |

### 4.3 输出 B：`refillables` 字段说明（与上表一致）

针对**任务驱动型**对话：历史 assistant 中可能已包含「订单状态、单号、结论」等，通过结构化条目避免待测 Agent **重复执行**同一外部任务。

| 字段 | 说明 |
|------|------|
| `refill_index` | 回填序号 |
| `trigger_condition` | 与「提出某类意图」对齐的自然语言条件；实现上可与 `intent_index` 显式绑定（建议扩展字段 `bind_intent_index`，首期可选） |
| `refill_reference` | 回填参考短文本（事实核心） |
| `key` | 可选，机器可读键 |
| `source_turn_index` | 证据轮次 |
| `confidence` | 可选：`high` / `medium` / `low` |
| `injection_text` | **已拼好**的可直接拼入待测 Agent system/developer 前缀的段落 |

**示例（与需求一致）：** 用户曾查订单 `12345`，历史中已出现「进行中/已完成」→ 抽取为 `refillables`，注入新 Agent：「以下为该会话已知事实，请在不重复调用外部系统的前提下据此作答……」。

### 4.4 调用约束（可复现）

- **固定** `prompt_version`、`temperature`（意图抽取建议 `≤0.3`）、`max_tokens`、`random_seed`。  
- 输出经 **JSON Schema** 校验（字段类型、必填项、`intent_sequence` 与 `refillables` 数组非空策略等）；失败则：重试 1 次 → 仍失败则该 session **整例跳过**并记 `pipeline_error`（不计入主指标分母需在 §7 声明）。

### 4.5 与「第二步评测」的边界

- 本步 **不负责** 与待测 Agent 多轮对话；**只负责** 离线生成「意图计划 + 回填包」。  
- **人工修订与锁版（已确认）：** 允许标注员在首轮自动抽取后，**仅依据同一 JSON Schema** 修订 `intent_sequence` / `refillables`（不增删顶层字段名）；修订后写入 `*_locked.json` 并 bump `schema_lock_revision`，动态回放**只消费锁版文件**。无修订时可将首轮校验通过的 JSON **复制**为 `*_locked.json` 视为锁版。建议 10 条全检后锁版再跑评测。

## 5. SimUser：意图指针、单意图轮次预算、防递归与跳转

SimUser **兼任**（1）用户话术生成者（2）当前意图是否达成的 **Judge**（可与你方「参考人类 query 时依据上一轮模型回复」一致：**仅根据上一轮 assistant 全文 + 当前意图 rubric** 判定，不引入自由闲聊）。

### 5.1 状态变量（每个 session 一份）

| 符号 | 含义 |
|------|------|
| `i` | 当前意图下标，`0 … k-1` |
| `c_i` | **当前意图**下已消耗的「评测轮次」计数（定义见下） |
| `n_i` | 历史 session 中，该意图从「开始被用户提出」到「历史对话显示已解决或进入下一意图」所经历的 **用户发起轮次数**（见 §5.2 默认算法） |
| `B_i` | 当前意图允许的最大评测轮次：`B_i = ceil(α · n_i)`，**默认 α=2**（与需求「≤2n」一致；`n_i=0` 时见 §5.4） |

**「一轮」计数定义（建议采用，便于实现与审计）：**

- **1 个评测轮次** = SimUser 发出 **一条** user 消息 + 待测 Agent 返回 **一条** assistant 消息 + Judge 完成一次判定。  
- 每完成这样一整拍，`c_i += 1`。

### 5.2 历史 `n_i` 的默认估计（无人工标边界时）

**优先（与 §4 Schema 对齐）：** 若 `intent_sequence[i]` 提供 **`turn_span_user_turns`（轮次长度）** 且通过 JSON Schema 类型校验，则直接令 **`n_i = turn_span_user_turns`**。

**否则：** 利用 §4 输出的 `historical_span`（若模型未给，则用启发式：`intent_i` 的 example query 首次出现近似 turn 作为起点，`intent_{i+1}` 起点前一 user 轮为终点）：

- **`n_i`** = 该 span 内 **`role=user` 的行数**（即用户主动发起次数）。  
- 若无法切 span，退化为：**整 session 的 user 轮数 / intent 数** 的上取整，并打标签 `n_i_heuristic=true` 供分析分层。

### 5.3 单意图内：Judge 结果与状态迁移

对当前意图 `I_i`，在每一评测轮次末尾：

1. **达成（SATISFIED）**  
   - `c_i` 清零与否不重要（进入下意图时重置）。  
   - `i ← i+1`，`c_{i} ← 0`（新意图的计数从零开始）。  
   - 若 `i == k`：**正常结束**（成功跑完意图链）。

2. **未达成（NOT_SATISFIED）且 `c_i < B_i`**  
   - SimUser 生成**追问**（模板化：换句式、缩小范围、引用 success_criteria 中的一条），**不前进** `i`，`c_i` 已在本轮末尾 +1。

3. **未达成且 `c_i == B_i`（本轮耗尽预算）**  
   - 记 **`intent_failed_budget`**，该意图计为**未达成**（计入意图完成率分母与失败分子，规则见 §7）。  
   - **强制** `i ← i+1`，`c ← 0`：**跳转下一意图**（与需求「超出则判定此意图失效，切换到下一意图」一致）。  
   - **首期已确认：** 跳转前 **不追加**「放弃本意图」式 user 句，直接指针前进，避免额外噪声。

4. **偏离（DEVIATION）**（首期 **仅 LLM 三分类**之一，与 SATISFIED / NOT_SATISFIED 同源 Judge 调用；**不上** embedding+关键词并行通道）  
   - 记 `deviation` 事件，`c_i += 1`。  
   - **首期已确认策略**：偏离**消耗预算**但不自动前进意图指针；若随后仍 NOT_SATISFIED 直至耗尽 `B_i`，同 (3) 跳转。  

### 5.4 边界：`n_i = 0` 或 `k = 1`

- **`n_i = 0`：** 设 `B_i = B_min`（建议 **3**），并在日志打 `n_i_imputed`。  
- **单意图长 session：** `B_i` 设全局硬顶 `B_global_cap`（建议 **24** 或 `2 × session_user_turns` 取小者），防止模型错估 `n_i` 导致爆炸。

### 5.5 全局终止条件

满足任一即结束该 session 评测：

- 意图指针已越过最后一个意图（成功或预算失败跳过后无剩余）。  
- **全局轮次** `Σ c` 或 **总 user 条数** 达到 `G_max`（建议 `G_max = sum_i(B_i) + k` 或单独配置常数如 **40**）。  
- 待测 Agent **连续错误**（HTTP/超时）达到阈值（如 3 次）→ session **abort**（单独统计，不混入主指标除非另有定义）。

## 6. 待测 Agent 侧：可回填 Prompt 注入

- **位置：** 每次调用待测 Agent 时，将 `refillables` 拼成固定模板的 **system / developer** 前缀（勿与 SimUser 的 user 消息混在同一条 user 里，避免角色污染）。  
- **原则：** 明确写出「已知事实仅供推理与回复一致性，不要求再次执行已完成的查询」。  
- **频次（默认建议，可改）：** 若拼接后 token 可控，**每一评测轮次**均在待测 Agent 的 system 侧附带同一 `refillables` 块，避免多轮后模型「遗忘」已知事实；若总长过大，则**首轮全文 + 后续仅摘要**（摘要中保留 `key/value` 列表与 `session_id`）。  
- **与 Zeval 对齐：** 若后续接入 Zeval 评测管线，该注入块需在 artifact 中记录 `prompt_hash`，避免与生产 prompt 混淆。

## 7. 评测指标（与本实验对齐 + 与 Zeval 对照）

### 7.1 本实验主指标（与调研文档方法二一致，可落地）

| 指标 | 定义 | 建议权重（报告内） |
|------|------|--------------------|
| **意图完成率** | `达成意图数 / k`（预算失败、偏离策略失败均计未达成） | 50% |
| **追问效率** | `总追问次数 / k`，追问 = NOT_SATISFIED 后产生的第二条及以后 user | 20% |
| **偏离率** | `判定为 DEVIATION 的轮次数 / 总会话轮次` | 20% |
| **轮次效率** | `T_eval / T_hist`，其中 `T_hist` = 该 session 历史 user 轮数，`T_eval` = 本实验实际评测轮次数；**小于 1 表示更省轮** | 10% |

**综合分（可选）：** 与调研稿相同线性加权，或仅并列报告四项不做总分，避免过度优化单一标量。

### 7.2 Judge 实现（首期已冻结：仅三分类）

- **偏离检测：** **只做 LLM 三分类**（`SATISFIED | NOT_SATISFIED | DEVIATION`），与「意图是否达成」同一 Judge 能力边界；**首期不上** embedding+关键词双通道。  
- **同一模型**承担「生成下一条 user」与「判定」时：使用 **两次独立 API 调用**（或同一调用内 strict JSON 两段字段），且 **判定调用 `temperature=0`**。  
- **判定输出** 限定上述三枚举，并强制 `rationale` 一句 + `evidence_quote` 从上一轮 assistant 复制子串，便于人工审计。

### 7.3 与 Zeval / zerore 管线可对齐项（首期可选）

在动态回放结束后，将**完整评测 transcript**（含注入块摘要、每轮判定）送入现有 **evaluate** 能力（若已部署）：

- **客观侧（有则算）：** 轮次数、重复提问率、平均响应间隔（若 CSV 有时间戳）。  
- **主观侧：** 共情、答非所问风险等维度（**注意：** 与「意图完成」可能相关但不等价，报告中分栏展示）。  
- **badcase 标记：** 若 `意图完成率 < 1` 或 `偏离率` 高，可自动打标进入案例池候选（与 `source` 策略另文约定）。

### 7.4 效果对比与呈现：原始 session 基线 × 雷达图（首期已冻结）

- **评测臂：** 首期 **仅跑「意图指针 + 动态回放」单臂**，不并行跑「固定历史 user 逐轮回放」第二臂。  
- **基线定义：** 对每个 session，在 **原始 CSV transcript**（未经过动态 SimUser 环）上计算一组**基线指标向量** `B`：  
  - **优先：** 将原始 session 送入既有 **Zeval `evaluate`**（或等价管线）一次，取与试验相关的标量（如客观汇总、主观维度、轮次统计等），归一化到与雷达轴一致的量纲；  
  - **保底：** 至少包含与本实验对齐的四项中的可静态计算项（如 `T_hist`、可由历史最终态代理的「意图完成」占位规则需在报告中显式写出，避免暗箱）。  
- **动态臂向量 `D`：** 本实验 §7.1 四项主指标 +（可选）同一 Zeval 在**动态回放完整 transcript** 上的复评结果。  
- **呈现：** 每个 session 或 10 条汇总 **一张雷达图**，**同一坐标轴**上叠绘 **`B`（原始 session）** 与 **`D`（动态评测）** 两条闭合折线（双色图例），用于**定性对比**可行性试验效果，不作统计假设检验。

### 7.5 待测与 Judge 调用：SiliconFlow（与 zerore 对齐）

与 `zerore-eval-system-main/.env.example` 保持一致，使用 **OpenAI 兼容** SiliconFlow 网关：

| 变量（示例） | 用途 |
|--------------|------|
| `ZEVAL_JUDGE_BASE_URL` | 默认 `https://api.siliconflow.cn/v1` |
| `ZEVAL_JUDGE_MODEL` | 与项目一致，如 `Qwen/Qwen3.5-27B` |
| `ZEVAL_JUDGE_ENABLE_THINKING` | **`false`**（与项目红线一致，避免思考链占满输出） |
| `ZEVAL_JUDGE_API_KEY` | Judge / 抽取 / SimUser 所用密钥（首期可与待测调用复用同一 key，或见下条拆分） |

**产品需求（已确认方向）：** 增加**独立密钥配置项** `ZEVAL_INTENT_EXPERIMENT_API_KEY`（若未设置则回退 `ZEVAL_JUDGE_API_KEY`），支持在 **前端填写并保存** 后，由**后端**写入运行环境（如更新 `.env` 或等价配置源）并供后续请求读取。待测 Agent 与 Judge/抽取 共用 **`ZEVAL_JUDGE_BASE_URL`** 与 **`ZEVAL_JUDGE_MODEL`** 时，仅通过路径或请求体区分路由（实现细节由工程定稿）。

**安全与工程注意（实施必读）：**

- 密钥 **不得** 明文写进前端仓库或日志；前端仅通过 HTTPS 提交至受信后端。  
- **自动写 `.env` 文件** 仅适合 **本地 / 内网 POC**；生产环境应使用密钥管理服务或运行时注入，避免多进程与文件竞态。  
- 若采用写 `.env`：需定义是否触发热重启、文件权限与备份策略。

## 8. 实验设计要素小结

| 要素 | 取值/说明 |
|------|-----------|
| Session 数 | 10 |
| 输入 | 单 CSV，UTF-8 |
| 结构化整理 | 每 session **1 次**模型调用 → `intent_sequence` + `refillables` |
| 单意图预算 | `B_i = ceil(2 · n_i)`，默认；`n_i` 来自历史 span 内 user 轮数 |
| 超预算 | 记意图失败，`i++` |
| SimUser / Judge | 模板化追问 + **LLM 三分类** Judge（`SATISFIED` / `NOT_SATISFIED` / `DEVIATION`）；首期 **无** embedding+关键词偏离通道 |
| 主指标 | 意图完成率、追问效率、偏离率、轮次效率 |
