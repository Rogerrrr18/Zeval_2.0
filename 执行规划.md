# 执行规划（Vibe Coding 依据）

本文用于指导 1 天内完成 ZERORE Eval MVP 第一版，重点是打通端到端 pipeline，而非做大而全产品。

## 0. 当前进度与阶段判断

截至当前代码状态：

- 已完成：
  - Next.js 项目初始化与前端控制台
  - 四种格式解析与 `ingest/evaluate` API
  - `Raw -> Enriched -> ChartPayload` 三层链路
  - 一版客观指标聚合
  - 一版主观评估与 `SiliconFlow` LLM judge 接入
  - 规则优先、LLM 兜底的 topic segment 切分
  - segment 级结构化 emotion scoring
  - 隐式推断信号层初版
  - enriched artifact 落盘
  - 图表与建议链路打通
- 当前缺口：
  - signal layer 还未完全接入自动策略触发
  - 仍缺少更多业务模板化指标

结论：

- 当前项目已经完成“第一阶段 P0 可运行 MVP”
- 接下来的目标不再是补主链路，而是提升准确率、泛化性与可配置化

## 1. 目标边界与约束

- 具体业务场景仍在调研，因此本版采用“场景无关的通用评测框架”实现。
- 第一版必须打通完整链路：
  - 前端接收输入
  - 后端解析数据
  - 评估逻辑执行（客观 + 主观）
  - 返回交付产物（图表 + 优化建议）
- 技术栈固定：`Next.js + TypeScript`
- 优先保证可跑通、可解释、可演示，UI 精细度不是当前重点。

## 2. 输入数据规划（必须准备）

### 2.1 Mock 长对话数据（四种格式各一份）

在 `mock-data/` 准备同语义等价数据：

- `long-dialog.csv`
- `long-dialog.json`
- `long-dialog.txt`
- `long-dialog.md`

### 2.2 Mock 数据字段建议

`Raw` 样本应尽量只包含：

- `sessionId`
- `role`（`user | assistant | system`）
- `content`
- `timestamp`（ISO 格式，便于时序指标）

说明：

- `topic`、`turnIndex`、情绪、行为信号都不应出现在 raw chatlog
- 这些字段应由预处理阶段写入 `Enriched`

### 2.3 Mock 数据内容要求

- 至少 3 个 session
- 每个 session 至少 30-50 轮（体现“长对话”）
- 覆盖至少 3 类 topic 切换
- 包含：
  - 正常推进片段
  - 情绪波动片段
  - 中断/流失片段
  - AI 回答偏离或说教倾向片段

## 3. 系统模块拆分

### 3.0 三层产物约定（新增）

为保证链路清晰与可复用，评估系统固定输出以下中间层：

- `Raw`：原始日志（sessionId/timestamp/role/content）
- `Enriched`：预处理补全字段后的明细 CSV（canonical artifact）
- `ChartPayload`：根据预定义模板填充后的图表渲染数据

约束：

- 评估算法与图表参数提取只读取 `Enriched` 层
- 前端图表渲染只读取 `ChartPayload` 层
- `Enriched` 字段需包含来源与置信度，保证可解释性
- `Raw` 不承担业务语义补全职责
- `session` 内多主题切分属于预处理核心能力，不属于前端或图表层逻辑

## 3.1 前端模块

- 上传组件：支持文件上传 + 格式提示
- 执行组件：触发评估并展示进度状态
- 结果组件：
  - 图表区（客观指标可视化）
  - 主观评估区（情绪曲线、体验评分）
  - 优化建议区（按优先级排序）

## 3.2 后端模块

- `parser`：四种格式转统一数据结构
- `normalizer`：字段补齐、session 切分、质量校验
- `segmenter`：`session` 内 topic segment 切分
- `objectiveEvaluator`：固定算法计算客观指标
- `signalExtractor`：隐式推断信号层
- `subjectiveEvaluator`：LLM Judge + 信号层聚合
- `reportBuilder`：组装统一输出（图表数据 + 建议）
- `chartParamExtractor`：从 enriched CSV 聚合参数并填充模板 schema

## 4. 评估逻辑设计（参考 chatlog 评估维度）

> 依据 `chatlog 评估维度.xlsx`，第一版按“客观先稳定、主观先可用”实施。

### 4.1 第一阶段 P0 客观指标（必须落盘）

必须落地以下指标：

- `sessionDepthDistribution`
  - 定义：按 session 聚合总轮次后的深度分布
  - 价值：找到流失发生在浅层还是深层
- `dropoffTurnDistribution`
  - 定义：用户停止继续对话时集中出现在哪一轮
  - 价值：定位死亡轮次
- `avgResponseGapSec`
  - 定义：基于 timestamp 的响应间隔统计
  - 价值：判断思考成本、犹豫程度与交互节奏
- `activeHourDistribution`
  - 定义：活跃时段分布
  - 价值：跨场景通用，可用于运营与陪伴时段判断
- `topicSwitchRate`
  - 定义：topic segment 切换频率
  - 价值：判断对话是否碎片化或缺乏主题延展
- `userQuestionRate`
  - 定义：用户消息中提问轮次占比
  - 价值：反映探索意愿与投入度
- `userMessageLengthTrend`
  - 定义：用户消息长度的变化趋势，而不只是平均值
  - 价值：识别兴趣下降或参与度上升
- `avgAssistantMessageLength`
  - 定义：AI 平均回复长度
  - 价值：识别过长压迫或过短敷衍

暂不列入第一阶段 P0：

- `24h 返回率`
- `最佳实践对话模式`
- `负面模式识别`
- `个人信息分享度`
- `emoji 使用率`

原因：

- 依赖更长时间窗口，或业务语义过强
- 容易因场景未定而导致指标解释漂移
- 当前阶段应先保证“通用、稳定、可解释”

### 4.2 第一阶段 P0 主观指标（必须落盘）

必须落地以下指标：

- `emotionCurve`
  - 核心亮点
  - 必须支持按 turn 与按 topic segment 观察
  - 分数采用 `0-100`，保留 `1` 位小数
- `emotionTurningPoints`
  - 标记明显上升或下探的 turn/segment
- `emotionRecovery`
  - 衡量负向情绪恢复到中性或正向所需轮次
- `empathyScore`
  - 评估接纳、理解与安抚能力
- `offTopicOrIgnoreRisk`
  - 将“答非所问”与“无视用户关键点”合并建模
- `preachinessRisk`
  - 评估说教感、压迫感、命令式表达

说明：

- `情绪曲线 + 隐式推断` 是 MVP 的核心亮点，优先级高于一些复杂但泛化差的业务指标
- `追问有效性` 很重要，但当前可先作为 `empathy/off-topic` 的补充证据，不必单独立成 P0 指标

执行策略：

- 先按 `session/topic segment` 切片，再送入 LLM 评估
- 每个切片输出结构化结果：`score + reason + evidenceTurnRange + confidence`
- 对全局结果做聚合，生成趋势、拐点与风险标签

### 4.2.1 Segment Emotion Scoring 规则

情绪分不再直接由 LLM 整体打分，而采用两段式：

1. `LLM` 只负责主观判断：
   - `emotionPolarity`
   - `emotionIntensity`
   - `emotionBaseScore`
   - `emotionEvidence`
   - `emotionConfidence`
2. 本地评分函数负责结构化修正：
   - `emotionValenceWeight`
   - `emotionLengthWeight`
   - `emotionStyleWeight`
   - `emotionGapWeight`
   - `emotionRecoveryWeight`
   - `emotionRiskPenalty`

最终计算：

- `emotionScore = emotionBaseScore + 各项加权修正 - riskPenalty`
- 分值区间固定 `0-100`
- 保留 `1` 位小数

设计原则：

- LLM 只介入情绪倾向、强度、基准分与证据
- 评分函数只使用确定性的客观特征
- 输出必须保留各因子权重，便于解释与复核

### 4.3 隐式推断信号层（第一阶段必须实现）

信号层目标：

- 将弱结构化行为模式转换为稳定的风险信号
- 作为 `subjectiveEvaluator` 与 `suggest` 的共享输入
- 避免让 LLM 直接从长 transcript 中“全靠感觉”得出结论

第一阶段只实现 3 个信号：

- `interestDeclineRisk`
  - 触发条件：
    - 连续短回复
    - 回复间隔拉长
    - 用户提问率下降
  - 输出：
    - `score`
    - `triggeredRules`
    - `evidenceTurnRange`
    - `reason`
- `understandingBarrierRisk`
  - 触发条件：
    - 重复提问
    - 澄清表达增多
    - 困惑词频升高（如“什么意思”“不懂”“你是说”）
  - 输出：
    - `score`
    - `triggeredRules`
    - `evidenceTurnRange`
    - `reason`
- `emotionRecoveryFailureRisk`
  - 触发条件：
    - 连续负向情绪
    - 多轮 assistant 介入后仍未恢复
    - 断点前情绪仍处于低位
  - 输出：
    - `score`
    - `triggeredRules`
    - `evidenceTurnRange`
    - `reason`

实现策略：

- 规则层负责稳定触发
- LLM 负责把规则信号翻译成可读解释与证据
- 若 LLM 不可用，保留规则结果并标记为 `degraded`

## 5. LLM Judge 设计规范（MVP）

### 5.1 输入

- topic segment 消息列表
- 评估维度列表
- 评分标准（1-5 或 1-10）
- 可选信号层输出

### 5.2 输出（强约束 JSON）

- `dimension`
- `score`
- `reason`
- `evidence`（引用轮次或原文片段）
- `confidence`

对于信号层补充输出：

- `signalKey`
- `triggeredRules`
- `evidenceTurnRange`

### 5.3 风险控制

- 输出 JSON schema 校验（失败则重试）
- 增加温度与提示词约束，减少漂移
- 若 LLM 不可用，返回“主观评估暂不可用”并保留客观结果
- 对长 session 先切 topic segment，再限制单次送入 LLM 的 turn 范围

## 7. 图表与建议产物（第一版最小集合）

图表建议先做 4 张：

- 情绪曲线图（折线）
- 断点分布图（柱状）
- 活跃时间图（热力或柱状）
- topic 切换流图（简化版可先柱状替代）

图表实现原则：

- 使用预定义模板，不写临时拼装图表代码
- 只通过替换 schema 参数名和参数值完成渲染

建议输出规则：

- 每条建议绑定触发指标
- 每条建议包含：问题 -> 影响 -> 建议动作
- 至少输出 5 条，按优先级排序（P0/P1/P2）

## 10. 风险与应对

- `timestamp` 缺失导致时序指标不稳定：
  - 应对：在报告中标记“降级计算”，并跳过相关指标
- LLM 返回不稳定：
  - 应对：JSON schema 校验 + 重试 + 兜底文案
- topic 切分误差大：
  - 应对：先规则预切，再用 LLM 做 segment 校正，保留阈值配置
- 隐式推断漂移：
  - 应对：先规则触发，再让 LLM 只负责解释，不让它独立发明信号
- 一天时间不足：
  - 应对：优先交付 topic segmentation、P0 指标与信号层，缩减次要图表和长尾维度
